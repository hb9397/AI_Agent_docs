# 스킬셋 리팩토링 2차 점검 보고서

> 점검일: 2026-06-28  
> 점검 대상: `skills/` 하위 전체 파일, `README.md`, `Harness_Engineering.md`, `Harness_Engineering_Intro.md`, `리팩토링 작업 계획서.md`, 원문 `스킬레포 개선안.txt`

## 1. 전체 판단

큰 방향은 원문 개선안과 계획서의 핵심 의도에 맞게 반영되어 있다.

- `skills/`는 현재 20개 스킬 디렉토리로 구성되어 있으며, 신규 `harness-setup`이 추가되어 있다.
- 대부분의 산출물/적용범위 스킬에는 STEP 0의 단일/복수 애플리케이션 확인 게이트가 들어갔다.
- `model:` 필드와 `agent: fork` frontmatter 하드코딩은 현재 `SKILL.md`에서 발견되지 않았다.
- 이 레포 자체의 `.claude/skills`와 `.agents/skills`에는 `harness-setup`, `custom-skill-design`만 배치되어 있다.
- README의 단일/복수 애플리케이션 설치 구조와 `.docs/root-context/CLAUDE.md`, `.docs/root-context/AGENTS.md` 관계도는 추가되어 있다.

다만 아래 항목들은 사실과 다르거나, 계획서/원문 의도와 어긋나거나, 문서 표현이 불완전하다.

## 2. 사실과 다른 점

### 2-1. `Harness_Engineering.md`의 "모든 스킬 C-1" 표현은 과하다

`Harness_Engineering.md`는 "모든 스킬은 STEP 0에서 프로젝트 유형(단일/복수)을 감지하고 사용자에게 확인한다"라고 설명한다.

실제 파일 기준:

- `git-scoped-account`는 C-1 게이트가 없다. 이 스킬은 여러 git repo에 계정을 적용하는 운영 스킬이라 원문에서도 "현재 유지"로 분류되었다.
- `custom-skill-design`은 새 스킬 설계용 메타 스킬이며, 산출물/적용범위 스킬의 C-1 필요 여부를 판단하도록 개선되었지만, 자기 자신이 프로젝트 유형을 항상 감지하는 구조는 아니다.

권장 수정:

- "모든 스킬"이 아니라 "산출물 생성 또는 코드/커밋 적용범위를 갖는 스킬"로 좁혀 표현한다.

### 2-2. `Harness_Engineering.md`에 오래된 `.instruction/*` 표기가 남아 있다

현재 표준은 단일 앱 `.docs/instruction/*-instruction.md`, 복수 앱 `.docs/{앱}/instruction/*-instruction.md`다.

그런데 `Harness_Engineering.md`에는 아래처럼 오래된 표기가 남아 있다.

- 주제별 `.instruction/*-instruction.md`
- `CLAUDE.md / AGENTS.md / .instruction/*-instruction.md`
- 관련 `.instruction/*-instruction.md`

권장 수정:

- 모두 `.docs/instruction/*-instruction.md`로 바꾸되, 복수 앱 설명이 필요한 문맥에서는 `.docs/{앱}/instruction/*-instruction.md`도 함께 병기한다.

### 2-3. `Harness_Engineering_Intro.md`의 `basic-instruction.md` 예시는 실제 산출물과 맞지 않는다

`Harness_Engineering_Intro.md`에는 `@./.docs/instruction/basic-instruction.md`와 `basic-instruction.md`가 예시로 나온다.

실제 `context-doc` 산출물 카탈로그에는 `basic-instruction.md`가 없다. 현재 생성 대상은 아래 7종이다.

- `architecture-instruction.md`
- `code-style-instruction.md`
- `framework-instruction.md`
- `api-instruction.md`
- `comm-instruction.md`
- `file-convention-instruction.md`
- `agent-instruction.md`

권장 수정:

- 예시를 `agent-instruction.md` 또는 실제 생성 가능한 instruction 파일명으로 바꾼다.

### 2-4. `design-prototype-docs`의 복수 앱 저장 경로 설명과 예시가 서로 충돌한다

`skills/design-prototype-docs/SKILL.md`는 복수 앱 저장 경로를 다음처럼 설명한다.

- 설명: `.docs/prototype/{사용자}/{식별자}/design-doc.md`
- 예시: `.docs/fe-portal/prototype/hb9397/SFR-019/design-doc.md`

계획서 C-2와 `create-prototype`은 복수 앱에서도 프로젝트 공통 prototype 경로를 사용하도록 되어 있다.

권장 수정:

- 복수 앱 예시도 `.docs/prototype/hb9397/SFR-019/design-doc.md`로 맞춘다.

### 2-5. `impl-fe-be-doc`에 `rfp-design-input 문서`라는 옛 표현이 남아 있다

D-7 확정값은 `rfp-ingest`가 `rfp-design-input-{SFR}.md` 파일을 만들지 않고 대화/후속 스킬 입력으로만 전달한다는 것이다.

그런데 `skills/impl-fe-be-doc/SKILL.md`에는 아래 표현이 남아 있다.

- `rfp-design-input 화면 후보`
- `rfp-design-input 문서 모두 가능합니다`

실제 `rfp-ingest`와 문서 3종은 "파일 미생성, 대화 컨텍스트 전달"로 정리되어 있으므로 이 표현은 오해를 만든다.

권장 수정:

- `rfp-ingest 대화 컨텍스트의 화면 후보`로 바꾼다.
- "문서"라는 표현은 제거한다.

### 2-6. `rfp-ingest`의 `allowed-tools: Agent`는 현재 공통 양식 관점에서 애매하다

다른 병렬 스킬들은 `Task`를 쓰고 있고, 계획서의 C-3는 플랫폼 전용 자동화를 frontmatter에 강제하지 말라고 했다.

`rfp-ingest`는 frontmatter에 `allowed-tools: Read, Glob, Grep, Bash, Agent`를 선언한다. `agent: fork`는 아니지만, `Agent`가 실제 허용 도구명인지와 Codex 중립성 측면이 불명확하다.

권장 수정:

- 실제 사용 도구명이 `Task`라면 `Agent`를 `Task`로 맞춘다.
- 플랫폼 중립성을 더 강하게 가져가려면 병렬 실행은 본문 STEP 0 게이트로만 남기고 frontmatter에는 실제 공통 도구만 선언한다.

## 3. 변경계획서와 다른 점

### 3-1. 계획서의 초기 인벤토리 19종은 현재 상태와 다르다

계획서 §1-1은 "총 19종"으로 현황을 기록한다. 현재 `skills/`에는 신규 `harness-setup` 포함 20종이 있다.

이 항목은 작성 당시 현황 기록이라면 문제가 아니다. 하지만 계획서를 현재 상태 안내 문서처럼 읽으면 틀린 정보가 된다.

권장 수정:

- 계획서가 의사결정 기록이면 "작성 당시 기준"임을 더 명확히 둔다.
- 현재 운영 문서로도 쓸 의도라면 최종 상태 섹션을 추가해 20종으로 갱신한다.

### 3-2. 계획서의 최종 DoD 체크리스트가 완료 상태로 갱신되지 않았다

커밋 내역상 Phase 0~8 완료로 보이지만, 계획서 §7 체크리스트는 모두 미체크 상태다.

권장 수정:

- 계획서가 실행 기록까지 겸한다면 완료된 항목을 체크하고, 검증 결과를 짧게 덧붙인다.
- 순수 계획서로 남길 거라면 별도 "수행 결과 요약" 문서를 참조하도록 한다.

### 3-3. 계획서 내부에 복수 앱 `.gitignore` 관련 잔여 모순이 있다

계획서 D-5와 Phase 1은 "복수앱 최상위는 git init도 하지 않는다"는 정책을 확정한다. 그런데 리스크 표에는 `.gitignore`로 루트 미관리 파일 실수 커밋을 완화한다고 되어 있다.

복수 앱 최상위가 git repo가 아니라면 `.gitignore` 완화책은 직접적으로 성립하지 않는다.

권장 수정:

- 리스크 완화 문구를 ".docs/root-context 복사본 + harness-setup 단독 갱신 + 루트가 git repo가 아닌지 확인"으로 바꾼다.

### 3-4. `harness-bootstrap`은 계획서의 "스킬 간 참조 금지" 원칙과 충돌할 여지가 있다

계획서 C-3/기존0-6은 규칙을 스킬 간 참조하지 말고 각 스킬에 명시하라고 했다.

그런데 `skills/harness-bootstrap/SKILL.md`는 아래처럼 다른 스킬 내부 파일을 직접 참조한다.

- `../design-doc/templates/OUTPUT_V2.md`
- `../context-doc/prompts/analysis-claude.md`
- `../context-doc/prompts/analysis-instruction.md`
- `../context-doc/templates/CLAUDE.md.template`

이것은 "템플릿 재사용" 의도라면 합리적일 수 있지만, 계획서 문구 그대로 보면 스킬 간 참조 금지와 충돌한다.

권장 수정:

- 둘 중 하나를 선택해야 한다.
- `harness-bootstrap`만 예외로 둘 경우, 계획서와 스킬 본문에 "템플릿 재사용 예외"를 명시한다.
- 예외를 두지 않을 경우, 필요한 출력 규칙을 `harness-bootstrap` 내부 prompts/templates로 복제한다.

## 4. 원문 텍스트 의도 반영 여부

### 4-1. 잘 반영된 의도

- 단일/복수 애플리케이션 프로젝트를 모두 고려한다는 대원칙은 대부분의 스킬과 README에 반영되었다.
- 복수 앱에서 프로젝트 최상위는 git 관리하지 않고, `.docs`·앱 레포·하네스 정본 레포를 독립 관리한다는 구조가 README와 `harness-setup`에 반영되었다.
- `.docs/root-context/CLAUDE.md`, `.docs/root-context/AGENTS.md`를 원본 복사본으로 두고, 루트 `CLAUDE.md`, `AGENTS.md`는 `harness-setup`이 갱신한다는 관계가 반영되었다.
- `agent-sync`는 정본 pull이 아니라 횡적 미러만 담당하도록 경계가 재정의되었다.
- `custom-skill-design` 리네이밍과 자체 개선 항목은 핵심 파일에 반영되어 있다.
- `.claude/skills`와 `.agents/skills` 시드가 `harness-setup`, `custom-skill-design` 2종으로 맞춰져 있다.

### 4-2. 덜 반영되었거나 오해될 수 있는 의도

#### `.docs` 별도 형상관리 정책이 README에서 약하게 표현된다

원문과 D-5는 복수 앱의 `.docs`를 독립 git 레포로 관리하는 정책에 가깝다. README는 일부 문장에서 "별도 git 관리 권장", "팀 공유가 필요하면 별도 레포"처럼 선택 사항처럼 읽힌다.

권장 수정:

- 복수 앱 기본 정책은 ".docs는 별도 git 레포로 관리"라고 쓰고, 단 개인 실험/초기 세팅 단계에서만 아직 remote가 없을 수 있다고 보완한다.

#### Intro 문서는 복수 앱 구조가 충분히 드러나지 않는다

`Harness_Engineering_Intro.md`는 도입 배경 문서라 상세 경로를 모두 담을 필요는 없다. 하지만 현재는 `context-doc` 설명과 예시가 단일 앱 `.docs/instruction/*` 중심이라, 복수 앱 구조의 핵심인 `.docs/{앱}-context.md`, `.docs/{앱}/instruction/`, `.docs/root-context/`가 잘 드러나지 않는다.

권장 수정:

- 도입 문서에는 한 문단만 추가해도 된다.
- "단일 앱에서는 루트 문서와 `.docs/instruction`, 복수 앱에서는 `.docs/{앱}-context.md`와 `.docs/root-context`로 나뉜다" 정도를 명시한다.

#### README의 스킬 역할 표는 복수 앱 경로 분기를 생략한다

README 상단의 빠른 시작은 복수 앱을 설명하지만, 하단 Agent Skills 표의 `context-doc`, `harness-bootstrap`, prototype 계열 설명은 단일 앱 표현이 강하다.

권장 수정:

- 표의 역할 문구를 "프로젝트 유형에 따라 단일/복수 경로로 생성" 정도로 보강한다.

## 5. 우선 수정 권장 순서

1. `skills/design-prototype-docs/SKILL.md`의 복수 앱 prototype 예시 경로를 수정한다.
2. `skills/impl-fe-be-doc/SKILL.md`의 `rfp-design-input 문서` 표현을 `rfp-ingest 대화 컨텍스트`로 바꾼다.
3. `Harness_Engineering.md`의 `.instruction/*` 잔재와 "모든 스킬 C-1" 표현을 수정한다.
4. `Harness_Engineering_Intro.md`의 `basic-instruction.md` 예시를 실제 생성 파일명으로 바꾸고, 복수 앱 컨텍스트 구조를 한 문단 보강한다.
5. README에서 복수 앱 `.docs` 별도 git 관리 표현을 "권장"이 아니라 기본 정책으로 정리한다.
6. `harness-bootstrap`의 다른 스킬 템플릿 참조를 예외로 인정할지, 내부 복제로 바꿀지 결정한다.
7. `rfp-ingest`의 `allowed-tools: Agent`가 실제 도구명과 맞는지 확인하고 정리한다.

## 6. 결론

현재 리팩토링은 핵심 방향을 잘 반영했다. 특히 단일/복수 프로젝트 구조, `harness-setup` 중심 배포, `agent-sync` 역할 경계, root-context 구조는 전체적으로 맞다.

남은 문제는 대형 설계 오류라기보다 "마지막 잔여 정합성"에 가깝다. 가장 위험한 것은 `impl-fe-be-doc`의 `rfp-design-input 문서` 표현과 `design-prototype-docs`의 복수 앱 경로 예시 충돌이다. 이 둘은 실제 사용자가 잘못된 파일 위치나 존재하지 않는 입력 문서를 찾게 만들 수 있으므로 먼저 고치는 편이 좋다.
