## 1. 입력 문서 분석 — 먼저 읽어야 할 것

설계 문서를 받으면 출력 전에 반드시 아래 항목을 내부적으로 파악한다.
파악이 안 되는 항목은 사용자에게 묻는다 (→ 2장 참고).

### 1-A. 스케일 확인

| 스케일 | 판단 기준 | 작업지침 성격 |
|--------|-----------|--------------|
| 프로젝트 전체 | DB 스키마, 인프라, 배포, 여러 기능 모듈 포함 | Phase 0(인프라)부터 전체 Phase 구성 |
| 기능 단위 | 특정 기능(온보딩, 결제 등) 중심, API 명세 포함 | 해당 기능 Phase만 구성 |
| 화면 단위 | 특정 페이지 UI/UX, 컴포넌트 구조 중심 | FE 위주, BE는 API 연동 수준 |
| 컴포넌트/로직 | 단일 컴포넌트, 훅, 유틸 함수 수준 | FE-XX 태스크 1~3개 수준 |

### 1-B. 추출해야 할 핵심 정보

```
[ ] BE 모듈 목록 (파일/폴더 단위로)
[ ] FE 페이지 / 컴포넌트 목록
[ ] API 엔드포인트 전체 (메서드, 경로, 설명)
[ ] WebSocket 엔드포인트 및 메시지 타입
[ ] DB 테이블 / 모델 목록
[ ] 외부 서비스 / 인프라 의존성 (Docker, Ollama 등)
[ ] 환경변수 목록 (개발/배포 분리 여부)
[ ] 기술 스택 (언어, 프레임워크, 버전)
[ ] 핵심 로직 (설계 문서에서 강조된 부분)
[ ] 주의사항 / 제약 조건
```

### 1-C. Phase 분할 기준 파악

아래 질문으로 Phase 경계를 결정한다.

- **독립 실행 가능한가?** — 각 Phase는 이전 Phase 없이는 실행 불가능해야 한다
- **테스트 가능한가?** — 각 Phase 끝에서 사람이 눈으로 확인할 수 있는 결과물이 있어야 한다
- **BE와 FE가 함께 완성되는가?** — BE만 또는 FE만 완성되는 Phase는 페어가 아니다

## 2. 사용자에게 되물어야 할 것

설계 문서를 분석한 후, 출력 전에 아래 항목 중 **불명확한 것만** 골라 한 번에 묻는다.
한 번에 3개 이상 묻지 않는다. 우선순위가 높은 것부터 선별.

### 2-A. 반드시 확인해야 할 것 (🔴 출력 전 필수)

```
1. 개발 환경과 배포 환경이 구분되어 있는가?
   → 환경별로 동작이 달라지는 곳(noVNC, headless 등)이 있으면 Phase에서 별도 처리 필요

2. AI / 비동기 처리가 포함된 기능이 있는가?
   → 있다면 WebSocket / 폴링 / 스켈레톤 UI 등 처리 방식 확인

3. 설계 문서에 "미정"이거나 "열린 결정 사항"으로 표시된 항목이 있는가?
   → 있다면 구현 전에 결정이 필요한 것과 나중에 해도 되는 것 분류
```

### 2-B. 있으면 확인하면 좋은 것 (🟡 선택)

```
4. 기존 코드베이스가 있는가?
   → 있다면 신규 파일 [NEW]과 수정 파일 [MODIFY]을 구분해서 표시

5. 테스트 방식에 제약이 있는가?
   → 예: 실제 외부 API를 쓸 수 없어 Mock 필요, 특정 환경에서만 동작 등

6. 작업지침을 읽는 대상이 사람인가, AI Agent인가, 둘 다인가?
   → Agent 전용이면 코드 스니펫을 더 상세하게, 사람 전용이면 의도 설명 위주로
```

### 2-C. 묻지 않아도 되는 것

```
- 기술 스택 (설계 문서에 있으면 그대로 사용, 없으면 AI가 제안)
- Phase 수 (분석 후 자동 결정)
- 태스크 세분화 수준 (스케일에 따라 자동 결정)
```

## 3. Phase 분할 사고 방식

### 3-A. 분할 패턴

대부분의 프로젝트는 아래 순서로 Phase가 구성된다.
**스케일이 프로젝트 전체가 아니면 Phase 0을 생략하고 해당 기능부터 시작.**

```
Phase 0: 인프라 & 환경 세팅
  → DB 컨테이너, 개발 환경 세팅, 폴더 구조
  → FE 없음 (BE/INF만)
  → 검증: 서버 기동 확인

Phase 1: 데이터 기반 + 가장 단순한 CRUD
  → BE: DB 모델 전체 + 기본 API
  → FE: 라우터 + 목록/추가 UI
  → 검증: FE → API → DB 파이프라인 확인

Phase N: 핵심 기능 (설계의 G섹션 "집중 로직" 기준)
  → BE: 핵심 로직 모듈
  → FE: 해당 기능 UI + 실시간 연동
  → 검증: 기능 E2E 동작

마지막 Phase: 배포 환경 (Dockerfile, compose.yaml)
  → BE Dockerfile + start.sh
  → FE Dockerfile + .env.production
  → 검증: 컨테이너 기동 + 전체 플로우 재확인
```

### 3-B. 페어 구성 규칙

```
✅ 올바른 페어
  BE: POST /sites API 구현
  FE: 사이트 추가 폼 + 목록 렌더링
  → 함께 완성되면 "사이트 추가" 기능이 동작함

❌ 잘못된 페어
  BE: DB 모델 5개 정의 + API 12개 구현
  FE: 전체 페이지 4개 구현
  → 너무 많아서 어디서 막혔는지 알 수 없음

❌ 잘못된 페어
  BE: 온보딩 API만 완성
  FE: 아직 없음
  → FE 없이는 사람이 결과를 확인할 수 없음
```

### 3-C. 복잡한 기능의 Phase 내 순서

Phase 안에서 BE 태스크가 여러 개일 때 구현 순서:

```
1. 의존성 없는 유틸/헬퍼 먼저 (AI chain, 변환 함수 등)
2. 데이터 접근 레이어 (DB 모델, 세션)
3. 핵심 비즈니스 로직 (크롤러, 매크로 엔진 등)
4. API 엔드포인트 / WebSocket (FE와 연결점)

FE는 BE API가 준비된 후에 연동 구현.
단, UI 퍼블리싱(레이아웃, 컴포넌트 껍데기)은 BE와 병렬 작업 가능.
```

## 4. 검증 시나리오 작성 규칙

각 Phase의 `✅ 통합 검증`은 아래 형식을 따른다.

### 4-A. 형식

```markdown
### ✅ Phase N 통합 검증

**시나리오**: [한 줄 요약 — 무엇을 확인하는 Phase인지]

[번호 매긴 단계별 시나리오. 사람이 직접 수행하는 것처럼 서술]
1. [URL 접속 또는 명령 실행]
2. [사용자 액션]
3. [기대 동작 확인]
4. [DB 또는 로그에서 확인]

**합격 조건** : [조건 A] ✅ / [조건 B] ✅ / [조건 C] ✅
```

### 4-B. 좋은 검증 vs 나쁜 검증

```
✅ 좋은 검증
  "DB에서 SELECT * FROM sites; 실행 — 레코드 존재 확인"
  "LogViewer에 SUCCESS 타입 로그 실시간 표시 확인"
  "VITE_NOVNC_URL 미설정 시 텍스트 안내 렌더링, 설정 시 iframe 렌더링"

❌ 나쁜 검증
  "정상적으로 동작하는지 확인" (모호함)
  "에러 없이 실행되는지 확인" (무엇이 실행되는지 불명확)
  "BE가 잘 동작하는지 확인" (FE에서 확인 불가)
```

### 4-C. 합격 조건 개수

- Phase가 단순할수록 합격 조건 적게 (2~3개)
- 핵심 기능 Phase는 4~5개
- 배포 Phase는 서비스별로 1개씩

## 5. 태스크 작성 규칙

### 5-A. 태스크 ID 체계

```
INF-XX : 인프라 (디렉토리 구조, 컨테이너, Dockerfile 등)
BE-XX  : 백엔드 (Python/FastAPI 모듈 단위)
FE-XX  : 프론트엔드 (React 컴포넌트/페이지/훅 단위)
```

번호는 Phase 내 순서가 아닌 **전체 문서 통합 일련번호**로 부여한다.
(예: Phase 1에서 BE-01~04, Phase 2에서 BE-05~06으로 이어짐)

### 5-B. 태스크 필수 항목

```markdown
### [태스크 ID] · [태스크 이름] (`파일 경로`)

**의존** : [선행 태스크 ID 목록]

[구현 내용 — 무엇을 왜 만드는지]

**Agent 지시** : (있을 때만)
[코드 스니펫 또는 구체적 지침]

**검증 기준** : [이 태스크 단독으로 완료 판단하는 조건]
```

### 5-C. Agent 지시 코드 스니펫 작성 기준

```
포함해야 할 것:
- 핵심 함수 시그니처 / 클래스 구조
- 중요한 환경변수 처리 방식
- 의도적인 설계 결정 (why)
- 주의해야 할 분기 처리

포함하지 말아야 할 것:
- 완성된 구현 코드 전체 (작성 여지를 남긴다)
- 표준 보일러플레이트 (import, 주석 등)
- 이미 다른 태스크에서 설명한 내용 반복
```

## 6. 설계 문서에서 자주 나오는 함정 — 검토 체크리스트

설계 문서를 받으면 출력 전에 아래를 스스로 점검한다.

### 6-A. 기술적 함정

```
[ ] 클릭 캡처 방식: JS 주입(CDP) vs Playwright 이벤트 리스너 — 설계 문서 기준 확인
[ ] 로그인 자동화 여부: 세션 쿠키 재사용인지, 실제 로그인 자동화인지
[ ] headless 값 하드코딩 여부: 반드시 환경변수로 분기해야 함
[ ] LLM 클라이언트 직접 호출 여부: 팩토리 함수로 추상화해야 함
[ ] DOM 크기 제한 여부: LLM에 전체 HTML 넘기면 토큰 초과 위험
[ ] CORS 설정 여부: FE-BE 다른 포트라면 필수
[ ] WebSocket 엔드포인트 수: 온보딩용 / 운영용 분리 여부
```

### 6-B. 문서 함정

```
[ ] API 경로 표기 오류: 실제 설계 문서의 엔드포인트 목록과 대조
[ ] 환경별 동작 차이 누락: 개발/배포 환경에서 다른 동작은 반드시 명시
[ ] DB 모델 누락: 설계 문서에 있는 모든 테이블이 BE-XX에 반영됐는지 확인
[ ] Alembic / 마이그레이션 태스크 누락: DB 모델 정의 다음에 반드시 포함
[ ] .gitignore 생성 태스크 누락: 세션 파일, .env 파일 보안 처리
[ ] start.sh 환경 혼용: 배포 전용임을 명시
[ ] noVNC 분기 누락: 개발/배포 환경에서 FE 렌더링이 다른 경우
```

### 6-C. 구조적 함정

```
[ ] Phase가 너무 큰가? — 한 Phase에 API 5개 이상이면 쪼갠다
[ ] 페어가 아닌가? — BE만 있거나 FE만 있는 Phase는 합치거나 분리
[ ] 검증이 너무 늦은가? — 핵심 기능은 최대한 앞 Phase에서 검증
[ ] 미결 사항이 구현을 막는가? — 결정 필요 항목은 해당 태스크 전에 표시
```

## 8. 이 방법론을 사용할 때 Agent에게 건네는 방법
