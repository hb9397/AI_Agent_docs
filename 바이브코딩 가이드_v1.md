# AI Agent + 웹 AI로 설계부터 구현까지 — 내가 쓰는 방식 v1

> 바이브코딩 학습 증적 / 개인 워크플로우 정리
> 이런 방식도 있다는 소개 목적의 글

---

## 왜 이 글을 쓰는가

- AI Agent(Claude Code 등)를 써서 코드를 짜다 보면 결국 **"AI에게 무엇을 어떻게 알려주는가"** 가 결과물의 품질을 결정한다는 걸 계속 느낌
- 아무 준비 없이 "이거 만들어줘" 하면 AI도 대충 만들고, 나도 방향을 잃음
- 그래서 설계 → 컨텍스트 구성 → 구현의 흐름을 템플릿화하고, 반복 가능한 형태로 정리하게 됨
- 이 글은 그 방식을 정리해두는 학습 증적 겸, 비슷한 방식을 찾는 사람에게 참고가 될 수 있도록 쓰는 소개 글

---

## 이 레포지토리에서 사용하는 도구들

크게 두 종류의 AI 환경에서 사용하는 도구들을 모아둔 곳

```
ai_agent_forms/
├── Agent Skills/        ← IDE 안의 AI Agent (Claude Code) 에서 사용하는 Skill 모음
├── Docs Skills/         ← 웹 AI (Gemini Gems, Claude Project) 에서 사용하는 템플릿 문서 모음
└── example/             ← 아래 1~4단계를 실제로 거쳐 나온 산출물 예시들
```

### Agent Skills — IDE Agent용 Skill

- Claude Code 같은 IDE 내 Agent에서 `/skill명` 형태로 호출해서 쓰는 자동화 워크플로우 묶음
- 각 Skill은 `SKILL.md` + `prompts/` + `templates/` 구조로 이루어짐
- Agent가 Skill을 읽고 정해진 Step대로 실행 → 사람이 확인 후 다음 Step 진행하는 방식

| Skill | 한줄 설명 |
| ----- | --------- |
| `design-doc` | 설계 인터뷰 → 구조화된 설계 문서(OUTPUT) 자동 도출 |
| `context-doc` | 설계 문서 → `CLAUDE.md` + `basic-instruction.md` 생성 |
| `impl-doc` | 설계 문서 → FE/BE 페어 기능 단위 작업지침서 생성 |
| `agent-sync` | Agent 문서(CLAUDE.md 등) · Skill 변경 감지 및 동기화 |
| `code-comment` | 변경된 코드 파일에 한글 주석 자동 작성 |
| `commit` | 변경 내용 분석 후 Conventional Commits 규칙 커밋 메시지 생성 |
| `doc-audit` | 실제 코드 ↔ AI 문서 간 괴리 분석 및 수정 제안 |
| `frontend-design` | 고품질·개성 있는 프론트엔드 UI 코드 생성 |
| `multi-review` | 보안·성능·유지보수·테스트 4개 관점 멀티 페르소나 코드 리뷰 |
| `pre-commit` | 커밋 전 에러 처리·민감 정보·TODO 형식 등 규칙 위반 검사 |
| `skill-designer` | 반복 워크플로우를 인터뷰 기반으로 새 Skill로 설계·생성·테스트 |

### Docs Skills — 웹 AI용 템플릿 문서

- Gemini Gems 또는 Claude Project(프로젝트에 문서를 올려두는 기능)에 넣어서 쓰는 인터뷰 템플릿 문서들
- 웹 AI가 이 문서를 시스템 프롬프트처럼 참조해서, 사용자와 인터뷰를 진행하고 산출물을 뽑아주는 방식

| 문서 폴더 | 용도 |
| --------- | ---- |
| `설계문서_도출/` | 프로젝트·화면·기능 스케일별로 설계 인터뷰를 진행해서 설계 문서를 도출 |
| `구현작업_지시서_도출/` | 완성된 설계 문서를 기반으로 구현 작업지침서(Phase별 태스크)를 도출 |

## 작업 스케일 — 먼저 이걸 정하고 시작

- 작업 전 "지금 이게 어느 수준의 일인가"를 먼저 판단하는 게 중요
- 스케일에 따라 어떤 단계를 거칠지, 어떤 문서를 만들지가 달라짐

| 스케일 | 기준 | 예시 |
|--------|------|------|
| **프로젝트** | 시스템 전체를 새로 만드는 수준 | 신규 서비스, 주요 서브시스템 신규 구축 |
| **화면 단위** | 특정 페이지나 뷰 하나를 만드는 수준 | 대시보드 화면, 설정 페이지 |
| **기능 단위** | 독립적으로 동작하는 기능 하나 | 예약 매크로 실행, 로그인 기능 |
| **세부 로직** | 함수·컴포넌트 내부 알고리즘 수준 | DOM 변경 감지 로직, 재시도 에이전트 |

- 세부 로직 수준은 아래 1~4단계를 다 거치지 않아도 됨 — 바로 구현해도 무방한 경우가 많음
- 프로젝트 스케일일수록 1~4단계를 꼼꼼히 밟을수록 후반부 삽질이 줄어드는 걸 경험함

## 핵심 흐름 — 4단계

```
아이디어 또는 요구사항
        │
        ▼
  [1단계] 설계 문서 도출          ← 웹 AI / Agent 둘 다 가능
        │ OUTPUT: 설계 스키마 문서
        ▼
  [2단계] 우려 사항 고도화 인터뷰  ← 주로 웹 AI
        │ OUTPUT: 보완된 설계 문서
        ▼
  [3단계] Context 문서 생성        ← 스케일에 따라 선택
        │ OUTPUT: CLAUDE.md + basic-instruction.md
        ▼
  [4단계] 작업지시서 도출          ← 웹 AI / Agent 둘 다 가능
        │ OUTPUT: Phase별 FE/BE 페어 작업지침서
        ▼
       구현
        │
        ├─ 상황에 따라 보조 Skill 사용
        └─ Phase 완료 → 다음 Phase 반복
```

## 1단계 — 설계 문서 도출

### 설계 문서 없이 바로 구현하면 생기는 일

- "이거 만들어줘" 한 줄로 시작하면 AI가 제멋대로 기술 스택을 고르고, 내가 생각한 구조와 전혀 다른 걸 만들어옴
- 설계 문서가 있으면 AI가 내 머릿속과 같은 그림을 공유한 상태에서 코드를 짜게 됨
- 또 설계하면서 내가 미처 생각 못했던 분기나 예외 케이스가 드러나는 부수 효과도 있음

### 1단계 도구

| 환경 | 도구 | 방식 |
| ---- | ---- | ---- |
| 웹 AI | `Docs Skills/설계문서_도출/` 문서를 Gems·Project에 업로드 | AI가 템플릿 기반으로 인터뷰 진행 후 산출물 출력 |
| Agent (IDE) | `Agent Skills/design-doc` 스킬 호출 | `/design-doc` 입력 → Step별 인터뷰 → 문서 도출 |

### design-doc 스킬이 하는 일

- **Step 1** — 스케일 확인 (프로젝트/화면/기능/컴포넌트) + 기존 문서(PRD, DB 설계서 등) 유무 확인
- **Step 2** — 인터뷰 진행
  - 섹션 순서대로 한 번에 하나씩 질문
  - 모호한 답변이 오면 반박하고 더 파고듦 — 그냥 넘어가지 않음
  - "뒤집기 확인" 섹션(H섹션)을 반드시 진행해서 설계의 약점을 미리 드러냄
  - 기존 문서가 있으면 인터뷰 답변과 교차 검증해서 불일치를 짚어냄
- **Step 3** — 스케일에 맞는 OUTPUT 양식으로 문서 작성
- **Step 4** — 초안 출력 후 사용자 확인 → 수정 반영 → 파일 저장

### OUTPUT 문서 구조 (`example/Acro.md` 참고)

```
01. 개요          스케일, 핵심 목표, 핵심 설계 결정
02. 핵심 흐름     단계별 동작, 분기 처리, 상태 전환
03. 집중 로직     가장 복잡한 핵심 알고리즘, 코드 스니펫 수준으로 명세
04. 파일 구조     디렉토리 트리, 각 파일의 책임
05. 데이터 구조   DB 테이블, 주요 컬럼, 인덱스
06. 라이브러리    사용 기술 스택 및 외부 구성 요소 목록
07. 대안 전략     복잡도 높은 항목의 구현 방향 및 트레이드오프
08. 주의사항      법적 이슈, 보안 제약, 성능 주의점
09. 열린 결정     아직 결정 못한 항목들
```

- 확실하지 않은 항목은 비워두지 않고 `미정 — [이유]` 로 표시
- 이 문서가 이후 2~4단계의 모든 입력이 됨
- Agent가 오해 없이 읽을 수 있도록 모호한 표현과 중복 서술을 최대한 없애는 것이 핵심

### 다운스트림 연계

```
design-doc OUTPUT
    ├─→ context-doc  →  CLAUDE.md + basic-instruction.md  (3단계)
    └─→ impl-doc     →  FE/BE 페어 작업지침서              (4단계)
```

## 2단계 — 우려 사항 고도화 인터뷰

### 왜 설계 문서만으로는 부족한가

- 설계 문서가 완성됐다고 해서 바로 구현하면, 나중에 "이 라이브러리 이렇게 쓰는 게 맞나?" "이 구조로 확장이 되나?" 같은 의문이 터짐
- 처음 써보는 기술이 들어간 경우, 그 기술의 함정과 제약을 미리 파악해두지 않으면 구현 중반에 구조를 갈아엎는 상황이 생김
- 1단계 OUTPUT을 들고 웹 AI와 한 번 더 대화하면서 이 부분들을 미리 정리하는 단계

### 고도화 대상 예시

| 구분 | 확인하는 내용 |
| ---- | ------------ |
| **기술 스택 불확실성** | "LangGraph가 꼭 필요한가, LangChain만으로 충분한가?" / "patchright + playwright-stealth 이 두 개가 같이 필요한 이유는?" |
| **핵심 로직 구현 가능성** | "토큰 제한 내에서 LangGraph 복구 에이전트가 실제로 동작하는가?" / "DOM 전체를 LLM에 넘기면 안 된다는 걸 어떻게 처리할 것인가?" |
| **확장 가능성** | "사이트를 추가할 때 `site_name` 키 하나로 멀티 사이트를 관리하는 구조가 실제로 버텨주는가?" |
| **처음 쓰는 기술** | 공식 문서·예제를 함께 붙여넣고 "이 프로젝트 구조에 어떻게 적용하면 되는가"를 물어봄 |

### 진행 방식

- 1단계 OUTPUT 문서를 웹 AI(Gemini Deep Research, Claude)에 붙여넣고, 특정 섹션을 중심으로 추가 인터뷰 진행
- 처음 쓰는 기술은 공식 문서 URL이나 예제 코드를 함께 올려서 구체적인 적용 방법을 확인
- 고도화된 내용은 1단계 문서의 대안 전략·주의사항 섹션에 반영하거나 별도 섹션으로 추가
- 이 단계의 결과물은 별도 파일이 아니라 1단계 설계 문서가 더 풍부해지는 형태

## 3단계 — Context 문서 및 기본 Instruction 생성

### 컨텍스트 문서가 없으면 생기는 문제

- AI Agent는 대화창에 올려진 정보만 보고 코드를 짬 — 컨텍스트가 없으면 매번 같은 설명을 반복해야 하고, 그래도 일관성이 없어짐
- `CLAUDE.md` 같은 컨텍스트 문서를 프로젝트 루트에 두면 Agent가 매 요청마다 이 문서를 참조해서 프로젝트 구조와 규칙을 이미 알고 있는 상태로 코드를 짬
- 특히 금지 패턴이나 아키텍처 제약을 `basic-instruction.md`에 명시해두면, 내가 따로 말하지 않아도 Agent가 그 규칙에 맞는 코드를 생성함

### 스케일별 적용 여부

| 스케일 | CLAUDE.md | basic-instruction.md |
| ------ | --------- | -------------------- |
| 프로젝트 | 필수 | 필수 |
| 화면 단위 | 선택 (기존 갱신으로 대체 가능) | 선택 |
| 기능 단위 | 선택 (단순 내용 추가 정도) | 선택 |
| 세부 로직 | 보통 생략 | 보통 생략 |

### 3단계 도구

| 환경 | 도구 | 방식 |
| ---- | ---- | ---- |
| 웹 AI | `Docs Skills/설계문서_도출/` 내 `BE_CONTEXT` / `FE_CONTEXT` 문서 활용 | 템플릿에 설계 문서 내용을 채워서 생성 |
| Agent (IDE) | `Agent Skills/context-doc` 스킬 | `/context-doc` → 설계 문서 입력 → 두 파일 자동 생성 |

### context-doc 스킬이 하는 일

- **Step 1** — 설계 문서 수집 (없으면 요청)
- **Step 2-A** — `CLAUDE.md` 분석: 프로젝트 맥락·아키텍처·통신 규칙·기술 스택 섹션 추출
- **Step 2-B** — `basic-instruction.md` 분석: 금지 패턴은 반드시 **금지 패턴 + 이유 + 대안** 삼위일체로 작성
- **Step 3** — 두 문서 초안 동시 생성
- **Step 4** — 대화창에 미리보기 출력 후 승인 요청
- **Step 5** — 승인 시 `CLAUDE.md` (프로젝트 루트) / `.instruction/basic-instruction.md` 저장

### basic-instruction.md 금지 패턴 작성법 — 삼위일체 예시

`basic-instruction.md`에서 가장 효과가 큰 부분이 금지 패턴이다. 단순히 "하지 마라"만 쓰면 Agent가 왜 안 되는지 모르고 돌아서 같은 실수를 반복한다. **패턴 + 이유 + 대안** 세 가지를 묶어서 써야 Agent가 규칙을 이해하고 스스로 판단할 수 있다.

```markdown
## ❌ 금지 패턴

### 1. DOM 전체를 LLM에 직접 전달
- **패턴**: `llm.invoke(full_html_string)`
- **이유**: 토큰 초과로 컨텍스트 창을 초과하거나 응답 품질이 급격히 낮아짐
- **대안**: `missing_selectors` 주변 영역만 추출하거나 3,000자로 잘라서 전달

### 2. 라우터에 비즈니스 로직 작성
- **패턴**: `routers/macro.py` 안에 Playwright 실행 코드 직접 삽입
- **이유**: 라우터는 요청/응답 변환만 담당. 비즈니스 로직이 섞이면 테스트·재사용 불가
- **대안**: `macro/engine.py`에 로직을 두고 라우터에서는 호출만

### 3. 환경변수 하드코딩
- **패턴**: `headless=False`, `max_retries=3` 등 값을 코드에 직접 작성
- **이유**: 개발/배포 환경 분리 불가, 변경 시 코드 수정 필요
- **대안**: `os.getenv("HEADLESS", "false")` 형태로 환경변수로 읽어서 처리
```

이렇게 쓴 금지 패턴은 설계가 바뀌어도 "왜" 부분은 그대로 유지되는 경우가 많아서, 문서 수명이 길어진다.

### design-doc OUTPUT의 섹션 → 두 문서로의 매핑

| design-doc 섹션 | 생성되는 문서 |
| --------------- | ------------ |
| 01 개요, 05 데이터, 06 파일 구성 | CLAUDE.md — 프로젝트 맥락·아키텍처 |
| 02 동작 흐름, 07 라이브러리 | CLAUDE.md — 통신 규칙·기술 스택 |
| 03 집중 로직, 04 인터페이스 | basic-instruction.md — 아키텍처 제약 |
| 10 주의사항, 12 열린 결정 | basic-instruction.md — 금지 목록·미결 항목 |

### 생성되는 두 문서의 역할 (`example/CLAUDE.md`, `example/basic-instruction.md` 참고)

**`CLAUDE.md`** — "이 프로젝트가 무엇인가"를 Agent에게 알려주는 문서
- 프로젝트 개요, 기술 스택 표, 아키텍처 다이어그램, API 엔드포인트 목록
- DB 테이블 구조, 핵심 로직 흐름, 실행 방법, 환경변수 목록
- Agent가 코드를 짜기 전에 "프로젝트의 전체 그림"을 파악하는 데 쓰는 문서

**`.instruction/basic-instruction.md`** — "어떻게 코드를 짜야 하는가"를 Agent에게 알려주는 문서

- 모듈 분리 원칙, 각 모듈의 책임과 금지 행위
- FastAPI 규칙, DB 쿼리 규칙, 외부 라이브러리 사용 규칙
- 금지 목록: 패턴 + 이유 + 대안을 세트로 명시
- Agent 전용 코딩 규칙 (ex. "코드 작성 시 새 주석 추가 금지")

## 4단계 — 작업지시서 도출 → 구현 시작

### 작업지침서가 없으면 생기는 문제

- 설계 문서가 있어도 "자, 이제 구현해줘" 하면 AI가 어디서부터 어떤 순서로 해야 할지 모르고 엉뚱한 것부터 시작하거나, FE와 BE가 따로 놀거나, 검증 없이 다음 기능으로 넘어가는 문제가 생김
- 작업지침서는 Phase라는 단위로 BE+FE를 페어로 묶어서, 한 Phase가 끝나면 사람이 눈으로 확인 가능한 결과물이 나오도록 설계함
- 이 문서가 있으면 내가 "Phase 2 해줘" 하면 Agent가 어떤 파일을 어떤 순서로 어떻게 만들어야 하는지 이미 알고 있음

### 4단계 도구

| 환경 | 도구 | 방식 |
| ---- | ---- | ---- |
| 웹 AI | `Docs Skills/구현작업_지시서_도출/` | 템플릿 문서 + 설계 문서로 인터뷰 진행 후 지침서 도출 |
| Agent (IDE) | `Agent Skills/impl-doc` 스킬 | `/impl-doc` → 설계 문서 입력 → Phase 분할 → 태스크 작성 |

### impl-doc 스킬이 하는 일

- **Step 1** — 설계 문서 수집 + 불명확한 항목 최대 3개만 추가 질문
- **Step 2** — Phase 분할 설계
  - 각 Phase는 이전 Phase 없이 실행 불가능해야 함
  - 각 Phase 끝에 사람이 눈으로 확인 가능한 결과물 필수
  - BE만 또는 FE만 완성되는 Phase는 금지 — 합치거나 재분할
  - 한 Phase에 API 5개 이상이면 쪼갬
  - 프로젝트 스케일이 아니면 Phase 0(인프라)은 생략
- **Step 3** — 태스크 작성 (ID, 의존, 구현 내용, Agent 지시, 검증 기준)
- **Step 4** — 각 Phase의 통합 검증 시나리오 작성 (단계별, 합격 조건 구체적으로)
- **Step 5** — 함정 체크리스트 실행 (누락 항목 자동 보완)
- **Step 6** — 초안 출력 후 사용자 확인 → 승인 시 파일 저장

### 태스크 ID 체계

| ID 형식 | 의미 |
| ------- | ---- |
| `INF-XX` | 인프라·환경 세팅 |
| `BE-XX` | 백엔드 구현 태스크 |
| `FE-XX` | 프론트엔드 구현 태스크 |

- 번호는 Phase 내 순서가 아니라 전체 문서 통합 일련번호

### 작업지침서 구조 (`example/ACRO_Impl.md` 참고)

```
읽는 법
  태스크 ID / 의존 / Agent 지시 / 통합 검증 의미 안내

Phase 0 — 인프라 & 개발 환경  (프로젝트 스케일만)
  INF-01 · 디렉토리 구조 생성
  INF-02 · DB 초기화 및 마이그레이션 설정
  INF-03 · Docker/Podman 구성
  ✅ 통합 검증: 각 서비스가 정상 기동되는지 확인

Phase 1 — [첫 번째 기능 페어]
  BE-01 · 백엔드 태스크
    의존: INF-02
    구현: 무엇을 왜 만드는지
    Agent 지시: 핵심 시그니처, 주의 분기
    검증 기준: 어떤 상태면 완료인가
  FE-01 · 프론트엔드 태스크
    의존: BE-01
    ...
  ✅ 통합 검증: 구체적인 시나리오 (클릭 → 어떤 응답 → 어떤 화면)

Phase 2 — ...
```

## 구현 중 상황에 따라 쓰는 보조 Skill들

4단계 구현이 진행되면서 필요할 때 꺼내 쓰는 도구들

### 코드 품질

| Skill | 언제 쓰는가 | 무엇을 하는가 |
| ----- | ---------- | ------------ |
| `code-comment` | 기능 구현 완료 후 변경 파일이 생겼을 때 | git diff 또는 mtime으로 변경 파일 감지 → 언어·프레임워크 자동 감지 → 한글 주석 작성 → 미리보기 후 승인 시 반영 |
| `multi-review` | Phase 완료 후, 또는 코드가 찜찜할 때 | 보안·성능·유지보수·테스트 4개 관점의 sub-agent가 병렬로 리뷰 → 심각도별 분류 후 종합 리포트 |
| `pre-commit` | 커밋 전 | 빈 catch 블록, 하드코딩 민감 정보, 기한 없는 TODO, 테스트 누락 등 규칙 위반 검사 |
| `commit` | 커밋할 때 | staged 변경 분석 → Conventional Commits 메시지 생성 → 복합 변경이면 분리 커밋 제안 |

### 문서 관리

| Skill | 언제 쓰는가 | 무엇을 하는가 |
| ----- | ---------- | ------------ |
| `doc-audit` | 코드가 설계와 달라졌다는 느낌이 올 때, 라이브러리가 바뀌었을 때 | 의존성 괴리·코드 패턴 괴리·규칙 위반 사례를 병렬 분석 → 변경 제안서 출력 → 승인 후 반영 |
| `agent-sync` | CLAUDE.md나 Skill을 수정한 후 다른 Agent 환경에도 반영할 때 | git diff 또는 mtime으로 변경된 문서·Skill 감지 → 대상 환경에 동기화 |

### UI / Skill 관리

| Skill | 언제 쓰는가 | 무엇을 하는가 |
| ----- | ---------- | ------------ |
| `frontend-design` | 화면 UI를 구현할 때 | 목적·톤앤매너·차별화 포인트를 먼저 정의하고 → 그에 맞는 개성 있는 UI 코드 생성. 보라색 그라데이션 같은 AI 클리셰 디자인을 의도적으로 피함 |
| `skill-designer` | 반복되는 워크플로우를 Skill로 만들고 싶을 때 | 인터뷰 → 설계 → 초안 생성 → eval 루프 테스트 → description 트리거 최적화까지 Skill 생애주기 전체 커버 |

## 웹 AI vs Agent(IDE) — 어떤 상황에 무엇을 쓰는가

| 상황 | 웹 AI (Gemini, Claude.ai) | Agent (Claude Code) |
| ---- | ------------------------ | ------------------- |
| 초기 아이디어 브레인스토밍 | ✅ | — |
| 설계 인터뷰 (긴 대화) | ✅ | ✅ (design-doc 스킬) |
| 처음 쓰는 기술 조사 (웹 검색 포함) | ✅ | — |
| 우려 사항 고도화 인터뷰 | ✅ | — |
| Context 문서 생성 | ✅ (템플릿 활용) | ✅ (context-doc 스킬) |
| 작업지침서 도출 | ✅ (템플릿 활용) | ✅ (impl-doc 스킬) |
| 실제 코드 작성 | — | ✅ |
| 코드 리뷰·주석·커밋 | — | ✅ |
| 문서 동기화·괴리 분석 | — | ✅ |

- 웹 AI는 **긴 대화와 탐색**에 강함 — 아직 방향이 정해지지 않은 초반 단계
- Agent(IDE)는 **실제 파일과 코드에 접근**할 수 있어서 구현 단계에 강함
- 두 환경이 서로 대체하는 게 아니라 **역할을 나눠서 순서대로 사용**하는 것이 핵심

## 바이브코딩 시 도움이 됐던 것들 — 따로 소개

아래는 이 워크플로우와 별개로, 바이브코딩 전반에서 써봤거나 알게 된 것들

### 1 — 스펙 문서를 먼저 쓰면 AI가 훨씬 잘 따라옴

- "스펙 주도 개발(Spec-First)"이라고 불리기도 하는 방식
- AI에게 코드를 짜게 하기 전에 요구사항을 명세한 문서(또는 테스트 케이스)를 먼저 만들고, 그것을 기준으로 구현을 지시함
- 내가 쓰는 설계 문서 + 작업지침서 방식과 같은 맥락
- 이게 없으면 AI가 "그럴듯한 코드"를 만들고 내가 사후에 맞는지 확인하게 되는데, 이게 훨씬 피곤한 방식임

### 2 — 컨텍스트 창을 낭비하지 않는 대화 설계

- AI와 대화가 길어지면 초반 내용이 잊혀지면서 일관성이 깨짐
- 긴 탐색 대화는 웹 AI에서, 실제 구현 지시는 Agent에서 짧고 명확하게 나눠서 하는 게 효과적
- Agent에게는 "Phase 2의 BE-05 태스크를 구현해줘. 작업지침서 참고해" 같은 짧고 구체적인 지시가 "이런저런 기능을 만들어줘" 보다 훨씬 결과가 좋음
- 대화 초반에 참조 문서(`CLAUDE.md`, 작업지침서)를 언급해두면 Agent가 알아서 읽고 컨텍스트를 채움

### 3 — AI가 만든 코드에 대한 체크포인트를 두는 습관

- AI가 Phase 하나를 완성하면 무조건 검증 시나리오를 직접 돌려봄 — 코드만 읽고 "맞겠지"하고 넘어가면 나중에 터짐
- git으로 체크포인트를 남겨두면 AI가 이상한 방향으로 갔을 때 되돌리기 쉬움
- `pre-commit` 스킬처럼 커밋 전 검사 단계를 두는 것도 이 맥락

### 4 — 한 번에 너무 많이 시키지 않기

- "Phase 1부터 Phase 5까지 한 번에 다 짜줘"는 거의 항상 중간에 무언가가 꼬임
- Phase 하나, 태스크 하나 단위로 시키고, 확인하고, 다음으로 넘어가는 리듬이 실제로 더 빠름
- AI가 한 번에 너무 많은 걸 하면 앞에서 만든 것과 뒤에서 만든 것이 인터페이스가 안 맞는 경우가 생김

### 7 — Agent에게 지시하는 방법이 결과를 가른다

- Agent에게 무언가를 시킬 때, 지시의 품질이 코드의 품질을 거의 그대로 결정함
- 좋은 지시는 짧고, 참조할 문서를 명시하고, 범위가 한정돼 있음
- 나쁜 지시는 길고 두루뭉술하거나, 여러 요청이 섞여 있거나, Agent가 어디서 어디까지 해야 하는지 알 수 없음

```text
❌ 나쁜 지시
"로그인 기능이랑 대시보드 화면이랑 API도 연결해줘. 근데 에러 처리도 잘 해주고
 코드 구조도 깔끔하게 짜줘"

✅ 좋은 지시
"작업지침서(impl.md)의 Phase 2 — BE-03 태스크를 구현해줘.
 CLAUDE.md와 basic-instruction.md 참고. 다른 Phase는 건드리지 말 것."
```

- Agent가 참조 문서를 언급해두면 내가 따로 설명 안 해도 컨텍스트를 스스로 채움
- "다른 건 건드리지 말 것"이라는 범위 한정 한 줄이 생각보다 큰 차이를 만듦

### 8 — 컨텍스트가 오염됐다는 신호를 알아두기

- Agent와 대화가 길어지면 초반의 설계 방향이나 규칙이 "희석"되는 현상이 생김
- 아래 신호가 보이면 새 대화를 시작하는 것이 낫다:
  - Agent가 이미 결정된 기술 스택이나 구조를 다시 물어보기 시작할 때
  - 이전에 금지했던 패턴이 다시 등장할 때
  - 구현한 코드가 `CLAUDE.md`의 아키텍처 구조와 달라지기 시작할 때
- 새 대화를 시작할 때는 `CLAUDE.md`와 작업지침서를 다시 언급하는 것으로 충분함
- Claude Code에서는 `/clear`로 대화 히스토리를 비우되, 참조 문서는 그대로 남아 있어서 컨텍스트가 금방 복원됨

### 5 — Skill 또는 템플릿이 없는 반복 작업은 자동화 대상

- 같은 작업을 세 번 이상 반복하게 되면, 그 작업을 `skill-designer`로 Skill로 만드는 걸 고려함
- 처음에는 수동으로 하다가, 패턴이 보이면 그때 자동화하는 순서가 효율적
- 너무 일찍 추상화하면 실제 쓸 일이 없는 Skill이 됨

### 6 — 설계 문서는 한 번 만들고 끝이 아님

- 구현하다 보면 설계가 바뀜 — 라이브러리가 교체되거나, 구조가 달라지거나, API가 추가됨
- 이 변경이 설계 문서나 `CLAUDE.md`에 반영되지 않으면 Agent가 "설계와 다른 코드"를 계속 참조하게 됨
- `doc-audit` → `agent-sync` 조합으로 주기적으로 갱신해주는 게 중요

## 자주 마주치는 상황들

바이브코딩을 하다 보면 반복적으로 마주치는 패턴들이 있다. 미리 알아두면 삽질을 줄일 수 있다.

### Agent가 설계와 다른 구조로 짜기 시작했을 때

1. 즉시 멈춘다 — 코드가 많이 쌓일수록 되돌리기 어려워짐
2. `git diff`로 무엇이 달라졌는지 확인
3. `CLAUDE.md`의 아키텍처 섹션과 비교해서 어디서 벗어났는지 짚어냄
4. "○○ 파일은 되돌려. 아키텍처는 CLAUDE.md 기준으로 다시 구현해줘" 식으로 재지시
5. 같은 실수가 반복된다면 `basic-instruction.md`의 금지 패턴에 추가

### Agent가 요청하지 않은 곳까지 건드렸을 때

- 바이브코딩의 흔한 함정 중 하나 — "개선"이라는 이름으로 건드리지 않아도 되는 파일까지 수정함
- `git diff`로 변경 범위를 먼저 확인
- Phase 외 파일이 수정됐다면 `git checkout -- <파일명>`으로 되돌림
- 이후 지시에 "이 파일들만 수정할 것: ○○, △△" 식으로 범위를 명시

### 설계 문서와 코드가 서로 달라진 느낌이 올 때

- 2~3 Phase를 넘기면 흔히 발생함 — 구현하면서 설계가 조금씩 바뀌는데 문서는 그대로인 상태
- `doc-audit` 스킬로 괴리를 분석
- 괴리가 있으면: 코드가 맞고 설계가 구식인 경우가 많음 → 설계 문서 갱신 후 `agent-sync`로 동기화
- 이 작업을 미루면 Agent가 "구식 설계"를 기준으로 코드를 짜는 상황이 반복됨

### Phase가 너무 커서 중간에 막혔을 때

- Phase 하나에 API 5개 이상, 또는 BE+FE 합쳐서 파일이 10개 이상이면 중간에 인터페이스가 안 맞는 경우가 생김
- 이 경우 Phase를 더 잘게 나눠서 `impl-doc`을 다시 작성하는 것이 낫다
- "나머지는 나중에, 지금 이것만" 식으로 현재 Phase의 범위를 좁혀서 우선 통합 검증을 통과시키는 방법도 유효
