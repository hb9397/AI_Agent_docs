# 컴포넌트 설계 규칙 (component-design)

---

## 컴포넌트 트리 작성 규칙

각 화면의 컴포넌트를 트리 구조로 표현한다.

```markdown
### 컴포넌트 트리

{화면명}Page
├── CommonLayout (공통)
│   ├── Header
│   └── Sidebar
├── {기능}Filter
│   ├── SearchInput
│   ├── DateRangePicker
│   └── StatusSelect
├── {기능}Table
│   ├── TableHeader (정렬 기능)
│   ├── TableRow × N
│   │   └── ActionButtons
│   └── Pagination
└── {기능}Modal (조건부 렌더링)
    ├── ModalHeader
    ├── DetailForm
    └── ModalFooter
```

---

## 컴포넌트 명세 형식

각 컴포넌트에 대해 아래를 기술한다:

```markdown
#### {컴포넌트명}

- **책임**: {이 컴포넌트가 하는 일 — 1문장}
- **Props**: {부모로부터 받는 데이터}
- **로컬 상태**: {이 컴포넌트 내부에서 관리하는 상태}
- **이벤트**: {부모에게 올리는 이벤트 / 콜백}
- **API 호출**: {이 컴포넌트가 직접 호출하는 API — 없으면 "없음"}
```

---

## 공통 컴포넌트 식별 기준

```
공통 컴포넌트로 분리하는 경우:
  - 2개 이상 화면에서 동일한 UI 패턴 사용
  - 동일한 API 호출 + 동일한 렌더링 로직
  - 프로젝트 전역 상태에 접근하는 UI (인증 상태, 알림 등)

분리하지 않는 경우:
  - 비슷해 보이지만 비즈니스 로직이 다른 경우
  - 1개 화면에서만 사용하는 경우
  - 분리 시 props가 10개 이상 필요한 경우 (과도한 추상화)
```

## 상태 관리 분류

```markdown
### 상태 분류

| 상태 | 유형 | 위치 | 설명 |
|------|------|------|------|
| 검색 필터 값 | 로컬 | Filter 컴포넌트 | URL 쿼리 파라미터와 동기화 |
| 목록 데이터 | 서버 | React Query / SWR | API 응답 캐싱 + 자동 재검증 |
| 선택된 행 | 로컬 | Table 컴포넌트 | 상세 모달 열기용 |
| 인증 토큰 | 전역 | Context / Store | 전체 앱 공유 |
| 모달 열림/닫힘 | 로컬 | Page 컴포넌트 | boolean 상태 |
```

## 로딩/에러/빈 상태 처리

각 화면에서 아래 3가지 상태를 반드시 정의한다:

```
로딩 상태: 스켈레톤 UI / 스피너 / 프로그레스 바
에러 상태: 에러 메시지 + 재시도 버튼 / 폴백 UI
빈 상태:  안내 메시지 + 데이터 생성 유도 CTA
```
