# 화면 분석 기준 (screen-analysis)

설계 문서를 받으면 출력 전에 아래 항목을 내부적으로 추출한다.

---

## 화면 식별 규칙

설계 문서에서 화면을 식별하는 소스:

```
1순위: 설계 문서의 "화면 목록" 또는 "인터페이스 설계" 섹션
2순위: rfp-design-input-*.md의 "화면 후보 목록" 섹션
3순위: 설계 문서의 핵심 흐름에서 사용자 접점 추론
4순위: API 엔드포인트에서 대응 화면 역추적
```

---

## 추출 항목 체크리스트

```
[ ] 전체 화면 목록 (이름, 유형, 대응 SFR)
[ ] 화면 간 이동 흐름 (네비게이션 구조)
[ ] 공통 레이아웃 요소 (헤더, 사이드바, 푸터)
[ ] 인증/권한 체계 (화면별 접근 권한)
[ ] 기술 스택 (FE 프레임워크, 상태관리, UI 라이브러리)
[ ] API 엔드포인트 전체 목록
[ ] 데이터 모델 / 엔티티
[ ] 공통 컴포넌트 후보 (테이블, 폼, 모달, 알림)
[ ] 외부 서비스 연동 (지도, 차트, 파일 업로드 등)
```

---

## 화면 분류

| 복잡도 | 기준 | 지침 상세 수준 |
|--------|------|---------------|
| **단순** | CRUD 1종, API 2개 이하 | 컴포넌트 트리 + API 목록 정도 |
| **보통** | CRUD + 필터/검색/페이징, API 3~5개 | 상태 관리 + 인터랙션 시나리오 포함 |
| **복합** | 실시간 연동, 복잡한 폼, 차트, 다단계 위자드 | 전체 항목 상세 기술 |

---

## 화면 의존 관계 분석

```
의존 관계 유형:
  데이터 의존: 화면A에서 생성한 데이터를 화면B에서 조회
  네비게이션 의존: 화면A에서 클릭해야 화면B에 진입
  인증 의존: 로그인 화면 → 인증 후 접근 가능한 화면들
  설정 의존: 설정 화면의 값이 다른 화면 동작에 영향
```

## 사용자에게 물어볼 우선순위

분석 후 불명확한 항목 중 아래 순서로 최대 3개만 선별:

🔴 반드시 확인:
1. 인증/권한 체계가 불명확 (누가 어떤 화면을 보는가)
2. 화면 간 데이터 전달 방식이 불명확 (URL 파라미터, 전역 상태, API 재호출)
3. 실시간 업데이트 필요 여부 (WebSocket, SSE, 폴링)

🟡 선택 확인:
4. 반응형 대응 범위 (데스크톱만 / 모바일 포함)
5. 기존 코드베이스 존재 여부 (신규 / 기존 프로젝트에 추가)
