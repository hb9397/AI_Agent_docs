# 화면별 검증 체크리스트 규칙 (verification)

---

## 검증 체크리스트 형식

```markdown
### 검증: {화면명}

**시나리오**: {한 줄 요약 — 이 화면에서 무엇을 확인하는지}

#### 기능 검증
- [ ] {데이터 조회}: {화면 진입 시 목록이 API로부터 로드되는가}
- [ ] {필터링}: {검색어 입력 → API 재호출 → 결과 갱신 확인}
- [ ] {등록}: {폼 작성 → 제출 → 성공 메시지 → 목록에 반영}
- [ ] {수정}: {기존 데이터 로드 → 수정 → 반영 확인}
- [ ] {삭제}: {확인 다이얼로그 → 삭제 → 목록에서 제거}

#### 상태 검증
- [ ] {로딩}: API 호출 중 로딩 인디케이터 표시
- [ ] {에러}: API 에러 시 에러 메시지 표시 + 재시도 가능
- [ ] {빈 상태}: 데이터 없을 때 안내 메시지 표시

#### 에지 케이스
- [ ] {대량 데이터}: 100건 이상에서 페이징 정상 동작
- [ ] {동시 편집}: 다른 사용자가 수정한 데이터 충돌 처리
- [ ] {권한 부족}: 권한 없는 기능 접근 시 적절한 안내

**합격 조건**: [조건 A] ✅ / [조건 B] ✅ / [조건 C] ✅
```

---

## 좋은 검증 vs 나쁜 검증

```
✅ 좋은 검증
  "검색어 '전기' 입력 후 Enter → 목록이 필터링되어 해당 항목만 표시"
  "필수 필드 비운 채 제출 → 해당 필드에 빨간색 에러 메시지 표시"
  "API 500 에러 발생 시 '서버 오류' 토스트 + '재시도' 버튼 표시"

❌ 나쁜 검증
  "검색 기능이 정상 동작하는지 확인"
  "에러 처리가 잘 되는지 확인"
  "화면이 제대로 나오는지 확인"
```

## SFR 매핑 검증

RFP 기반 프로젝트에서는 각 검증 항목에 SFR 번호를 태그한다:

```markdown
- [ ] [SFR-019] 점검 결과 목록 조회 → 10건 이상 표시 확인
- [ ] [SFR-019] AI 분석 결과 위험도 3단계 표시 확인
- [ ] [SFR-020] 설비 상세에서 점검 이력 탭 접근 확인
```

이를 통해 sfr-trace 스킬이 커버리지를 추적할 수 있다.
