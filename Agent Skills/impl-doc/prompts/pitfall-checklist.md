# 함정 체크리스트 (pitfall-checklist)

작업지침서 초안 완성 후 출력 전에 아래를 자체 점검한다.
**스케일에 따라 해당하는 섹션만 실행한다.**

| 스케일 | 실행할 섹션 |
|--------|------------|
| 프로젝트 전체 | 기술적 함정 + 문서 함정 + 구조적 함정 전체 |
| 기능 단위 | 기술적 함정 + 구조적 함정 |
| 화면 단위 | 기술적 함정(FE 관련만) + 구조적 함정 |
| 컴포넌트/로직 | 구조적 함정만 |

---

## 기술적 함정 (범용)

```
[ ] 외부 API 호출에 타임아웃·재시도 전략이 명시되어 있는가?
[ ] 비동기 처리(async/await, 큐)가 필요한 곳에 동기 처리가 가정되어 있지 않은가?
[ ] 환경별 설정값(개발/배포)이 하드코딩 없이 환경변수로 분리되는가?
[ ] 인증/인가가 필요한 엔드포인트에 누락이 없는가?
[ ] 대용량 데이터 처리 시 페이징/스트리밍 전략이 있는가?
```

## 문서 함정

```
[ ] API 경로 표기 오류: 설계 문서의 엔드포인트 목록과 대조
[ ] 환경별 동작 차이 누락: 개발/배포 환경에서 다른 동작은 반드시 명시
[ ] DB 모델 누락: 설계 문서에 있는 모든 테이블이 BE-XX에 반영됐는지
[ ] 마이그레이션 태스크 누락: DB 모델 정의 다음에 반드시 포함
[ ] .gitignore 생성 태스크 누락: 세션 파일, .env 파일 보안 처리
[ ] noVNC 분기 누락: 개발/배포 환경에서 FE 렌더링이 다른 경우
[ ] start.sh 환경 혼용: 배포 전용임을 명시
```

---

## 구조적 함정

```
[ ] Phase가 너무 큰가? — 한 Phase에 API 5개 이상이면 쪼갠다
[ ] 페어가 아닌가? — BE만 있거나 FE만 있는 Phase는 합치거나 분리
[ ] 검증이 너무 늦은가? — 핵심 기능은 최대한 앞 Phase에서 검증
[ ] 미결 사항이 구현을 막는가? — 결정 필요 항목은 해당 태스크 전에 표시
[ ] 태스크 의존 관계가 순환하는가? — A→B→A 형태의 순환 의존 금지
```
