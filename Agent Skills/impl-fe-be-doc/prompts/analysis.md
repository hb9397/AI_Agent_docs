# 설계 문서 분석 기준 (analysis)

설계 문서를 받으면 출력 전에 아래 항목을 내부적으로 추출한다.

---

## 스케일 확인

| 스케일 | 판단 기준 | Phase 구성 |
|--------|-----------|------------|
| 프로젝트 전체 | DB 스키마, 인프라, 여러 기능 모듈 포함 | Phase 0(인프라)부터 전체 구성 |
| 기능 단위 | 특정 기능 중심, API 명세 포함 | 해당 기능 Phase만 |
| 화면 단위 | 특정 페이지 UI/UX, 컴포넌트 구조 중심 | FE 위주, BE는 API 연동 수준 |
| 컴포넌트/로직 | 단일 컴포넌트, 훅, 유틸 수준 | FE-XX 태스크 1~3개 수준 |

---

## 구현 모드 판정

| 모드 | 판단 기준 | 산출물 형태 |
|------|-----------|-------------|
| FE/BE 페어 모드 | 기능 흐름과 API/DB 구현 순서가 중심 | 기능 Phase별 BE-XX + FE-XX 태스크 |
| 화면 중심 모드 | 화면정의서, 화면 목록, SFR↔화면 매핑, 컴포넌트/상태/인터랙션이 중심 | 화면 1개 = Phase 1개, 화면별 컴포넌트/API/상태 명세 포함 |

화면 중심 모드라도 BE API, DTO, DB 변경이 있으면 같은 Phase 안에 BE-XX 태스크를 포함한다.

## 추출 항목 체크리스트

```
[ ] BE 모듈 목록 (파일/폴더 단위)
[ ] FE 페이지 / 컴포넌트 목록
[ ] 전체 화면 목록 (이름, 유형, 대응 SFR/REQ)
[ ] 화면 간 이동 흐름 (네비게이션 구조)
[ ] 공통 레이아웃 요소 (헤더, 사이드바, 푸터)
[ ] 인증/권한 체계 (화면별 접근 권한)
[ ] 공통 컴포넌트 후보 (테이블, 폼, 모달, 알림)
[ ] API 엔드포인트 전체 (메서드, 경로, 설명)
[ ] WebSocket 엔드포인트 및 메시지 타입
[ ] DB 테이블 / 모델 목록
[ ] 외부 서비스 / 인프라 의존성
[ ] 환경변수 목록 (개발/배포 분리 여부)
[ ] 기술 스택 (언어, 프레임워크, 버전)
[ ] 핵심 로직 (설계 문서 G섹션 또는 강조된 부분)
[ ] 주의사항 / 제약 조건
[ ] 열린 결정 사항 (미정 항목)
```

---

## Phase 경계 판단 질문

아래 질문으로 Phase 경계를 결정한다.

1. 이 기능에서 가장 단순한 BE-FE 통신 파이프라인은 무엇인가? → Phase 1
2. 설계 문서의 "집중 로직"(G섹션)은 어느 Phase에서 완성되는가? → 핵심 Phase
3. 화면 중심이면 공통 레이아웃/인증/데이터 입력 화면 중 선행되어야 할 화면은 무엇인가?
4. 배포 관련 작업(Dockerfile, compose)이 있는가? → 마지막 Phase
5. 각 Phase가 독립적으로 테스트 가능한가? → 불가능하면 재분할

## 사용자에게 물어볼 우선순위

분석 후 불명확한 항목 중 아래 순서로 최대 3개만 선별해 한 번에 묻는다.

🔴 반드시 확인:
1. 개발 환경과 배포 환경 동작 차이가 있는가? (noVNC 분기 등)
2. AI / 비동기 처리 포함 기능이 있는가? (WebSocket, LLM 호출 등)
3. 설계 문서에 "미정" 또는 "열린 결정 사항"이 있는가?
4. 화면별 인증/권한 체계가 불명확한가?
5. 화면 간 데이터 전달 방식이 불명확한가? (URL 파라미터, 전역 상태, API 재호출)

🟡 선택 확인:
6. 기존 코드베이스가 있는가? ([NEW] / [MODIFY] 구분 필요 여부)
7. 작업지침을 읽는 대상이 사람인가, AI Agent인가, 둘 다인가?
