# 태스크 작성 규칙 (task-rules)

---

## 태스크 ID 체계

```
INF-XX : 인프라 (디렉토리 구조, 컨테이너, Dockerfile 등)
BE-XX  : 백엔드
FE-XX  : 프론트엔드
```

번호는 Phase 내 순서가 아닌 **전체 문서 통합 일련번호** 로 부여한다.
(예: Phase 1에서 BE-01~04, Phase 2에서 BE-05~06으로 이어짐)

화면 중심 Phase에서도 별도 `SCR-XX` 태스크를 만들지 않는다.
화면 설명은 Phase 본문에 두고, 실제 코드 작업은 `BE-XX` / `FE-XX` / `INF-XX`로 분리한다.

---

## 태스크 작성 형식

```markdown
### [태스크 ID] · [태스크 이름] (`파일 경로`)

**의존** : [선행 태스크 ID 목록. 없으면 "없음"]

[구현 내용 — 무엇을 왜 만드는지]

**Agent 지시** : (있을 때만 포함)
[핵심 함수 시그니처, 중요 분기 처리, 설계 결정 이유]

**검증 기준** : [이 태스크 단독 완료 판단 조건]
```

---

## Agent 지시 작성 기준

포함할 것:
- 핵심 함수 시그니처 / 클래스 구조
- 중요한 환경변수 처리 방식 (개발/배포 분기 등)
- 의도적인 설계 결정 이유
- 주의해야 할 분기 처리
- 화면 중심 Phase의 컴포넌트 책임, Props, 상태 위치, API 호출 시점
- API 요청/응답 스키마와 주요 HTTP 에러 처리

포함하지 말 것:
- 완성된 구현 코드 전체 (작성 여지를 남긴다)
- 표준 보일러플레이트 (import, 단순 초기화 등)
- 이미 다른 태스크에서 설명한 내용 반복

---

## 화면 중심 Phase 본문 형식

화면을 Phase 단위로 삼는 경우, 태스크 목록 전에 아래 요약 블록을 둔다.

````markdown
### 화면 개요

- **대응 SFR/REQ**: {SFR-XXX / REQ-XXX}
- **목적**: {이 화면이 존재하는 이유}
- **사용자**: {역할}
- **진입 경로**: {어디서 오는가}
- **이탈 경로**: {어디로 이동하는가}

### 컴포넌트 구조

```text
{화면명}Page
├── {공통 레이아웃}
├── {기능 컴포넌트}
└── {조건부 모달/패널}
```

### API 연동

| # | 메서드 | 경로 | 호출 시점 | 요청 | 응답 | 에러 처리 |
|---|--------|------|----------|------|------|----------|
| 1 | GET | /api/... | 화면 마운트 | query/body | 응답 구조 | 토스트 + 재시도 |

### 상태 관리

| 상태 | 유형 | 위치 | 설명 |
|------|------|------|------|
| filters | 로컬 | Page/Filter | 검색 조건 |
| list | 서버 | Query cache | 목록 API 응답 |
````
