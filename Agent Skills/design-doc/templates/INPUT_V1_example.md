## A. 한 줄 요약

```
사용자가 예약 사이트에서 요소를 클릭하면 AI가 역할을 추론하고 저장하는 최초 1회 온보딩 화면
```

---

## B. 문제와 목적

```
- 사이트마다 예약 요소(출발역, 도착역 등)의 HTML 구조가 다름
- 미리 이름을 하드코딩하면 사이트가 바뀔 때마다 코드를 수정해야 함
- 성공 기준: 사용자가 클릭만 하면 AI가 이름을 추론하고, 확인 후 저장까지 3~5분 안에 완료
```

---

## C. 사용자와 맥락

```
- 나 혼자 씀. 나중에 가족에게 줄 수도 있어서 UI는 단순하게.
- 새 사이트 등록 시 딱 1회만 진행. 이후엔 이 화면에 올 일 없음.
- 기술 지식: 개발자지만 이 화면을 쓸 때는 그냥 일반 사용자처럼 씀
```

---

## D. 범위

**포함할 것**
- 가상 브라우저(noVNC) 화면 임베드
- 클릭한 요소 목록 + AI 추론 역할명 표시
- 역할명 수정 가능
- 전체 확인 후 저장

**포함하지 않을 것**
- 로그인 (사용자가 가상 브라우저에서 직접)
- 사이트 URL 등록 (이전 화면에서 완료된 상태)
- 결제 관련 요소

---

## E. 핵심 흐름

```
1. 화면 왼쪽: 가상 브라우저(noVNC, :6080)에 예약 사이트가 열려 있음
2. 사용자가 예약에 필요한 요소를 순서대로 클릭
3. 클릭마다 오른쪽 패널에 요소 추가 + AI 추론 역할명 표시
4. 사용자가 역할명 확인 (틀리면 직접 수정)
5. 필요한 요소를 다 클릭했으면 저장 버튼 → DB 저장 완료
```

---

## F. 기술 스택

| 레이어 | 기술 |
|--------|------|
| Frontend | React + Vite, Tailwind CSS |
| Backend | FastAPI, Playwright |
| DB | PostgreSQL, SQLAlchemy |
| AI / 외부 서비스 | LangChain + Ollama (로컬 LLM) |
| 인프라 / 배포 | Docker (Xvfb + noVNC 포함) |

기타 제약사항:
```
- noVNC는 iframe으로 React 화면 안에 임베드
- AI 추론 응답은 2초 이내 목표. 느리면 스켈레톤 표시
- 가상 브라우저는 :6080 포트 고정
```

---

## G. 집중할 로직

```
클릭 이벤트 감지 → HTML 정보 수집 → LangChain 프롬프트 구성 → Ollama 추론 → 결과 반환

이 흐름이 이 화면의 핵심.
특히 "HTML 정보를 어떻게 수집해서 어떤 형태로 LLM에 넘기는지"를 자세히 다뤄줘.
추론 실패 시 fallback 처리도 포함.
```

## H. 뒤집기 확인

뒤집어진 결정:
| 처음 생각 | 바뀐 것 | 이유 |
|-----------|---------|------|
| 요소 이름을 사전에 목록으로 정의 | AI가 추론, 사람이 확인 | 사이트마다 다른 구조에 동일 플로우 적용 가능 |
| 로그인도 이 화면에서 처리 | 사용자가 가상 브라우저에서 직접 | 구현 복잡도 감소, 세션만 저장하면 충분 |
| 온보딩은 로컬 환경에서만 | noVNC로 컨테이너 안에서도 | Docker 환경 통일 |

## I. 열린 질문

```
- AI 추론 실패 시 fallback: 빈칸으로 남길지 vs 기본값 제안할지

- 저장 완료 후 다음 화면 이동 vs 현재 화면 유지
```
