# 최소 인터뷰 (interview)

코드에서 **절대 알 수 없는 것**만 묻는다. 최대 2회.
답변이 없거나 "모름"이면 `미정 — 사용자 미제공` 으로 기록하고 진행한다.

---

## 질문 원칙

1. **한 번에 하나만** 묻는다.
2. **코드에서 답할 수 있는 건 묻지 않는다.** (스택, 엔드포인트, 디렉토리, 환경 변수 등)
3. **비즈니스 의사결정·설계 배경은 묻지 않는다.** 알 수 없음을 인정하고 `미정`으로 둔다.
4. 사용자가 답변 제공을 거부하면 즉시 인터뷰 종료.

---

## 질문 1 (필수) — 도메인·목적·사용자

```
이 프로젝트는 어떤 사용자가, 어떤 문제를 해결하기 위해 쓰는 시스템인가요?
2~3줄로 설명해 주세요.

예시 답변:
"사내 엔지니어가 사용하는 배포 파이프라인 관리 도구.
버튼 하나로 여러 환경에 동일 빌드를 배포할 수 있게 하는 게 목적."
```

**사용처**: OUTPUT_V2 `01 개요` 섹션의 한 줄 요약 + 도메인 배경.

답변이 모호하면 **한 번만** 반박한다:

> "답변이 짧아서 목적이 불명확합니다.
> 'A를 하는 B'처럼 한 문장으로 다시 정리해 주실 수 있을까요?"

두 번째 반박은 하지 않는다. 그대로 받아 `미정 — 설명 불충분` 표기.

---

## 질문 2 (선택) — 상위 맥락·운영 제약

아래 두 경우에만 묻는다:
- 코드에서 **다른 시스템 연동**이 보이는데(예: S3, 외부 API, 메시지 큐), 상위 시스템이 불명확할 때
- **배포 환경 힌트**가 복잡해서 운영 제약이 있어 보일 때

질문:
```
이 프로젝트가 속한 상위 시스템이 있거나, 반드시 지켜야 할 운영 제약이 있으면 알려주세요.
(예: "고객사 VPC 안에서만 돈다", "오프라인 설치 가능해야 한다")
없으면 "없음"이라고 답해 주세요.
```

**사용처**: OUTPUT_V2 `01 개요` + `10 주의사항`.

답변이 "없음"이면 제약 항목을 비워둔다. 답변이 있으면 그대로 옮긴다.

---

## 묻지 않는 것 (묻고 싶어도 참는다)

| 묻지 않을 것 | 이유 |
|-------------|------|
| 기술 스택 선정 이유 | 설계 의도라 사후 알 수 없음 |
| 코딩 컨벤션 세부 | 기존 코드 스타일에서 역추론 |
| 레이어 의존성 규칙 | 현재 구조에서 관찰해서 기술 |
| 금지 패턴 목록 | 코드에 없으면 사용자가 만들어줘야 함 — 이 스킬 범위 밖 |
| 향후 로드맵 | 부트스트랩 범위 밖 |

**중요**: 금지 패턴(instruction의 핵심)은 코드에서 역추론하기 어렵다.
이 스킬은 **뼈대만** 만든다. 상세한 금지 목록은 사용자가 이후 `design-doc`·`context-doc`를 재실행하면서 추가해야 한다는 점을 마지막에 안내한다.

---

## 인터뷰 종료 후

인터뷰가 끝나면 사용자에게 알린다:

> "인터뷰 완료. 이제 코드 분석 결과와 답변을 조합해 설계 문서를 작성합니다.
>
> 참고: 코딩 컨벤션·금지 패턴 같은 팀 규칙은 코드에서 자동 추출이 어렵습니다.
> 부트스트랩이 끝난 뒤, 필요하면 `design-doc` → `context-doc`를 다시 실행해
> 팀 규칙을 추가해 주세요."
