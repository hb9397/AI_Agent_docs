# 불명확 항목 확인 규칙 (clarification)

---

## 원칙

1. **이 단계에서 해결한다.** 불명확 항목을 다음 스킬(design-doc)로 넘기지 않는다.
2. **인라인으로 묻는다.** 별도 스킬 호출 없이 대화 안에서 처리한다.
3. **한 번에 최대 5개.** 질문이 많으면 우선순위 순으로 5개만 선별한다.
4. **선택지를 제시한다.** "어떻게 할까요?"가 아니라 "A / B / 기타" 형태로 묻는다.

---

## 질문 우선순위

🔴 **P1 — 구현을 막는 불명확**
- 처리 로직 알고리즘이 명시되지 않음 (AI 판정 기준, 계산 공식 등)
- 데이터 입출력 형식이 불명확 (필수 필드, 데이터 타입)
- 외부 시스템 연계 방식이 미정 (API vs 파일 vs DB 공유)
- 사용자 권한 체계가 불명확 (누가 어떤 기능을 쓰는가)

🟡 **P2 — 설계 방향에 영향**
- 화면 레이아웃 상세가 없음 (테이블 vs 카드, 차트 종류 등)
- 비기능 요구사항이 기능 설계에 영향 (동시 접속자 수, 응답 시간)
- 데이터 보존·삭제 정책이 미정

🟢 **P3 — 나중에 결정해도 무방**
- UI 세부 스타일 (색상, 아이콘 등)
- 알림 채널 선택 (이메일 vs 푸시 vs SMS)
- 리포트 출력 형식 (PDF vs Excel)

---

## 질문 형식

```markdown
분석 중 아래 항목이 불명확합니다. 확인 부탁드립니다:

1. **[SFR-019] AI 판정 기준**: RFP에 "AI를 활용한 위험도 판정"으로만 기술되어 있습니다.
   - A) 규칙 기반 판정 (임계값 테이블)
   - B) ML 모델 기반 판정 (학습 데이터 필요)
   - C) 기타: ___

2. **[SFR-019] 점검 데이터 입력 방식**: 센서 데이터가 자동 수집인지 수동 입력인지 불명확합니다.
   - A) IoT 센서 자동 연동
   - B) 점검자가 모바일/웹에서 수동 입력
   - C) 둘 다 지원
```

---

## 답변 반영 규칙

- 사용자 답변은 즉시 해당 SFR의 해석 결과에 반영한다.
- "기타"로 새로운 정보가 오면 화면 후보 매핑도 재검토한다.
- "나중에 결정"이 오면 중간 문서에 `미정 — [이유]`로 기록한다.
- 답변으로 새로운 화면이 필요해지면 화면 후보에 추가한다.
