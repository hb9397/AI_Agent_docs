# 화면 후보 매핑 규칙 (screen-mapping)

---

## 화면 도출 원칙

SFR의 기능을 수행하기 위해 **사용자가 직접 보거나 조작하는 단위**를 화면으로 도출한다.

```
1개 SFR = 1~N개 화면
  - 단순 조회 기능 → 1개 화면 (목록 + 상세 통합 가능)
  - CRUD 기능 → 2~3개 화면 (목록, 등록/수정, 상세)
  - 복합 기능 → 3개 이상 (단계별 위자드, 대시보드 + 상세 등)
```

---

## 화면 후보 분류 체계

| 유형 | 설명 | 예시 |
|------|------|------|
| 목록 화면 | 데이터를 테이블/카드로 나열 | 점검 이력 목록, 설비 목록 |
| 등록/수정 화면 | 폼 입력으로 데이터 생성·수정 | 점검 결과 입력, 설비 등록 |
| 상세 화면 | 단건 데이터 조회·분석 결과 표시 | 점검 상세, AI 분석 결과 |
| 대시보드 | 집계·통계·현황 종합 표시 | 설비 현황 대시보드, 위험도 현황 |
| 설정 화면 | 시스템 파라미터·권한 관리 | 사용자 관리, 코드 관리 |
| 리포트 화면 | 출력·다운로드 목적 정형 보고서 | 점검 보고서, 월간 통계 |

---

## 매핑 표 형식

```markdown
### SFR ↔ 화면 매핑

| SFR | 화면명 | 유형 | 목적 | 주요 액션 | 우선순위 |
|-----|--------|------|------|-----------|---------|
| SFR-019 | 점검 결과 목록 | 목록 | 점검 이력 조회·필터링 | 검색, 필터, 정렬, 엑셀 다운로드 | 높음 |
| SFR-019 | AI 분석 결과 상세 | 상세 | 개별 점검 AI 판정 결과 확인 | 판정 결과 확인, 재분석 요청 | 높음 |
| SFR-019 | 위험도 현황 대시보드 | 대시보드 | 전체 설비 위험도 현황 종합 | 기간 선택, 드릴다운 | 중간 |
```

---

## 화면 간 이동 흐름 도식

```markdown
### 화면 흐름

점검 결과 목록 ──[행 클릭]──→ AI 분석 결과 상세
      │                              │
      │                        [재분석 버튼]
      │                              │
      └──[대시보드 탭]──→ 위험도 현황 대시보드
                              │
                        [설비 클릭]
                              │
                              ▼
                     설비 상세 (SFR-020 영역)
```

---

## 우선순위 판단 기준

| 우선순위 | 기준 |
|---------|------|
| **높음** | SFR 핵심 기능을 직접 수행하는 화면. 없으면 SFR 미충족 |
| **중간** | SFR 보조 기능. 있으면 사용성이 향상되지만 핵심은 아님 |
| **낮음** | 편의 기능. 관리자용 화면, 부가 리포트 등 |

---

## 화면 통합·분리 판단

```
통합 고려:
  - 목록 + 상세가 같은 컨텍스트에서 전환 → 탭 또는 분할 뷰로 통합 가능
  - 등록 + 수정 폼이 필드가 동일 → 같은 화면에서 모드 전환

분리 필수:
  - 접근 권한이 다른 화면 (일반 사용자 vs 관리자)
  - 데이터 소스가 완전히 다른 화면
  - 사용 빈도·맥락이 완전히 다른 화면
```
