# prompts/checklist.md
# 역할: 스킬 생성·고도화 완료 전 자가 검증 체크리스트

---

## 라우팅 표

| 목적 | 실행할 섹션 |
|------|-----------|
| 신규 생성 스킬 점검 | 전체 |
| 기존 스킬 고도화 점검 | [구조] + [토큰] + [안전성] |
| Bash 사용 스킬 추가 점검 | [속도] 추가 실행 |
| 병렬 처리 스킬 추가 점검 | [병렬 처리] 추가 실행 |

---

## [구조]

```
[ ] SKILL.md는 흐름(Step 목록·참조)만 담고 있는가?
    → 규칙 내용이 SKILL.md에 직접 서술되어 있으면 prompts로 이동
[ ] prompts 파일이 단일 책임인가?
    → 두 역할이 한 파일에 있으면 분리
[ ] 연계 스킬이 있으면 SKILL.md 상단에 연계 흐름 표시가 있는가?
[ ] 헤더에 name, description, allowed-tools가 모두 있는가?
[ ] allowed-tools가 실제 사용 도구만 선언했는가?
```

**판정 기준**: 전체 통과 → [토큰] 진행 / 미통과 항목 → 즉시 수정 후 재점검

---

## [토큰]

```
[ ] templates 파일에 예시 데이터가 없는가?
    → 예시는 HTML 주석으로만
[ ] SKILL.md와 prompts 간 중복 내용이 없는가?
    → 동일 내용이 두 곳에 있으면 prompts에만 남기고 SKILL.md는 참조로
[ ] 조건 분기가 있는 prompts 파일 상단에 라우팅 표가 있는가?
[ ] 한 번에 묻는 질문이 3개 이하인가?
[ ] 파일 생성 기본값이 "대화창 출력"이고 저장은 요청 시에만인가?
```

---

## [속도] ← Bash 사용 스킬만

```
[ ] 파일 감지를 먼저 하고 존재하는 것만 실행하는가?
[ ] cat / grep 등 대용량 명령에 범위 제한(head -N, -A N)이 있는가?
[ ] 변경 없을 때 전체 스캔 fallback이 없는가?
    → 변경 없으면 즉시 종료
[ ] git 명령이 --stat / --name-only 등 최소 범위를 사용하는가?
[ ] package-lock.json, yarn.lock 등 lock 파일을 직접 읽지 않는가?
```

---

## [병렬 처리] ← 병렬 Task가 있는 스킬만

```
[ ] 병렬 Task가 실제로 독립적인가? (선행 결과 의존 없음)
[ ] Task 수가 6개 이하인가?
[ ] 2개 이하 관점에 병렬을 쓰고 있지 않은가?
[ ] sub-agent 미지원 환경의 순차 fallback이 명시되어 있는가?
```

---

## [안전성]

```
[ ] 파일 수정·생성 전 사용자 확인 게이트가 있는가?
    → "위 내용으로 파일을 저장할까요? (승인 / 수정 / 취소)"
[ ] 수정 요청 시 전체 재작성이 아닌 해당 섹션만 수정하는가?
[ ] 재귀 호출 위험이 있는 스킬에 disable-model-invocation이 선언됐는가?
```

---

## 안티패턴 빠른 감지표

| 증상 | 안티패턴 | 수정 방향 |
|------|---------|----------|
| SKILL.md에 규칙 표가 있음 | 이중 명세 | prompts로 이동, SKILL.md는 참조만 |
| prompts 파일이 두 역할 담당 | 단일 파일 다중 책임 | 목적별 파일 분리 |
| templates에 실제 값 있음 | 예시 데이터 혼입 | 빈 구조 + HTML 주석 |
| 한 번에 질문 4개 이상 | 질문 남발 | 🔴 2개로 압축 |
| 변경 없어도 전체 grep | Fallback 전체 스캔 | 변경 없으면 즉시 종료 |
| 관점 2개도 Task 분기 | 과도한 병렬화 | 3개 미만은 순차 |
| 결과를 항상 파일로 저장 | 무조건 파일 생성 | 기본은 대화창, 저장은 요청 시 |
