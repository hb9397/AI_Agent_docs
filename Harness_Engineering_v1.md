# Harness Engineering Guide v1

> AI Agent 하네스 엔지니어링 관점에서 정리한 전체 워크플로우.
> RFP 기반 흐름과 일반 아이디어 기반 흐름으로 분기하며,
> create-prototype 는 제안 단계와 구현 단계 두 곳에서 사용 가능하다.

---

## 1. 이 저장소의 목적

AI Agent(Claude Code 등)를 써서 코드를 짜다 보면 결국 **"AI에게 무엇을 어떻게 알려주는가"** 가 결과물의 품질을 결정한다.

아무 준비 없이 "이거 만들어줘" 하면 AI도 대충 만들고, 방향을 잃는다.
그래서 **설계 → 컨텍스트 구성 → 구현** 의 흐름을 템플릿화하고, 반복 가능한 형태로 정리한 것이 이 저장소다.

### 저장소 구조

```
ai_agent_forms/
├── Agent Skills/        ← IDE 안의 AI Agent (Claude Code) 에서 사용하는 Skill 모음
├── Docs Skills/         ← 웹 AI (Gemini Gems, Claude Project) 에서 사용하는 템플릿 문서 모음
└── example/             ← 실제 산출물 예시들
```

### Agent Skills 목록

Claude Code 같은 IDE 내 Agent에서 `/skill명` 형태로 호출해서 쓰는 자동화 워크플로우 묶음.
각 Skill은 `SKILL.md` + `prompts/` + `templates/` 구조로 이루어진다.

| Skill | 한줄 설명 |
| ----- | --------- |
| `design-doc` | 설계 인터뷰 → 구조화된 설계 문서 자동 도출 |
| `context-doc` | 설계 문서 → `CLAUDE.md` + `basic-instruction.md` 생성 |
| `impl-fe-be-doc` | 설계 문서 → FE/BE 페어 Phase별 작업지침서 생성 (구 `impl-doc`) |
| `impl-screen-doc` | 설계 문서 → RFP SFR 기반 화면 단위 구현 지침서 생성 *(추가 예정)* |
| `impl-doc` | 범용 단계별 구현 지침 — FE/BE·화면 구분 없이 사용 *(추가 예정)* |
| `design-prototype-docs` | 기능요구사항 → 화면 설계 문서(.md) 생성 (create-prototype 입력) |
| `create-prototype` | 설계 기반 HTML 프로토타입 생성 (다중 파일 + JSON 데이터 분리) |
| `frontend-design` | 고품질·개성 있는 프론트엔드 UI 코드 생성 |
| `agent-sync` | Agent 문서·Skill 변경 감지 및 동기화 |
| `code-comment` | 변경된 코드 파일에 한글 주석 자동 작성 |
| `commit` | 변경 내용 분석 후 Conventional Commits 규칙 커밋 메시지 생성 |
| `doc-audit` | 실제 코드 ↔ AI 문서 간 괴리 분석 및 수정 제안 |
| `multi-review` | 보안·성능·유지보수·테스트 4개 관점 멀티 페르소나 코드 리뷰 |
| `pre-commit` | 커밋 전 에러 처리·민감 정보·TODO 형식 등 규칙 위반 검사 |
| `skill-designer` | 반복 워크플로우를 인터뷰 기반으로 새 Skill로 설계·생성·테스트 |

### Docs Skills — 웹 AI용 템플릿 문서

Gemini Gems 또는 Claude Project에 넣어서 쓰는 인터뷰 템플릿 문서들.
웹 AI가 이 문서를 시스템 프롬프트처럼 참조해서 인터뷰를 진행하고 산출물을 뽑아주는 방식.

| 문서 폴더 | 용도 |
| --------- | ---- |
| `설계문서_도출/` | 프로젝트·화면·기능 스케일별로 설계 인터뷰를 진행해서 설계 문서를 도출 |
| `구현작업_지시서_도출/` | 완성된 설계 문서를 기반으로 구현 작업지침서(Phase별 태스크)를 도출 |

## 2. 전체 스킬 맵

```text
[RFP 해석 계열]          [설계 계열]              [구현 지침 계열 — 하나만 선택]   [목업·구현 계열]
rfp-ingest ✗            design-doc ✓             impl-fe-be-doc ✓              create-prototype ✓
rfp-compliance-         design-prototype-docs ✓  impl-screen-doc ✗  (예정)     frontend-design ✓
review ✗                context-doc ✓            impl-doc ✗         (예정)     code-comment ✓
                        sfr-trace ✗

[품질 게이트 계열]        [메타·하네스 계열]
multi-review ✓          skill-designer ✓
pre-commit ✓            doc-audit ✓
commit ✓                agent-sync ✓
                        pdf ✓ / pptx ✓ / xlsx ✓ / docx ✓
```

> ✓ = 구현 완료  ✗ = 스펙 정의만 있거나 미존재 (구현 필요)
>
> **구현 지침 계열 선택 규칙**: design-doc 이후 세 스킬 중 **하나만** 선택 실행한다.
> - `impl-fe-be-doc`: FE/BE 역할이 분리된 프로젝트에서 Phase별 페어 작업지침서가 필요한 경우
> - `impl-screen-doc`: RFP SFR 기반 화면 단위 명세가 중심인 경우
> - `impl-doc` (예정): FE/BE·화면 구분 없이 범용 단계별 구현이 필요한 경우

## 3. 작업 스케일 — 먼저 이걸 정하고 시작

작업 전 "지금 이게 어느 수준의 일인가"를 먼저 판단하는 게 중요하다.
스케일에 따라 어떤 단계를 거칠지, 어떤 문서를 만들지가 달라진다.

| 스케일 | 기준 | 예시 |
|--------|------|------|
| **프로젝트** | 시스템 전체를 새로 만드는 수준 | 신규 서비스, 주요 서브시스템 신규 구축 |
| **화면 단위** | 특정 페이지나 뷰 하나를 만드는 수준 | 대시보드 화면, 설정 페이지 |
| **기능 단위** | 독립적으로 동작하는 기능 하나 | 예약 매크로 실행, 로그인 기능 |
| **세부 로직** | 함수·컴포넌트 내부 알고리즘 수준 | DOM 변경 감지 로직, 재시도 에이전트 |

- 세부 로직 수준은 아래 흐름을 다 거치지 않아도 됨 — 바로 구현해도 무방한 경우가 많음
- 프로젝트 스케일일수록 단계를 꼼꼼히 밟을수록 후반부 삽질이 줄어든다

## 4. 어느 흐름으로 시작할 것인가

```
작업 시작
    │
    ├── RFP(제안요청서)가 있는가?
    │         │
    │       YES ──► Flow B: RFP 기반 흐름  (§6)
    │         │
    │        NO
    │         │
    └── 아이디어 / 내부 기획만 있는가?
              │
            YES ──► Flow A: 일반 흐름      (§5)
```

| 구분 | 진입점 | 특징 |
|------|--------|------|
| **Flow A** | 아이디어 · 내부 기획 | 자유도 높음. 설계부터 직접 시작 |
| **Flow B** | RFP 파일 (`@pdf`) | SFR 번호 기반. 요구사항 추적 필수 |

## 5. Flow A — 일반 바이브코딩 흐름 (RFP 없음)

```

│                      Flow A: 일반 아이디어 기반                        │

  ▼                          ▼
```

### 프로토타입 파이프라인 — 실행 시점 선택

`design-prototype-docs → create-prototype`은 **impl-* 전후 어느 시점에든** 실행 가능하다.

| 시점 | 언제 선택하는가 | 특징 |
|------|----------------|------|
| **impl 전** | 화면 구성을 먼저 확정하고 싶을 때. 고객 확인용 목업이 필요할 때 | 프로토타입 기반으로 impl 지침이 더 구체적으로 나옴 |
| **impl 후** | 구현 설계가 확정된 뒤 정밀 프로토타입이 필요할 때 | API·데이터 구조가 반영된 정밀 목업 |
| **양쪽 다** | 제안 단계 → 추상 목업, 구현 단계 → 정밀 목업 | 나중 버전이 이전 버전을 덮어씀 |

### Flow A 스케일별 단축 경로

| 스케일 | 생략 가능한 단계 |
|--------|----------------|
| 프로젝트 | 전 단계 필수 |
| 화면 단위 | context-doc 선택. 프로토타입은 impl 전에 먼저 뽑는 것이 효과적 |
| 기능 단위 | context-doc 선택. 구현 지침 스킬은 상황에 맞게 하나 선택 |
| 세부 로직 | design-doc만 하고 바로 구현 가능 |

> **`impl-doc` (추가 예정)**: FE/BE 페어도 화면 중심도 아닌 범용 단계별 구현이 필요한 경우 사용.

### Flow B 프로토타입 파이프라인 — 실행 시점 선택

`design-prototype-docs → create-prototype`은 Flow A와 마찬가지로 **impl-* 전후 어느 시점에든** 실행 가능하다.

| 시점 | 흐름 | 상태 | 목적 |
|------|------|------|------|
| **제안 단계** | rfp-ingest → **design-prototype-docs** → create-prototype | 설계 미확정 허용 | 발주처 설득용 추상 목업 |
| **impl 전** | design-doc → **design-prototype-docs** → create-prototype → impl-* | 화면 확정 우선 | 프로토타입으로 화면 확정 후 구현 지침 작성 |
| **impl 후** | impl-* → **design-prototype-docs** → create-prototype | 설계 확정 기반 | API·데이터 반영된 정밀 프로토타입 |

> 같은 SFR에 대해 여러 번 실행이 정상. 나중 버전이 이전 버전을 덮어씀.
> `design-prototype-docs`는 어느 시점에서든 `create-prototype`의 선행 스킬로 동작한다.

## 7. 각 단계 상세

### Step 1 — 설계 문서 도출 (`/design-doc`)

**설계 문서 없이 바로 구현하면 생기는 일**

- "이거 만들어줘" 한 줄로 시작하면 AI가 제멋대로 기술 스택을 고르고, 내가 생각한 구조와 전혀 다른 걸 만들어옴
- 설계 문서가 있으면 AI가 내 머릿속과 같은 그림을 공유한 상태에서 코드를 짜게 됨
- 설계하면서 미처 생각 못했던 분기나 예외 케이스가 드러나는 부수 효과도 있음

**도구**

| 환경 | 도구 | 방식 |
| ---- | ---- | ---- |
| 웹 AI | `Docs Skills/설계문서_도출/` 문서를 Gems·Project에 업로드 | AI가 템플릿 기반으로 인터뷰 진행 후 산출물 출력 |
| Agent (IDE) | `Agent Skills/design-doc` 스킬 호출 | `/design-doc` 입력 → Step별 인터뷰 → 문서 도출 |

**design-doc 스킬이 하는 일**

- **Step 1** — 스케일 확인 + 기존 문서(PRD, DB 설계서 등) 유무 확인
- **Step 2** — 인터뷰 진행: 섹션 순서대로 한 번에 하나씩 질문, 모호한 답변이 오면 반박하고 더 파고듦
- **Step 3** — 스케일에 맞는 OUTPUT 양식으로 문서 작성
- **Step 4** — 초안 출력 후 사용자 확인 → 수정 반영 → 파일 저장

**OUTPUT 문서 구조 (`example/Acro.md` 참고)**

```
01. 개요          스케일, 핵심 목표, 핵심 설계 결정
02. 핵심 흐름     단계별 동작, 분기 처리, 상태 전환
03. 집중 로직     가장 복잡한 핵심 알고리즘, 코드 스니펫 수준으로 명세
04. 파일 구조     디렉토리 트리, 각 파일의 책임
05. 데이터 구조   DB 테이블, 주요 컬럼, 인덱스
06. 라이브러리    사용 기술 스택 및 외부 구성 요소 목록
07. 대안 전략     복잡도 높은 항목의 구현 방향 및 트레이드오프
08. 주의사항      법적 이슈, 보안 제약, 성능 주의점
09. 열린 결정     아직 결정 못한 항목들
```

- 확실하지 않은 항목은 비워두지 않고 `미정 — [이유]` 로 표시
- 이 문서가 이후 모든 단계의 입력이 됨

---

### Step 2 — 우려 사항 고도화 인터뷰

**왜 설계 문서만으로는 부족한가**

- 처음 써보는 기술이 들어간 경우, 그 기술의 함정과 제약을 미리 파악해두지 않으면 구현 중반에 구조를 갈아엎는 상황이 생김
- 1단계 OUTPUT을 들고 웹 AI와 한 번 더 대화하면서 이 부분들을 미리 정리하는 단계

**고도화 대상 예시**

| 구분 | 확인하는 내용 |
| ---- | ------------ |
| **기술 스택 불확실성** | "LangGraph가 꼭 필요한가, LangChain만으로 충분한가?" |
| **핵심 로직 구현 가능성** | "토큰 제한 내에서 복구 에이전트가 실제로 동작하는가?" |
| **확장 가능성** | "멀티 사이트 구조가 실제로 버텨주는가?" |
| **처음 쓰는 기술** | 공식 문서·예제를 함께 붙여넣고 적용 방법 확인 |

- 고도화된 내용은 1단계 문서의 대안 전략·주의사항 섹션에 반영
- 이 단계의 결과물은 별도 파일이 아니라 설계 문서가 더 풍부해지는 형태

---

### Step 3 — Context 문서 생성 (`/context-doc`)

**컨텍스트 문서가 없으면 생기는 문제**

- AI Agent는 대화창에 올려진 정보만 보고 코드를 짬 — 컨텍스트가 없으면 매번 같은 설명을 반복해야 함
- `CLAUDE.md` 같은 컨텍스트 문서를 프로젝트 루트에 두면 Agent가 매 요청마다 이 문서를 참조해서 이미 알고 있는 상태로 코드를 짬

**스케일별 적용 여부**

| 스케일 | CLAUDE.md | basic-instruction.md |
| ------ | --------- | -------------------- |
| 프로젝트 | 필수 | 필수 |
| 화면 단위 | 선택 (기존 갱신으로 대체 가능) | 선택 |
| 기능 단위 | 선택 (단순 내용 추가 정도) | 선택 |
| 세부 로직 | 보통 생략 | 보통 생략 |

**context-doc 스킬이 하는 일**

- **Step 1** — 설계 문서 수집 (없으면 요청)
- **Step 2-A** — `CLAUDE.md` 분석: 프로젝트 맥락·아키텍처·통신 규칙·기술 스택 섹션 추출
- **Step 2-B** — `basic-instruction.md` 분석: 금지 패턴은 반드시 **금지 패턴 + 이유 + 대안** 삼위일체로 작성
- **Step 3** — 두 문서 초안 동시 생성
- **Step 4** — 대화창에 미리보기 출력 후 승인 요청
- **Step 5** — 승인 시 `CLAUDE.md` (프로젝트 루트) / `.instruction/basic-instruction.md` 저장

**생성되는 두 문서의 역할**

**`CLAUDE.md`** — "이 프로젝트가 무엇인가"를 Agent에게 알려주는 문서
- 프로젝트 개요, 기술 스택 표, 아키텍처 다이어그램, API 엔드포인트 목록
- DB 테이블 구조, 핵심 로직 흐름, 실행 방법, 환경변수 목록

**`.instruction/basic-instruction.md`** — "어떻게 코드를 짜야 하는가"를 Agent에게 알려주는 문서
- 모듈 분리 원칙, 각 모듈의 책임과 금지 행위
- 금지 목록: 패턴 + 이유 + 대안을 세트로 명시

---

### Step 4 — 작업지시서 도출 (`/impl-doc`)

**작업지침서가 없으면 생기는 문제**

- 설계 문서가 있어도 "자, 이제 구현해줘" 하면 AI가 어디서부터 어떤 순서로 해야 할지 모르고 엉뚱한 것부터 시작하거나, FE와 BE가 따로 놂
- 작업지침서는 Phase 단위로 BE+FE를 페어로 묶어서, 한 Phase가 끝나면 사람이 눈으로 확인 가능한 결과물이 나오도록 설계함

**impl-doc 스킬이 하는 일**

- **Step 1** — 설계 문서 수집 + 불명확한 항목 최대 3개만 추가 질문
- **Step 2** — Phase 분할 설계
  - 각 Phase는 이전 Phase 없이 실행 불가능해야 함
  - 각 Phase 끝에 사람이 눈으로 확인 가능한 결과물 필수
  - BE만 또는 FE만 완성되는 Phase는 금지 — 합치거나 재분할
  - 한 Phase에 API 5개 이상이면 쪼갬
- **Step 3** — 태스크 작성 (ID, 의존, 구현 내용, Agent 지시, 검증 기준)
- **Step 4** — 각 Phase의 통합 검증 시나리오 작성

**태스크 ID 체계**

| ID 형식 | 의미 |
| ------- | ---- |
| `INF-XX` | 인프라·환경 세팅 |
| `BE-XX` | 백엔드 구현 태스크 |
| `FE-XX` | 프론트엔드 구현 태스크 |

## 10. 훅(Hook) 전략

### 판단 기준

> **써야 하는 훅**: "빠뜨리면 반드시 문제가 생기는 것"
> **쓰지 말아야 하는 훅**: "하면 좋은 것" — 수동으로 유지

### 권장 훅

| 훅 타입 | 시점 | 실행 내용 | 이유 |
|---------|------|-----------|------|
| `PreToolUse` (Bash) | git commit 명령 전 | scan.sh 자동 실행 | /pre-commit 잊는 경우 방지 |
| `Stop` | 세션 종료 시 | 코드 변경 감지 → doc-audit 알림 출력 | CLAUDE.md 괴리 축적 방지 |

**Stop 훅 예시:**

```json
"Stop": [{
  "hooks": [{
    "type": "command",
    "command": "git diff --name-only HEAD 2>/dev/null | grep -qE '\\.(ts|tsx|js|java|py)' && echo '⚠ 코드 변경 감지 — /doc-audit 실행 권장'"
  }]
}]
```

### 쓰지 말아야 하는 훅

| 훅 | 이유 |
|----|------|
| design-doc 완료 → context-doc 자동 실행 | 설계 문서 추가 수정 가능. 중간 자동 실행은 노이즈 |
| rfp-ingest 완료 → design-doc 자동 실행 | 제안 단계에서 바로 create-prototype으로 가는 분기가 막힘 |
| commit 완료 → rfp-compliance-review 자동 실행 | 매 커밋마다 RFP 전체를 읽는 건 비용 낭비 |

## 13. 하네스 설계 원칙

1. **명시적 트리거**: Hook 자동 실행 없음. 모든 스킬은 사용자가 명시적으로 호출.
2. **SFR 선택 실행**: RFP 전체를 일괄 처리하지 않음. 그때그때 필요한 SFR만.
3. **산출물 연결**: 각 스킬의 출력이 다음 스킬의 입력이 되는 파이프라인 구조.
4. **스킬 통합 원칙**: 유사 기능은 별도 스킬 분리 대신 옵션/플래그로 흡수.
   - `sfr-change-impact` → `sfr-trace --impact`
   - `rfp-clarify` → `rfp-ingest` Step 3 인라인 처리
5. **게이트 포지셔닝**: 게이트는 언제, 무엇을 막는지로 구분.
   - `pre-commit`: 코드 규칙 게이트 (커밋 전)
   - `rfp-compliance-review`: 요구사항 게이트 (납품 전)
6. **프로토타입 유동 배치**: `design-prototype-docs` → `create-prototype` 파이프라인은 impl-* 전·후·양쪽 어디든 실행 가능. 화면을 먼저 보고 impl을 쓸지, impl을 먼저 확정하고 정밀 목업을 뽑을지는 상황에 따라 선택.
7. **서브에이전트 사용자 위임**: 병렬 서브에이전트 사용 여부·모델·개수는 스킬이 강제하지 않고 사용자가 선택.

## 14. 바이브코딩 운영 노하우

### 웹 AI vs Agent(IDE) 역할 분담

| 상황 | 웹 AI (Gemini, Claude.ai) | Agent (Claude Code) |
| ---- | ------------------------ | ------------------- |
| 초기 아이디어 브레인스토밍 | ✅ | — |
| 설계 인터뷰 (긴 대화) | ✅ | ✅ `/design-doc` |
| 처음 쓰는 기술 조사 (웹 검색 포함) | ✅ | — |
| 우려 사항 고도화 인터뷰 | ✅ | — |
| Context 문서 생성 | ✅ 템플릿 | ✅ `/context-doc` |
| 작업지침서 도출 | ✅ 템플릿 | ✅ `/impl-fe-be-doc` 또는 `/impl-screen-doc` |
| 실제 코드 작성 | — | ✅ |
| 코드 리뷰·주석·커밋 | — | ✅ |
| 문서 동기화·괴리 분석 | — | ✅ |

- 웹 AI는 **긴 대화와 탐색**에 강함 — 아직 방향이 정해지지 않은 초반 단계
- Agent(IDE)는 **실제 파일과 코드에 접근**할 수 있어서 구현 단계에 강함
- 두 환경이 서로 대체하는 게 아니라 **역할을 나눠서 순서대로 사용**하는 것이 핵심

### 컨텍스트 창 낭비하지 않는 대화 설계

- 긴 탐색 대화(설계·우려 사항)는 웹 AI에서, 실제 구현 지시는 Agent에서
- Agent에게는 `"Phase 2의 BE-05 태스크 구현해줘. 작업지침서 참고해"` 같이 짧고 범위가 한정된 지시가 효과적
- 대화 초반에 `CLAUDE.md`, 작업지침서를 언급해두면 Agent가 알아서 읽고 컨텍스트를 채움

### Phase 단위 체크포인트

- Phase 하나 완료 → 검증 시나리오 직접 실행 → `/multi-review` → `/pre-commit` → `/commit`
- git 체크포인트를 남겨두면 Agent가 이상한 방향으로 갔을 때 되돌리기 쉬움
- "Phase 1~5 한 번에 다 짜줘"는 거의 항상 중간에 인터페이스가 안 맞는 문제 발생

### Agent에게 지시하는 방법이 결과를 가른다

좋은 지시는 짧고, 참조할 문서를 명시하고, 범위가 한정돼 있다.

```text
❌ 나쁜 지시
"로그인 기능이랑 대시보드 화면이랑 API도 연결해줘. 근데 에러 처리도 잘 해주고
 코드 구조도 깔끔하게 짜줘"

✅ 좋은 지시
"작업지침서(impl.md)의 Phase 2 — BE-03 태스크를 구현해줘.
 CLAUDE.md와 basic-instruction.md 참고. 다른 Phase는 건드리지 말 것."
```

- "다른 건 건드리지 말 것"이라는 범위 한정 한 줄이 생각보다 큰 차이를 만든다

### 컨텍스트가 오염됐다는 신호를 알아두기

Agent와 대화가 길어지면 초반의 설계 방향이나 규칙이 "희석"되는 현상이 생긴다.
아래 신호가 보이면 새 대화를 시작하는 것이 낫다:

- Agent가 이미 결정된 기술 스택이나 구조를 다시 물어보기 시작할 때
- 이전에 금지했던 패턴이 다시 등장할 때
- 구현한 코드가 `CLAUDE.md`의 아키텍처 구조와 달라지기 시작할 때

새 대화를 시작할 때는 `CLAUDE.md`와 작업지침서를 다시 언급하는 것으로 충분하다.
Claude Code에서는 `/clear`로 대화 히스토리를 비우되, 참조 문서는 그대로 남아 있어서 컨텍스트가 금방 복원된다.

### 금지 패턴 작성 삼위일체

`basic-instruction.md` 금지 패턴은 반드시 **패턴 + 이유 + 대안** 세트로 작성.
단순히 "하지 마라"만 쓰면 Agent가 이유를 모르고 같은 실수를 반복한다.

```markdown
### ❌ DOM 전체를 LLM에 직접 전달
- **패턴**: `llm.invoke(full_html_string)`
- **이유**: 토큰 초과로 컨텍스트 창 초과 또는 응답 품질 급락
- **대안**: missing_selectors 주변 영역만 추출하거나 3,000자로 잘라서 전달
```

### 설계 문서는 한 번 만들고 끝이 아님

- 구현하다 보면 설계가 바뀜 — 라이브러리가 교체되거나, 구조가 달라지거나, API가 추가됨
- 이 변경이 설계 문서나 `CLAUDE.md`에 반영되지 않으면 Agent가 "설계와 다른 코드"를 계속 참조하게 됨
- `doc-audit` → `agent-sync` 조합으로 주기적으로 갱신해주는 것이 중요

### 반복 작업은 자동화 대상

- 같은 작업을 세 번 이상 반복하게 되면, 그 작업을 `skill-designer`로 Skill로 만드는 걸 고려함
- 처음에는 수동으로 하다가, 패턴이 보이면 그때 자동화하는 순서가 효율적

## 15. 자주 마주치는 상황들

### Agent가 설계와 다른 구조로 짜기 시작했을 때

1. 즉시 멈춘다 — 코드가 많이 쌓일수록 되돌리기 어려워짐
2. `git diff`로 무엇이 달라졌는지 확인
3. `CLAUDE.md`의 아키텍처 섹션과 비교해서 어디서 벗어났는지 짚어냄
4. "○○ 파일은 되돌려. 아키텍처는 CLAUDE.md 기준으로 다시 구현해줘" 식으로 재지시
5. 같은 실수가 반복된다면 `basic-instruction.md`의 금지 패턴에 추가

### Agent가 요청하지 않은 곳까지 건드렸을 때

- `git diff`로 변경 범위를 먼저 확인
- Phase 외 파일이 수정됐다면 `git checkout -- <파일명>`으로 되돌림
- 이후 지시에 "이 파일들만 수정할 것: ○○, △△" 식으로 범위를 명시

### 설계 문서와 코드가 서로 달라진 느낌이 올 때

- `doc-audit` 스킬로 괴리를 분석
- 괴리가 있으면: 코드가 맞고 설계가 구식인 경우가 많음 → 설계 문서 갱신 후 `agent-sync`로 동기화

### Phase가 너무 커서 중간에 막혔을 때

- Phase 하나에 API 5개 이상, 또는 BE+FE 합쳐서 파일이 10개 이상이면 중간에 인터페이스가 안 맞는 경우가 생김
- 이 경우 Phase를 더 잘게 나눠서 `impl-doc`을 다시 작성하는 것이 낫다
