## 4. 사람이 놓쳤을 가능성이 있는 추가 항목

### 4-1. context-doc / harness-bootstrap 복수 앱 저장 단계 미완 (P1)

**발견**: `context-doc` Step 6과 `harness-bootstrap` Step 7의 저장/검증 단계가 단일 앱 경로(`승인 시 .docs/instruction/ 디렉토리가 없으면 생성`, `@.docs/instruction/*` 검증)로만 되어 있다.

**영향**: 복수 앱에서 실행하면 `.docs/{앱}/instruction/`이 아닌 `.docs/instruction/`에 저장하고, `.docs/{앱}-context.md`와 `.docs/root-context/*` 검증이 빠진다.

**원인 추정**: Phase 2에서 STEP 0(게이트)과 미리보기(Step 5/7)에만 단일/복수 분기를 넣고, 실제 저장/검증 로직까지 분기를 넣지 못한 것.

### 4-2. 프로토타입 경로 의사결정 재확정 필요 (P1)

**발견**: 계획서 C-2와 운영 문서는 복수 앱 프로토타입을 프로젝트 루트 `.docs/prototype/{사용자}/{식별자}/`로 명시. 스킬 SKILL.md는 `.docs/{앱}/prototype/{사용자}/{식별자}/`로 구현.

**영향**: 후속 스킬, 운영 문서, 루트 컨텍스트 템플릿이 서로 다른 경로를 참조한다.

**판단**: 개선안 원문 12-3, 12-4, 13-3, 13-4를 다시 보면 "프로젝트 상위폴더/.docs/prototype/"로 되어 있어 **앱별 분리 없이 프로젝트 공통 위치**가 원래 의도에 가깝다. 재확정 필요.

### 4-3. agent-sync의 docs-sync.md가 복수 앱 경계를 반영하지 않음

**발견**: SKILL.md에는 "복수 앱 루트 미관리 파일은 harness-setup 전담, agent-sync 접근 금지"라고 인라인되어 있지만, 실제 실행 절차인 `prompts/docs-sync.md`는 여전히 "프로젝트 루트부터 재귀적으로 탐색"하고 CLAUDE.md↔AGENTS.md를 체크섬 비교 후 덮어쓴다.

**영향**: 복수 앱 루트의 미관리 파일을 agent-sync가 건드릴 수 있다. SKILL.md의 경계 선언이 prompts에서 무시된다.

### 4-4. parallel-setup.md 7개 파일에 구체 모델명 잔존

**발견**: `agent-sync`, `code-comment`, `context-doc`, `create-prototype`, `doc-audit`, `multi-review`, `rfp-ingest`의 `prompts/parallel-setup.md`에 `claude-haiku-4-5`, `gpt-5.4-mini`, `gemini-3.5-flash` 등이 하드코딩되어 있다.

**개선안 취지**: 모델은 계속 바뀌므로 하드코딩하지 않는다 (개선안 2번, 계획서 C-3).

### 4-5. harness-setup prompts의 Windows/PowerShell 미대응

**발견**: 실행 절차 prompts(`update-mode.md`, `single-app-setup.md`, `multi-app-setup.md`)가 `cp -r`, `rm -rf`, `mkdir -p`, `md5sum` 등 POSIX 명령 중심. 사용자 환경은 Windows 11.

### 4-6. commit 스킬에 사용자 재확인 게이트 없음

**발견**: 개선안 8-2는 "무조건 사용자 재확인"을 요구하지만, `commit` SKILL.md의 C-1 게이트에는 판정+scope 규칙만 있고 "맞습니까? (승인/수정/취소)" 게이트가 없다.

### 4-7. frontend-design에 "프로토타입/실제 소스" 구분 확인 누락

**발견**: 개선안 14-2는 "프로토타입에 적용하는 건지, 실제 소스코드에 적용하는건지 확인받는다"를 요구하지만, STEP 0에 이 질문이 없다.

### 4-8. Intro 문서 "빠른 선택 가이드"에 harness-setup 누락

**발견**: 스킬 테이블에는 추가됐지만, 226~235줄의 빠른 선택 가이드 목록에 "프로젝트에 하네스를 처음 설치한다 → `harness-setup`" 항목이 없다.

### 4-9. Intro 문서 `.instruction` 경로가 3곳 남아 있음

**발견**:
- 125줄: `.instruction/*-instruction.md` → `.docs/instruction/*-instruction.md`
- 152줄: `주제별 .instruction/*` → `주제별 .docs/instruction/*`
- 379줄: `@./.instruction/basic-instruction.md` → `@./.docs/instruction/basic-instruction.md`

### 4-10. custom-skill-design 체크리스트의 `.docs` 표준이 실제 C-2와 맞지 않음

**발견**: 체크리스트 32~34줄이 "단일앱: `.docs/{산출물유형}/...`, 복수앱: `.docs/{앱}/{산출물유형}/...`"이지만 실제 C-2는 스킬별 예외가 많다(design-doc: `.docs/DESIGN.md`, context-doc: 루트 `CLAUDE.md` 등). 새 스킬 생성 시 잘못된 기준으로 검증할 위험.

### 4-11. `.claude/skills`, `.agents/skills` 시드와 정본 동기화 확인

**발견**: Phase 7에서 `.claude/skills`와 `.agents/skills`에 `harness-setup`과 `custom-skill-design`을 복사했다. 이후 수정 사항이 있으면 시드도 함께 갱신해야 하는데, 이 과정을 자동화하는 절차가 없다. CLAUDE.md에 "스킬 수정 뒤 시드도 함께 갱신해야 한다"고 규칙은 있지만, 실수 방지 장치는 없다.

## 6. 결론

이번 리팩토링은 **방향과 큰 틀은 올바르다**. 개선안의 핵심인 단일/복수 앱 일반화, harness-setup 신설, agent-sync 경계 재정의, custom-skill-design 리네이밍은 SKILL.md 수준에서 잘 반영됐다.

**하지만 "SKILL.md에서 선언한 원칙"과 "prompts/templates에서 실행하는 절차"가 아직 완전히 붙지 않은 곳이 있다.** 가장 큰 위험은:

1. `context-doc` / `harness-bootstrap`의 저장 단계가 복수 앱 경로를 모른다
2. `harness-setup` 갱신 모드의 복사 명령이 디렉토리를 깨뜨릴 수 있다
3. `agent-sync`의 docs-sync가 SKILL.md의 경계 선언을 무시한다

이 3건은 실제 사용 시 바로 문제를 만들므로 **다른 작업보다 먼저 보완해야 한다.**

나머지는 혼란 방지(잔재 정리), 안전장치 보강(게이트 추가), 문서 품질(경로 통일, 용어 설명) 순서로 처리하면 된다.
