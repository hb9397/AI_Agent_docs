## 0. 이 문서의 목적과 사용법

개선안의 핵심 목표는 **"단일 애플리케이션만 가정하던 스킬셋을, 단일/복수 애플리케이션 프로젝트 모두에서 동작하도록 일반화"** 하는 것이다.

이 계획서는 그 목표를 **실행 가능한 Phase 단위**로 분해한다. 각 Phase는 아래 형식을 따른다.

- **목표** — 무엇을 끝내야 하는가
- **대상** — 어떤 파일/스킬을 건드리는가
- **작업 내용** — 구체적 변경 항목
- **완료 기준(DoD)** — 이 Phase가 끝났다고 판단하는 조건
- **선행 의존성** — 어떤 Phase가 먼저 끝나야 하는가

Phase는 원칙적으로 순차 실행이지만, 일부는 병렬 가능하다(§ 실행 순서 참조).

### 1-2. 점검에서 드러난 핵심 사실

1. **현재 어떤 스킬도 단일/복수 애플리케이션을 분기하지 않는다.** 모든 산출물 경로가 "스킬 수행 루트 = 애플리케이션 루트"라는 단일 가정에 묶여 있다. → 리팩토링의 본질은 **"모든 산출물·적용범위 스킬에 단일/복수 분기 + 애플리케이션(폴더) 확인 게이트를 주입"** 하는 것이다.
2. **`.claude/skills`, `.agents/skills`가 현재 비어 있다.** 개선안 README 항목(163줄)이 요구하는 "이 레포엔 신규 setup 스킬 + skill-design만 둔다"는 아직 적용 전 상태다.
3. **디렉토리명과 name 불일치**: 디렉토리는 `skill-design`인데 frontmatter `name`은 `skill-designer`다. 문서(README·Harness)도 `skill-designer`로 참조한다. → D-3 확정: **디렉토리·name 모두 `custom-skill-design`으로 통일**.
4. **frontmatter에 `model:` 필드는 없음**(개선안 2 만족). 단, `agent: fork`가 일부 스킬(agent-sync, code-comment, doc-audit)에 하드코딩되어 있다 → 개선안 2의 "서브에이전트/병렬 사용 여부를 반드시 물어보게" 와 충돌. 자동 fork를 **물어보는 게이트**로 전환 필요.
5. **impl 산출물 경로가 최근 커밋에서 `.docs/{사용자}/impl-{기능}.md`로 통일됐다**(git log `753db78`). 그런데 개선안 6-2/6-3은 `.docs/(애플리케이션)/impl-doc/{사용자}/{기능}.md`를 요구한다. → **직전 통일 작업과 경로 규칙이 충돌**한다 → 의사결정 D-2(가장 중요).

### 2-1. harness-setup ↔ agent-sync 책임 경계 (D-4 세부)

두 스킬은 표면상 겹치지만 동기화 **방향**이 다르므로, 아래 경계를 **두 스킬 본문에 각각 인라인 명시**한다(기존0-6).

| 영역 | 소유 스킬 | 비고 |
|------|-----------|------|
| 정본 레포(`{프로젝트명}-AI-Harness-docs`) → 프로젝트 `.claude/skills`·`.agents/skills` **pull** | **harness-setup 전담** | 상류→하류 단방향 |
| 복수앱 **루트 미관리** `CLAUDE.md`/`AGENTS.md` + `.docs/root-context/` 복사본 | **harness-setup 전담** | agent-sync 접근 **금지**(이중 관리 방지) |
| git 관리 컨텍스트 일치(단일앱 루트, 앱별 `{앱}-context.md`) + 로컬 편집 스킬 횡적 미러(`.claude/skills↔.agents/skills`) | **agent-sync** | 횡적·변경기반 |

> 충돌 위험은 "복수앱 루트 미관리 컨텍스트" 한 곳뿐이며, agent-sync를 그 영역에서 배제하면 단일/복수 모두 안전하다.

## 3. 공통 규약 (모든 Phase가 참조하는 표준)

개선안 기존0-6 "규칙을 스킬 간 참조하지 말고 각 스킬에 명시" 원칙에 따라, 아래 표준은 **각 스킬 본문에 인라인 복제**한다(이 계획서는 단일 출처가 아니라 작업 기준일 뿐).

### 3-1. 단일/복수 애플리케이션 판별·확인 게이트 (공통 블록 C-1)

모든 산출물·적용범위 스킬의 **맨 앞**에 삽입할 표준 절차:

```
1. 현재 수행 위치 탐색 (git repo 경계, 하위 애플리케이션 폴더 후보 스캔)
2. 단일 애플리케이션 프로젝트인지 / 복수 애플리케이션 프로젝트인지 판정
3. 판정 결과 + 적용 대상 애플리케이션(폴더)을 사용자에게 반드시 재확인 (무조건)
4. 확인된 범위 밖은 건드리지 않음
```

### 3-2. `.docs` 경로 표준 (C-2)

| 스킬 | 단일 앱 경로 | 복수 앱 경로 |
|------|-------------|-------------|
| design-doc | `.docs/context-base/DESIGN.md` | `.docs/{앱}/context-base/DESIGN.md` |
| context-doc (instruction) | `.docs/instruction/*-instruction.md` | `.docs/{앱}/instruction/*-instruction.md` |
| context-doc (컨텍스트) | 루트 `CLAUDE.md`/`AGENTS.md` | `.docs/{앱}-context.md` (+ 루트 통합 인덱스 `CLAUDE.md`/`AGENTS.md`) |
| impl-doc / impl-fe-be-doc | `.docs/impl-doc/{사용자}/{기능}.md` | `.docs/{앱}/impl-doc/{사용자}/{기능}.md` |
| design-prototype-docs / create-prototype | `.docs/prototype/{사용자}/{기능\|요구사항ID\|RFP ID}/` | `.docs/prototype/{사용자}/{기능\|요구사항ID\|RFP ID}/` (프로젝트 루트 기준) |

- `{사용자}` = git 계정 우선 탐색, **반드시 사용자에게 확인**.
- 모든 산출물은 **신규 생성 전 중복/유사 문서 존재 여부 확인** 후 있으면 갱신, 상단에 **갱신일자** 표기.
- (2026-06-28 개정) design-doc 산출 경로를 `context-base/` 하위로 이동 — 단일 `.docs/context-base/DESIGN.md`, 복수 `.docs/{앱}/context-base/DESIGN.md`. 상단 §3-1 게이트·아래 회차 기록의 구 경로 표기는 당시 기록으로 보존.

### 3-3. SKILL.md 양식 표준 (C-3)

- frontmatter는 Claude 양식 유지(`name`, `description`, `allowed-tools`), **Codex에서도 해석 가능하도록 본문은 플랫폼 중립 서술**.
- `model:` 필드 명시 금지(있으면 제거).
- `agent: fork` 등 플랫폼 전용 자동화는 frontmatter 하드코딩 대신 **"서브에이전트/병렬 처리 사용 여부를 반드시 질문"** 하는 본문 게이트로 전환(Codex는 Codex 방식에 위임).
- 스킬의 근본 기능은 변형하지 않는다(개선안 기존0-4).
- 초기화(생성) vs 갱신을 구분 처리(개선안 기존0-5).

§1-1에서 "산출물 생성 + 경로 분기 필요"로 분류된 스킬들을 §3 규약에 맞춰 일괄 개편한다. 스킬별로 동일 패턴(C-1 게이트 + C-2 경로 + C-3 양식)을 적용하되, 규칙은 각 스킬에 인라인 복제한다.

- **목표**: 설계·컨텍스트·구현·프로토타입·RFP 계열의 모든 산출물 경로를 단일/복수 분기.
- **대상 및 개별 작업**:

  | 스킬 | 핵심 변경 (개선안 항목) |
  |------|--------------------------|
  | `design-doc` | C-1 게이트 추가. 단일 `.docs/DESIGN.md` / 복수 `.docs/{앱}-DESIGN.md` (3) |
  | `context-doc` | C-1 + DESIGN 참조. 단일=루트 CLAUDE/AGENTS + `.docs/instruction/*` / 복수=`.docs/{앱}-context.md` + `.docs/{앱}/instruction/*` + 루트 통합 인덱스 (4) |
  | `harness-bootstrap` | design-doc·context-doc와 동일 분기 원칙 적용 (5) |
  | `impl-doc` | C-1. D-2 경로. 기능명 적절성·갱신일자·중복 확인·타 사용자 문서 표기 (6) |
  | `impl-fe-be-doc` | impl-doc과 동일 규칙 (6) |
  | `impl-reuse-scan` | C-1. 입력 impl 경로를 D-2와 일치시키고 사용자 재확인 (7) |
  | `impl-verify` | C-1. 입력 impl 경로 D-2 일치, 사용자 재확인 (7) |
  | `design-prototype-docs` | C-1. RFP ID/요구사항 ID 확인. `.docs/prototype/{사용자}/{id}/design-doc.md` (12) |
  | `create-prototype` | C-1. 동일 prototype 경로에 산출 (13) |
  | `rfp-ingest` | 해석 위주 유지, `rfp-design-input-{SFR}.md` 산출물 폐지 (16, D-7) |

- **완료 기준**: 각 스킬이 단일/복수 두 경우 모두에서 올바른 경로를 안내하고, 시작 시 C-1 확인 게이트를 거침. impl 3종 입력/출력 경로 정합.
- **선행 의존성**: Phase 0(특히 D-2). Phase 1과 병렬 가능(구조 계약만 합의되면).

### Phase 3 — 적용범위 확인 스킬군 개편

산출물이 아니라 **코드/커밋/리뷰처럼 "어떤 애플리케이션(폴더)에 적용할지"** 가 중요한 스킬들. (개선안 8·9·14, 기존0-6)

- **목표**: 복수 앱 환경에서 작업 범위를 탐색→사용자 재확인 후에만 적용.
- **대상 및 작업**:

  | 스킬 | 핵심 변경 |
  |------|-----------|
  | `code-comment` | C-1. 적용 대상 애플리케이션(폴더) 탐색·재확인. `agent: fork` → 질문 게이트 (8) |
  | `multi-review` | C-1. 리뷰 대상 폴더 확인. `Task` 병렬 사용 여부 질문 (8) |
  | `pre-commit` | C-1. 검사 대상 폴더 확인 (8) |
  | `commit` | C-1. 커밋 대상 애플리케이션 repo 확인(복수 앱은 repo가 분리됨) (8) |
  | `doc-audit` | C-1. 산출물 위치 + 앱별 소스 위치 확인·준수, 재확인. `agent: fork` → 질문 게이트 (9) |
  | `frontend-design` | C-1. 프로토타입 vs 실제 소스 대상 확인. 산출물/소스 위치 준수 (14) |

- **완료 기준**: 복수 앱 환경에서 각 스킬이 대상 폴더를 좁히고 사용자 확인 없이는 적용하지 않음.
- **선행 의존성**: Phase 0. Phase 2와 병렬 가능.
