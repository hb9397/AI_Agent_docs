## 1. 사실과 다른 점

### 1-1. README의 주요 문서 링크가 현재 파일 위치와 맞지 않는다

현재 실제 파일:

- `Docs/Harness_Engineering_Intro.md`
- `Docs/Harness_Engineering.md`
- `Docs/Agent_Skills_Repo_Structure_Analysis.md`

하지만 `README.md`는 여전히 아래 링크를 사용한다.

- `./Harness_Engineering_Intro.md`
- `./Harness_Engineering.md`
- `./Agent_Skills_Repo_Structure_Analysis.md`

마크다운 링크 실존 검사 결과, README의 위 링크들은 모두 깨진 상태다.

권장 수정:

- README 상단 링크와 하단 `Harness Engineering 문서` 섹션을 `./Docs/...`로 바꾼다.
- 디렉토리 구조 예시에도 `Docs/`와 `improvement_plan/`을 추가한다.

### 1-2. AGENTS.md / CLAUDE.md의 참조 문서 경로가 현재 구조와 맞지 않는다

현재 실제 파일:

- `Docs/Harness_Engineering.md`
- `Docs/Harness_Engineering_Intro.md`
- `Docs/Agent_Skills_Repo_Structure_Analysis.md`
- `improvement_plan/20260627/리팩토링 작업 계획서.md`

하지만 `AGENTS.md`, `CLAUDE.md`는 아래처럼 루트 파일을 전제한다.

- `Harness_Engineering.md`
- `Harness_Engineering_Intro.md`
- `리팩토링 작업 계획서.md`
- `Agent_Skills_Repo_Structure_Analysis.md`

영향:

- 이 레포에서 작업하는 Agent가 운영 문서 위치를 잘못 찾는다.
- "스킬 변경 시 Harness 문서도 함께 업데이트" 규칙이 실제 `Docs/` 경로를 가리키지 않는다.
- C-2 표준 참조가 `improvement_plan/20260627/리팩토링 작업 계획서.md`로 이어지지 않는다.

권장 수정:

- `AGENTS.md`, `CLAUDE.md`의 디렉토리 구조와 참조 문서를 현재 구조로 바꾼다.
- "스킬 변경 시 함께 업데이트" 문구도 `Docs/Harness_Engineering.md`, `Docs/Harness_Engineering_Intro.md`로 명시한다.

### 1-3. `harness-setup`의 정본 레포 감지 조건이 문서 이동 후 깨질 수 있다

`skills/harness-setup/prompts/detection.md`는 하네스 정본 레포 판별을 다음 조건으로 한다.

```bash
ls skills/harness-setup/SKILL.md 2>/dev/null && ls Harness_Engineering.md 2>/dev/null
```

현재 `Harness_Engineering.md`는 루트가 아니라 `Docs/Harness_Engineering.md`에 있다. 따라서 하네스 정본 레포 내부에서 `/harness-setup`을 실행하면 1차 감지가 실패할 수 있다.

영향:

- 하네스 정본 레포 내부 실행 시 "부모 폴더를 프로젝트 루트 후보로 설정"하는 흐름이 동작하지 않을 수 있다.
- 사용자가 직접 프로젝트 루트 경로를 입력해야 하는 fallback으로 밀릴 수 있다.

권장 수정:

```bash
ls skills/harness-setup/SKILL.md 2>/dev/null \
  && { ls Docs/Harness_Engineering.md 2>/dev/null || ls Harness_Engineering.md 2>/dev/null; }
```

또는 더 안정적으로 `skills/harness-setup/SKILL.md + README.md + Docs/Harness_Engineering.md` 조합으로 감지한다.

### 1-4. `Docs/Harness_Engineering.md`의 저장소 구조 예시가 현재 구조와 다르다

`Docs/Harness_Engineering.md`의 저장소 구조 예시는 아직 아래처럼 루트 문서 배치를 전제한다.

```text
AI_Agent_docs/
├── skills/
├── example/
├── Harness_Engineering_Intro.md
├── Harness_Engineering.md
└── README.md
```

현재는 `Docs/`와 `improvement_plan/`이 있다. 운영 가이드가 현재 저장소 기준 문서라면 이 구조 예시는 사실과 다르다.

권장 수정:

- `Docs/`와 `improvement_plan/20260627/`을 포함한 구조로 갱신한다.

### 1-5. `Docs/Agent_Skills_Repo_Structure_Analysis.md`가 현재 스킬 동작과 맞지 않는 설명을 포함한다

검색 결과 이 문서에는 아래 옛 설명이 남아 있다.

- `rfp-ingest`: `rfp-design-input-{SFR}.md`를 만든다고 설명
- `context-doc`: `.instruction/*-instruction.md`를 만든다고 설명

현재 확정 동작:

- `rfp-ingest`는 파일을 만들지 않고 대화 컨텍스트로 전달한다.
- `context-doc`는 단일 앱 `.docs/instruction/*-instruction.md`, 복수 앱 `.docs/{앱}/instruction/*-instruction.md`를 만든다.

해석:

- 이 문서가 "외부 하네스 비교 분석" 문서라면 과거 스냅샷으로 둘 수 있다.
- 하지만 README에서 현재 참고 문서로 링크한다면, 최소한 "현재 스킬 동작은 README/Harness_Engineering을 우선한다"는 주석이 필요하다.

권장 수정:

- 상단에 "이 문서는 외부 사례 비교 및 과거 분석 문서이며, 현재 레포의 최신 동작은 `Docs/Harness_Engineering.md`를 따른다"는 안내 추가.
- 가능하면 위 두 줄은 현재 동작에 맞게 직접 수정한다.

### 1-6. `rfp-ingest`의 `allowed-tools: Agent`는 아직 공통 양식 관점에서 불명확하다

현재 대부분의 병렬 스킬은 `Task`를 선언한다. 그런데 `skills/rfp-ingest/SKILL.md`는 다음처럼 선언한다.

```yaml
allowed-tools: Read, Glob, Grep, Bash, Agent
```

문제:

- `Agent`가 실제 지원 도구명인지 불명확하다.
- 계획서 C-3의 "플랫폼 전용 기능은 질문 게이트로 처리" 원칙과 맞물릴 때, `Agent` 선언은 Codex 중립성을 흐릴 수 있다.

권장 수정:

- 실제 도구명이 `Task`라면 `Agent`를 `Task`로 정정한다.
- 아니라면 `rfp-ingest`의 병렬화는 STEP 0 게이트에서만 다루고 frontmatter는 실제 공통 도구만 남긴다.

### 1-7. `harness-setup`의 `.docs` 형상관리 표현이 원문 의도보다 약하다

원문과 D-5는 복수 앱에서 `.docs`를 별도 레포로 형상관리하는 정책을 강하게 전제한다.

현재 `skills/harness-setup/prompts/multi-app-setup.md`는 `.docs`를 "별도 git 레포로 형상관리되는 프로젝트 전체 AI 문서 저장소"라고 설명한 뒤, 바로 아래에서 "별도 git 레포로 관리하는 것은 사용자의 선택"이라고 쓴다.

문제:

- 정책과 선택지가 한 문서 안에서 충돌한다.
- README도 "팀 공유가 필요하면 별도 GitHub/GitLab/Gitea 레포로 관리"라고 표현해 기본 정책이 약하게 읽힌다.

권장 수정:

- 복수 앱 표준은 `.docs` 별도 git 레포 관리라고 쓰고, "초기 개인 실험 단계에서는 아직 remote가 없을 수 있다" 정도로만 완화한다.

### 1-8. `Docs/AI_Agent_활용을 위한 개념.pptx`는 현재 하네스 운영 기준과 다른 일반 설명을 포함한다

PPTX 텍스트를 추출해 확인한 결과, 일반적인 AI Agent 개념 설명 자료다. 다만 현재 레포 운영 기준과 섞이면 오해될 수 있는 표현이 있다.

- Subagent 예시에 `model: opus`가 나온다.
- AI Docs 예시에 `.ai/instruction/`, `.ai/rules/`가 나온다.
- "Subagent는 Claude만 Production-ready" 같은 도구 상태 설명이 들어 있다.

이 자료는 소개용으로는 문제없지만, 현재 하네스의 C-3 원칙(`model:` 하드코딩 금지)과 직접 운영 기준으로 읽히면 충돌한다.

권장 수정:

- `Docs/` 인덱스 또는 PPT 주변 설명에 "개념 소개 자료이며, 현재 레포 운영 규칙은 README/AGENTS/Docs/Harness_Engineering을 따른다"는 안내를 둔다.

## 2. 변경계획서와 다른 점

### 2-1. 계획서의 대상 경로는 루트 문서 기준인데, 실제 문서는 `Docs/`로 이동되었다

`improvement_plan/20260627/리팩토링 작업 계획서.md`는 Phase 5~7 대상 파일을 아래처럼 잡는다.

- `Harness_Engineering.md`
- `Harness_Engineering_Intro.md`
- `Agent_Skills_Repo_Structure_Analysis.md`

현재 실제 파일은 `Docs/` 아래에 있다. 계획서가 의사결정 기록이면 괜찮지만, 현재 작업 기준으로 재사용하면 경로가 틀린다.

권장 수정:

- 계획서 상단에 "문서 파일은 후속 정리에서 `Docs/`로 이동됨"이라는 주석 추가.
- 또는 "현재 경로" 표를 추가해 루트 기준 경로와 실제 경로를 매핑한다.

### 2-2. Phase 8 통합 검증의 완료 판단은 문서 이동 후 다시 성립하지 않는다

계획서 §7 체크리스트는 "문서-스킬 정합, Phase 8 통합 검증 통과"를 완료로 체크한다.

하지만 이후 문서 이동으로 다음 항목이 다시 깨졌다.

- harness-setup 정본 레포 감지 조건
- Docs/Harness_Engineering 저장소 구조 예시

따라서 Phase 8 완료 기록은 "이동 전 기준"으로는 맞지만, 현재 구조 기준으로는 재검증이 필요하다.

권장 수정:

- 3차 후속 Phase를 추가한다.
- 이름 예: "Phase 9 — 문서 디렉토리 재배치 후 경로 정합성 검증".

### 2-3. 원문 대원칙 0-0의 정리 순서가 현재 이동 구조에는 반영되었지만, 안내 문서에는 반영되지 않았다

원문은 스킬 → Harness_Engineering → Intro → README 순서를 요구했다.

현재 실제 정리는 그 순서를 따른 뒤, 문서를 `Docs/`와 `improvement_plan/`으로 재배치한 상태로 보인다. 하지만 README/AGENTS/CLAUDE는 아직 이 재배치 사실을 반영하지 못했다.

권장 수정:

- "문서 재배치"를 별도 후속 작업으로 기록한다.
- 루트 문서들은 사용자 진입점, `Docs/`는 운영 문서, `improvement_plan/`은 의사결정·점검 이력이라는 역할을 명확히 한다.

### 2-4. `harness-bootstrap`의 다른 스킬 파일 참조는 계획서의 "규칙 인라인" 원칙과 여전히 긴장 관계가 있다

- `../context-doc/prompts/analysis-instruction.md`
- `../context-doc/templates/CLAUDE.md.template`

이 구조는 중복을 줄이는 장점이 있지만, 원문의 "스킬 적용 법칙을 스킬 파일간 참조하지 말고 각각의 스킬에 명시"와 충돌할 수 있다.

권장 결정:

- 예외로 인정한다면 계획서와 `harness-bootstrap` 본문에 명시한다.

## 3. 원문 텍스트 의도 반영 여부

### 3-1. 잘 반영된 의도

- 단일/복수 애플리케이션 프로젝트를 모두 지원한다는 핵심 방향은 `skills/`와 `Docs/Harness_Engineering.md`에 반영되어 있다.
- `harness-setup`이 정본 레포에서 프로젝트로 스킬을 설치·갱신하는 진입점으로 추가되었다.
- 복수 앱의 루트 미관리 `CLAUDE.md`/`AGENTS.md`와 `.docs/root-context/` 관계가 스킬과 README에 반영되어 있다.
- `agent-sync`는 정본 pull이 아니라 횡적 동기화로 역할이 재획정되었다.
- `custom-skill-design`은 `skill-design`/`skill-designer` 혼선을 정리하고 새 표준(C-1/C-2/C-3)을 생성기에 반영했다.
- `Docs/Harness_Engineering.md`는 단일 메인 흐름으로 재구성되었고, RFP/prototype 흐름은 옵션으로 분리되었다.

### 3-2. 덜 반영되었거나 현재 구조에서 다시 깨진 의도

#### 루트 진입 문서가 현재 레포 구조를 제대로 안내하지 못한다

원문은 README에 "해당 레포지토리를 설정하는 방법"을 잘 정리하라고 했다. 현재 README 내용 자체는 좋아졌지만, `Docs/` 이동 후 링크가 깨져 진입 문서 역할이 약해졌다.

#### Agent 운영 문서가 실제 문서 위치를 잘못 알려준다

원문은 CLAUDE.md/AGENTS.md를 "앞으로 이 하네스를 관리하기 위한 대원칙 및 스킬 내용"으로 만들라고 했다. 현재 두 파일은 기본 규칙은 좋지만, 실제 문서 위치가 틀려 Agent 운영 가이드로서 위험하다.

#### 복수 앱 `.docs` 별도 레포 정책이 문서마다 강도가 다르다

원문은 복수 앱에서 `.docs`를 별도 레포로 형상관리하는 의도가 강하다. 현재 일부 문서는 "별도 git 레포", 일부 문서는 "사용자 선택"처럼 읽힌다.

#### 비교 분석 문서와 PPT가 현재 운영 기준과 구분되지 않는다

`Docs/Agent_Skills_Repo_Structure_Analysis.md`와 PPTX는 참고/소개 문서로는 가치가 있지만, 현재 하네스 운영 기준과 일부 표현이 다르다. README에서 직접 링크한다면 "최신 운영 기준이 아님"을 명시해야 한다.

### Phase 11 — 복수 앱 `.docs` 형상관리 정책 표현 통일

대상:

- `README.md`
- `skills/harness-setup/prompts/multi-app-setup.md`
- `.claude/skills/harness-setup/prompts/multi-app-setup.md`
- `.agents/skills/harness-setup/prompts/multi-app-setup.md`
- `Docs/Harness_Engineering.md`

작업:

1. 복수 앱 표준 정책을 `.docs` 별도 git 레포 관리로 통일한다.
2. "사용자의 선택" 표현은 "초기 개인 실험 단계에서는 remote 연결 전일 수 있음" 정도로 낮춘다.
3. `.docs` git init을 harness-setup이 자동으로 할지, 안내만 할지 결정한다.

결정 필요:

- `harness-setup`이 `.docs`에 `git init`까지 자동 수행할 것인가?
- 아니면 `.docs` 디렉토리 생성 후 사용자가 별도 레포 연결하도록 안내만 할 것인가?

권장:

- 자동 `git init`은 하지 않고, "별도 git 레포로 관리해야 한다"는 안내와 체크리스트를 출력한다. 사용자의 remote/권한/조직 정책까지 스킬이 결정하면 과하다.

### Phase 12 — C-3 도구 선언 잔여 정리

대상:

- `skills/rfp-ingest/SKILL.md`

완료 기준:

- 지원되지 않는 도구명 없음.

## 6. 결론

현재 리팩토링의 핵심 기능 방향은 대체로 맞다. 1차·2차에서 발견된 스킬 실행 경로 문제도 상당 부분 보완되었다.

하지만 `Docs/`와 `improvement_plan/`으로 문서를 이동하면서, 루트 README/AGENTS/CLAUDE 및 `harness-setup` 감지 로직이 새 구조를 따라오지 못했다. 이 상태에서는 사용자가 README 링크를 눌렀을 때 문서를 못 찾고, Agent는 잘못된 참조 문서 경로를 기준으로 작업하며, `harness-setup`은 정본 레포 내부 실행을 제대로 감지하지 못할 수 있다.

따라서 3차 개선은 스킬 본문 대수정보다 **문서 재배치 후 경로 정합성 복구**를 최우선으로 잡아야 한다.
