# Design Prototype Docs — 프로토타입 화면 설계 문서 생성기

기능요구사항을 분석하여, `/create-prototype` 스킬의 입력으로 사용할 수 있는 **목업 디자인 문서(.md)** 를 생성하는 스킬이다.

```
design-prototype-docs OUTPUT (.md)
    └─→ create-prototype INPUT  →  HTML/CSS/JS/JSON 프로토타입
```

---

## 워크플로우

### STEP 0 — 플랫폼·실행 방식 확인 + 프로젝트 유형 확인

#### STEP 0-A — 플랫폼·실행 방식 확인

사용자에게 아래를 확인한다:

> 1. 서브에이전트(병렬 처리)를 사용할 수 있는 환경인가요? (Claude Code / Codex / 기타)
> 2. 사용할 경우 병렬 실행을 원하시나요?

서브에이전트 미지원 또는 미사용 선택 시 순차 실행한다.

#### STEP 0-B — 프로젝트 유형 확인 (C-1 확인 단계)

프로토타입 설계 문서 생성 범위를 확정하기 위해 **반드시** 아래를 수행한다.

1. 현재 수행 위치에서 프로젝트 구조를 탐색한다.
2. **단일 애플리케이션 프로젝트**인지 **복수 애플리케이션 프로젝트**인지 판정한다.
3. 판정 결과 + 적용 대상을 사용자에게 **반드시 재확인**한다.

#### STEP 0-C — 기능 이름/요구사항 ID/RFP ID 확인

폴더명에 적용할 식별자를 **반드시** 사용자에게 확인한다.

> "폴더명에 사용할 식별자를 알려주세요:
> - 제안 단계면 → RFP ID (예: RFP-001)
> - 설계/구현 단계면 → 요구사항 ID (예: SFR-019)
> - 둘 다 없으면 → 기능 이름 (예: bidding-analysis)"

#### STEP 0-D — 사용자 이름 확인

`git config user.name`으로 현재 git 계정을 탐색하고, **반드시** 사용자에게 확인한다.

### STEP 1 — 기능요구사항 수집

스킬 호출 시 **반드시 먼저** 기능요구사항을 수집한다.

사용자에게 이렇게 물어본다:
> "프로토타입으로 만들 기능의 요구사항을 알려주세요.
>
> **입력 방법** (아무 방식이든 OK):
> - 📎 파일 첨부 (제안요청서, 기획서, PRD, 노션 export 등)
> - 📝 텍스트로 직접 설명
> - 📌 `@파일경로` 로 문서 위치 알려주기
> - 위 방식을 조합해도 됩니다
>
> 기능요구사항 번호(SFR, REQ 등)가 있으면 함께 알려주세요."

사용자가 제공한 자료에서 아래 항목을 추출한다:

| 추출 항목 | 설명 |
|---|---|
| 요구사항 번호 | SFR-XXX, REQ-XXX 등 (없으면 기능명에서 추론) |
| 요구사항 명칭 | 공식 명칭 또는 핵심 기능 요약 |
| 프로젝트명 | 소속 프로젝트/시스템명 |
| 핵심 기능 목록 | 제안요청서/기획서에 명시된 세부 기능 전체 |
| 사용자 역할 | 이 화면을 쓰는 사람 (관리자, 실무자, 일반 사용자 등) |

### STEP 2 — 기능 분석 및 화면 분리

수집된 요구사항을 분석하여 **화면 분리 기준**을 도출한다.

**화면 분리 원칙**:
- 요구사항의 **대분류 기능 단위**로 화면을 나눈다
- 하나의 화면 = 하나의 가장 큰 독립 기능 단위
- 사용자가 "이 기능을 수행하기 위해 다른 화면으로 이동할 필요가 없는" 단위
- 화면 간에 데이터 흐름이 있으면 네비게이션 연계로 표현

분석 결과를 사용자에게 확인받는다:
> "요구사항을 분석한 결과, **{N}개 화면**으로 구성하는 것이 적절합니다:
>
> | # | 화면명 | 대분류 기능 | 레이아웃 패턴 |
> |---|---|---|---|
> | 1 | {화면명} | {기능} | {패턴} |
> | ... | ... | ... | ... |
>
> 이 구성으로 진행할까요? 화면을 합치거나 나누고 싶은 부분이 있으면 알려주세요."

### STEP 3 — 메인 색상 수집 (미제공 시에만)

사용자에게 **메인 색상 하나**만 물어본다. 이미 제공된 경우(기존 프로젝트 색상 등) 건너뛴다.

> "디자인의 메인 색상을 알려주세요. (예: #2563EB, 또는 '네이비', '초록' 등)
> 기존 프로젝트 색상 체계가 있으면 그것을 알려주셔도 됩니다."

### STEP 4 — 문서 생성

STEP 1~3에서 수집한 정보를 종합하여 목업 디자인 문서를 생성한다.

**생성 전 반드시 아래 파일을 읽는다:**
- `example/mock_output.md` — 실제 산출물 예시. 섹션 구성, 서술 수준, 표 형식, 도식화 스타일의 기준으로 삼는다.

## 문서 구조 (필수 섹션)

생성되는 .md 파일은 아래 구조를 **반드시** 포함해야 한다.

### 1. 기본 정보 테이블

```markdown
| 항목 | 내용 |
|---|---|
| **요구사항 번호** | SFR-020 |
| **요구사항 명칭** | 전기넷 입찰-낙찰정보 분석 |
| **프로젝트명** | 한국전기안전공사 AI플랫폼 |
| **메인 색상** | `#0D47A1` (딥 네이비) |
| **화면 수** | 5개 |
```

### 2. 요구사항 요약

- 원문 또는 핵심 요약
- 세부 기능 목록 (번호 매긴 리스트)

### 3. 화면 목록 테이블

각 화면에 대해 HTML, CSS, JS, JSON 파일명을 모두 명시한다.

| 화면 번호 | 화면명 | HTML | CSS | JS | JSON | 레이아웃 패턴 |
|---|---|---|---|---|---|---|
| 1 | {화면명} | `display/{PREFIX}-XXX-{slug}.html` | `css/{PREFIX}-XXX-{slug}.css` | `script/{PREFIX}-XXX-{slug}.js` | `data/{PREFIX}-XXX-{slug}-data.json` | {패턴} |

레이아웃 패턴은 create-prototype에서 사용하는 패턴명을 사용한다:
- `대시보드` — 상단 카드 + 차트/테이블
- `앱 셸` — 사이드바 + 메인 + 우측패널
- `목록 화면` — 검색바 + 그리드/테이블
- `마스터-디테일` — 검색 필터 + 목록 + 상세 패널
- `폼 화면` — 라벨 + 입력필드 + 제출
- `풀페이지` — 단일 콘텐츠 영역 (탭 전환 가능)
- `진입 화면` — 중앙 정렬 카드 버튼

### 4. 화면 간 흐름 도식화

**반드시** `text` 코드 블록으로 화면 간 네비게이션 흐름을 도식화한다.

````markdown
```text
[메인 nav 탭]
  ① 수집·적재 현황     ② 분석 워크벤치     ③ 분석 결과 뷰어
  ④ 수익사업 분석      ⑤ 공고·입찰 이력

[화면 간 연결]
  수집·적재 현황(1) → (오류 건 클릭) → 공고·입찰 이력(5)
  분석 워크벤치(2) → (결과 보기 클릭) → 분석 결과 뷰어(3)
  공고·입찰 이력(5) → ([분석에 포함] 클릭) → 분석 워크벤치(2)
```
````

포함 사항:
- 메인 nav 탭 구조
- 화면 간 이동 트리거 (어떤 버튼/링크 클릭 시 어디로 이동)
- 데이터 전달 흐름 (파라미터, 필터 자동 적용 등)

### 5. 화면별 상세 기능 명세 (각 화면마다 반복)

각 화면에 대해 아래 내용을 **모두** 작성한다:

#### 5-1. 목적
- 이 화면이 존재하는 이유 1~2줄
- 어떤 요구사항 항목을 이 화면이 담당하는지

#### 5-2. 화면 내 기능 목록 + 이유

해당 화면에 포함되는 기능과 **왜 이 화면에 배치하는지** 이유를 명시한다.

```markdown
| 기능 | 이유 |
|---|---|
| 일별 수집량 차트 | 수집 추이를 한눈에 파악해야 파이프라인 이상 감지 가능 |
| 오류 로그 재처리 버튼 | 실패 건을 즉시 재시도할 수 있어야 운영 효율 향상 |
```

#### 5-3. 화면 내 기능 흐름 도식화

**반드시** `text` 코드 블록으로 화면 내부의 중분류 기능 흐름을 도식화한다.

````markdown
```text
[화면 2 — 분석 워크벤치 기능 흐름]

  ┌─ 조건 프리셋 선택/등록 ─┐
  │                          │
  ▼                          ▼
  조건 편집 폼            분석 주제 선택/등록
  (기간, 지역, 업종 등)    (주제명, 설명)
  │                          │
  └──────── 합류 ────────────┘
               │
               ▼
         [분석 실행] 버튼
               │
               ▼
         Job 생성 → 상태 추적 (대기→처리중→완료)
               │
               ▼
         [결과 보기] → 화면 3으로 이동
```
````

#### 5-4. 레이아웃 구조

`text` 코드 블록으로 화면 영역 배치를 와이어프레임 수준으로 표현한다.

````markdown
```text
[상단 요약 카드 4개]
  • 오늘 수집 건수 / 총 누적 건수 / 수집 실패 건수 / DB 적재 대기 건수

[중간 2열 영역]
  좌: 일별 수집량 시계열 라인 차트
  우: 수집 채널 상태 카드

[하단 탭]
  탭: 수집 로그 | 적재 로그 | 오류 로그
  테이블: 수집 일시 | 데이터 유형 | 건수 | 상태
```
````

### 7. create-prototype 전달 전제조건

이 섹션은 create-prototype 스킬이 이 문서를 받았을 때 따라야 할 구조 규칙을 명시한다.

```text
[산출물 디렉토리 구조]
{PREFIX}-001/
├── display/   ← HTML만
├── data/      ← JSON (원본 보관, fetch 사용 안 함)
├── css/       ← 공통 CSS 1개 + 화면별 CSS 각 1개
├── script/    ← 공통 JS (선택) + 화면별 JS 각 1개 (필수)
└── docs/      ← 이 문서 포함

[파일 짝 규칙 — 화면마다 4개 파일 세트]
- display/{PREFIX}-001-{slug}.html
- css/{PREFIX}-001-{slug}.css
- script/{PREFIX}-001-{slug}.js   ← 반드시 생성, <head>에서 로드
- data/{PREFIX}-001-{slug}-data.json

[데이터 전달 방식]
- JSON 데이터는 <script type="application/json" id="page-data">에 직접 임베드
- fetch() 사용 금지 — file:// 직접 열기 지원
- data/ 폴더 JSON은 원본 보관·diff용으로만 유지

[CSS 규칙]
- <style> 블록 금지
- 공통 CSS: css/{PREFIX}-001.css
- 화면별 CSS: css/{PREFIX}-001-{slug}.css (화면마다 1개 필수)
- 경로: ../css/, ../data/, ../script/ 상대경로만

[JS 로드 순서 규칙]
- 모든 <script src> 선언은 <body>가 아닌 <head>에 위치
- <head> 선언 순서: Tailwind CDN → 공통 CSS → 화면별 CSS → 공통 JS (선택) → 화면별 JS
- 화면별 JS는 반드시 DOMContentLoaded 이벤트 리스너로 DOM 조작 코드를 감싼다
- <body> 내에 로직 <script> 블록 금지 — <script type="application/json" id="page-data">만 허용
- window.{PREFIX} 같은 공통 네임스페이스는 공통 JS에서 정의, 화면별 JS에서 참조
```

### 8. 특이 사항

구현 시 참고할 기술적 특이사항을 나열한다:
- 차트 구현 방식 (더미 SVG/Canvas 등)
- 특수 인터랙션 (타이핑 효과, 프로그레스바 애니메이션 등)

## 파일 저장 규칙

### 식별자 결정

STEP 0-C에서 확인한 식별자를 폴더명에 사용한다.

| 입력 유형 | 폴더명 예시 |
|---|---|
| RFP ID | `RFP-001` |
| 요구사항 ID | `SFR-019` |
| 기능 이름 | `bidding-analysis` (kebab-case) |

### 저장 경로

프로젝트 유형(STEP 0-B)과 사용자(STEP 0-D)에 따라 분기한다:

- **단일 앱**: `.docs/prototype/{사용자}/{식별자}/design-doc.md`
- **복수 앱**: `.docs/prototype/{사용자}/{식별자}/design-doc.md` (앱 구분 없이 프로젝트 공통)

예시:
- 단일앱: `.docs/prototype/hb9397/SFR-019/design-doc.md`
- 복수앱: `.docs/prototype/hb9397/SFR-019/design-doc.md` (앱 구분 없이 프로젝트 공통)

### 저장 시 규칙

- 저장 디렉토리가 없으면 생성한다.
- **새로 작성하기 전에, 해당 prototype 경로 하위에 같은 식별자 폴더가 없는지 반드시 확인**한다. 있으면 신규 생성 대신 갱신 여부를 사용자에게 묻는다.
- 같은 경로에 파일이 이미 있으면 새로 만들지 않고 **해당 파일을 갱신**한다.
- **무조건 파일 상단에 갱신일자를 작성/갱신**한다.
- 다른 사용자/git 계정의 문서를 갱신하는 경우, 파일 상단에 **갱신한 사용자 이름 또는 git 계정을 추가 표기**한다.

## 품질 기준

1. **create-prototype 입력으로 바로 사용 가능** — 이 문서만 보고 프로토타입을 생성할 수 있어야 한다
2. **화면 분리 근거가 명확** — 각 화면이 왜 독립 화면인지 이유가 있다
3. **기능 흐름이 도식화됨** — 화면 간 흐름 + 화면 내 기능 흐름 모두 `text` 코드 블록으로 시각화
4. **기능 배치 이유가 있음** — 각 기능이 왜 이 화면에 있는지 근거 명시
5. **더미 데이터가 사실적** — 한국어, 실제 업무 맥락에 맞는 구체적 예시
6. **누락 없음** — 요구사항의 모든 세부 기능이 어딘가의 화면에 배치됨
7. **파일명이 완전함** — HTML, CSS, JS, JSON 파일명이 화면 목록 테이블에 모두 명시됨
