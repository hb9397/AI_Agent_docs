# SFR-021 목업 디자인 문서

> 이 문서는 `design-prototype-docs` 산출물 예시이자 `create-prototype` 입력 예시로 바로 사용할 수 있는 형태를 목표로 작성한다.
> 실제 구현 코드가 이미 있다는 전제를 두지 않고, 사용자 의도와 업무 흐름을 먼저 해석해 화면을 설계한 문서로 본다.

## 2. 사용자 의도 해석

이 기능의 사용자는 단순히 자료를 많이 모으려는 것이 아니라, 외부 변화 신호를 빠르게 감지하고 내부 의사결정에 바로 연결하려는 목적을 가진다. 따라서 화면은 "수집 설정"만 잘 보여주는 수준으로 끝나면 안 되고, 수집 결과를 해석하고, 반복 브리핑으로 소비하고, 필요 시 자연어 검색으로 깊이 파고드는 흐름까지 하나의 업무 묶음으로 설계해야 한다.

이 요구를 다음 네 가지 사용자 의도로 정리한다.

1. 여러 외부 채널에서 자료를 자동으로 모으되, 어떤 키워드와 어떤 채널 조합으로 수집하는지 운영자가 직접 통제하고 싶다.
2. 수집이 잘 되고 있는지, 최근 어떤 이슈가 부상하는지 대시보드에서 한눈에 파악하고 싶다.
3. 등록한 키워드 중 중요한 주제만 골라 아침 브리핑처럼 반복 전달받고 싶다.
4. 정형 대시보드만으로 부족한 경우, 자연어 질문으로 관련 문서를 탐색하고 AI 요약까지 받고 싶다.

## 3. 요구사항 요약

외부자료 수집 및 분석 기능은 아래의 세부 기능으로 정의한다.

1. 수집 키워드 룰을 등록, 수정, 중지, 재활성화할 수 있어야 한다.
2. 정부기관, 정책포털, 언론사, 포털 뉴스 등 수집 채널을 카테고리 단위와 세부 채널 단위로 조합할 수 있어야 한다.
3. 키워드 조합은 AND, OR, 괄호를 포함한 식으로 표현할 수 있어야 한다.
4. 룰별 수집 상태, 마지막 수집 일시, 매칭 건수, 매칭 문서를 확인할 수 있어야 한다.
5. 수집 결과를 KPI, 목록, 출처별 현황, 키워드 노출, 개체 추이, 단어 연결망으로 요약해야 한다.
6. 등록된 키워드를 브리핑 주제로 재사용하고, 전달 채널과 시간을 설정할 수 있어야 한다.
7. 브리핑 미리보기와 과거 발송 이력을 확인할 수 있어야 한다.
8. 자연어 검색, AI 요약, 주요 근거 문서, 관련 검색 제안까지 포함한 탐색 화면을 제공해야 한다.
9. 전체 기능은 공통 상단 내비게이션으로 연결되어야 하며, 각 화면은 같은 서비스 안의 연속된 경험으로 인지되어야 한다.

## 4. 화면 설계 원칙

### 4-1. 화면 분리 기준

하나의 화면은 하나의 대표 업무 목표를 가져야 한다. SFR-021은 아래 네 가지 독립 업무 목표가 명확하므로 4개 화면으로 나눈다.

| 구분 | 대표 업무 목표 | 분리 이유 |
|---|---|---|
| 수집관리 | 수집 조건을 설계하고 결과를 점검 | 운영 설정과 검수 업무가 밀집되어 있어 입력/수정 UI가 많다. |
| 대시보드 | 수집 결과를 요약 분석 | KPI, 차트, 연결망, 목록 등 분석형 시각화가 중심이다. |
| 브리핑 | 관심 주제를 반복 배포 | 주제 선택, 전달 채널, 시간 설정, 미리보기, 이력이 한 덩어리로 움직인다. |
| 지능형 검색 | 질문 중심 탐색 | 검색창, AI 답변, 근거 문서, 관련 검색이 포털형 구조를 형성한다. |

### 4-2. 공통 설계 방향

- 공공기관 업무 도구의 신뢰감은 유지하되, AI 보조 기능이 전면에 드러나는 현대적 스타일을 사용한다.
- 모든 화면은 상단 고정 탭 내비게이션을 공유한다.
- 카드 기반 레이아웃을 사용하되, 운영 화면은 밀도 높게, 검색 화면은 호흡 넓게 설계한다.
- 더미 데이터는 실제 정책·안전·에너지 분야 문맥에 맞는 한국어 데이터로 채운다.

## 7. 화면별 상세 기능 명세

### 화면 1. 수집관리 & 키워드 관리

#### 7-1. 목적

- 외부자료 수집의 기준이 되는 키워드 룰과 채널 구성을 운영자가 직접 관리하는 핵심 운영 화면이다.
- "무엇을 어떻게 수집할 것인가"와 "실제로 무엇이 수집되었는가"를 한 자리에서 검수하도록 설계한다.

#### 7-2. 화면 내 기능 목록 + 이유

| 기능 | 이유 |
|---|---|
| 수집 키워드 검색 | 등록된 룰이 많아질수록 운영자가 특정 조합을 빠르게 찾아야 한다. |
| 상태 필터와 정렬 | 정상, 지연, 오류, 중지 상태를 기준으로 우선 대응 대상을 분류해야 한다. |
| 키워드 룰 목록 | 룰명, 매칭 건수, 수집 주기, 최근 수집일을 한 번에 비교해야 한다. |
| 신규 키워드 추가 | 운영자가 직접 새로운 이슈를 발견했을 때 즉시 수집 룰을 만들 수 있어야 한다. |
| 키워드 조합식 빌더 | AND, OR, 괄호 조합이 핵심이므로 일반 텍스트 입력보다 조합식 편집 경험이 중요하다. |
| 수집 채널 카테고리/세부 채널 선택 | 정책브리핑, 부처, 언론사, 포털 등 출처 성격별로 수집 범위를 정밀하게 통제해야 한다. |
| 기간 필터와 퀵 필터 | 어제, 일주일, 한달 등 최근 수집 결과를 빠르게 확인할 수 있어야 한다. |
| 매칭 뉴스 목록 | 룰이 실제로 어떤 문서를 잡아냈는지 검수해야 오탐과 미탐을 줄일 수 있다. |
| 우측 인사이트 패널 | 선택한 뉴스의 핵심 요약과 AI 인사이트를 즉시 확인해 판단 시간을 줄인다. |
| 원문 보기 액션 | 프로토타입에서도 실제 업무처럼 후속 확인 행동이 보이도록 해야 한다. |

#### 7-6. 더미 데이터 예시

- 룰명: `전기안전 AND 사고`
- 룰명: `한전 OR 전력거래 OR 전기요금`
- 룰명: `( 태양광 OR ESS ) AND 화재`
- 상태: `정상수집`, `수집지연`, `수집오류`, `수집중지`
- 채널 카테고리: `정책포털`, `부처`, `정부위원회/공공기관`, `뉴스`
- 세부 채널: `대한민국 정책브리핑 RSS`, `산업통상부`, `국토교통부`, `한국경제`, `연합뉴스`
- 매칭 기사: `전기차 충전기 점검 주기 단축 개정 예고`
- 매칭 기사: `ESS 연계 설비 화재 사고 분석 보고서 공개`
- AI 인사이트 문구: `이번 주 언급량 +89% 급상승`

### 화면 2. 외부자료 분석 대시보드

#### 7-7. 목적

- 수집된 데이터를 운영 관점과 분석 관점에서 동시에 읽을 수 있도록 구성한 요약 분석 화면이다.
- 단순 리스트가 아니라 KPI, 추세, 출처 분포, 키워드 노출, 개체 언급, 단어 연결망까지 한 번에 보여줘야 한다.

#### 7-8. 화면 내 기능 목록 + 이유

| 기능 | 이유 |
|---|---|
| 오늘 수집 건수 KPI | 최근 파이프라인 출력량을 즉시 확인하기 위해 필요하다. |
| 총 누적 건수 KPI | 시스템 규모와 데이터 축적 수준을 파악하기 위해 필요하다. |
| 등록한 키워드 건 수 KPI | 운영 범위가 어느 정도인지 보여준다. |
| 데이터 연계 오류 KPI | 운영 대응이 필요한 예외 상황을 눈에 띄게 드러내야 한다. |

## 8. 디자인 가이드

- 서비스 톤: 공공기관 업무 도구의 신뢰감을 유지하되, AI 보조 화면이라는 현대성을 분명하게 드러낸다.
- 메인 컬러: 플랫폼 블루(`#2563EB`)를 기준으로 주요 CTA와 활성 상태를 표현한다.
- 보조 컬러: 그린은 정상, 앰버는 지연/주의, 레드는 오류/긴급, 퍼플은 AI 또는 확장 기능을 나타낸다.
- 상단 nav: 모든 화면에 동일한 구조로 고정하며, 활성 탭은 채도 높은 블루 강조를 사용한다.
- 카드 스타일: 반경 10~14px, 얕은 그림자, 미세한 테두리, 연한 그라데이션 배경을 허용한다.
- 타이포그래피: 본문은 `Noto Sans KR`, 수치/KPI는 `JetBrains Mono`를 사용한다.
- 대시보드 화면은 정보 밀도를 높게, 브리핑과 검색 화면은 여백을 넉넉히 두어 목적 차이를 시각적으로 구분한다.
- 검색 화면은 포털 검색보다 "AI가 먼저 답하는 경험"이 느껴지도록 답변 카드의 비중을 높인다.

## 9. 프로토타입 데이터 구조 제안

각 화면의 JSON은 아래 수준까지 포함하면 `create-prototype`에서 풍부한 인터랙션을 구현하기 쉽다.

| 화면 | 권장 JSON 키 |
|---|---|
| 수집관리 | `baseDateValue`, `govChannelCategories`, `newsCategories`, `rules`, `ruleStatusMeta`, `news`, `insightTemplates` |
| 대시보드 | `kpis`, `newsItems`, `sourceMetrics`, `keywordExposure`, `entityTrend`, `networkNodes`, `networkEdges` |
| 브리핑 | `topicStatusMeta`, `keywordCatalog`, `initialTopicRuleIds`, `deliveryChannels`, `schedule`, `preview`, `history` |
| 지능형 검색 | `defaultQuery`, `results`, `aiSummary`, `evidenceCards`, `priorityIssues`, `relatedQueries` |

데이터 설계 시 유의사항은 아래와 같다.

1. 수집관리와 브리핑은 같은 키워드 룰 ID를 공유해야 한다.
2. 대시보드 차트 값은 실제 수치처럼 보여야 하므로 KPI와 목록 수치가 서로 어긋나지 않아야 한다.
3. 지능형 검색의 AI 요약, 근거 카드, 검색 결과 리스트는 같은 주제를 기준으로 정렬되어야 한다.
4. 모든 날짜는 최근 1개월 이내의 사실적인 더미 날짜로 통일한다.

## 11. 특이 사항

1. 수집관리 화면은 시스템 레벨 설정 느낌을 주기 위해 정보 밀도를 높이고, 우측 인사이트 패널을 항상 열려 있는 상태로 시작한다.
2. 대시보드의 연결망과 차트는 정적 SVG 또는 Canvas 더미 렌더링으로 충분하지만, 시각적으로 실제 분석 도구처럼 보여야 한다.
3. 브리핑 화면은 Google Alerts 패턴을 참고하되, 공공기관 AI 비서 연동이라는 차별점을 칩과 설명 문구로 드러낸다.
4. 지능형 검색 화면은 일반 검색 결과보다 AI 요약이 먼저 읽히도록 상단 카드의 비중을 크게 잡는다.
5. 프로토타입 단계에서는 실제 저장 대신 `alert()` 또는 화면 내 상태 갱신으로 동작을 시뮬레이션한다.
6. 더미 데이터는 전기안전, 전기차 충전, ESS, 정책 브리핑, 전기공사업법, 입찰/조달 문맥을 중심으로 통일한다.
7. 브리핑과 검색 화면은 시각적으로 조금 더 밝고 부드러운 배경 그라데이션을 사용해 운영 화면과 구분한다.
8. 후속 화면 확장이 필요하면 "상세 문서 보기", "규제 비교표", "브리핑 템플릿 관리" 같은 서브 화면을 추가할 수 있지만, 이번 SFR-021 범위에서는 4개 화면으로 닫는다.
