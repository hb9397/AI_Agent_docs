# RFP 요구사항 분석 (rfp-ingest)

---

## STEP 0 — 플랫폼·실행 방식 확인 + 프로젝트 유형 확인

#### STEP 0-A — 플랫폼·실행 방식 확인

`prompts/parallel-setup.md`의 [플랫폼 확인] → [모델 목록 표시] → [실행 방식 선택 — 선호도만 저장] 절차를 따른다.

병렬 선호 시: Task 목록(SFR 목록)은 Step 1에서 확정된다.
SFR 개수가 2개 이상일 때 Task 목록을 제시하고 `prompts/parallel-setup.md`의 [모델 확정] 절차를 실행한다.
SFR이 1개이면 병렬 실행하지 않는다.

순차 선택 시 Step 1로 직접 진행한다.

#### STEP 0-B — 프로젝트 유형 확인 (C-1 게이트)

후속 스킬(design-doc 등)에 전달할 컨텍스트 범위를 확정하기 위해 **반드시** 아래를 수행한다.

1. 현재 수행 위치에서 프로젝트 구조를 탐색한다 (git repo 경계, 하위 앱 폴더 후보 스캔).
2. **단일 애플리케이션 프로젝트**인지 **복수 애플리케이션 프로젝트**인지 판정한다.
3. 판정 결과 + 적용 대상을 사용자에게 **반드시 재확인**한다.

---

RFP 파일에서 지정된 SFR 번호의 기능요구사항을 추출·분석하고,
화면 후보 목록과 요구사항-화면 매핑을 대화 컨텍스트에 출력한다.

> 이 스킬의 해석 결과는 **별도 파일로 저장하지 않는다**.
> 대화 컨텍스트 또는 후속 스킬(design-doc, design-prototype-docs 등)의
> 입력으로만 전달한다.
> 요구사항 해석이 부정확하면 이후 전체 파이프라인이 틀어진다.
> 불명확한 항목은 반드시 이 단계에서 사용자에게 확인한다.

## 스킬 연계

```
@RFP 파일 (PDF)
    ↓
rfp-ingest  ← 지금 여기
    ↓
대화 컨텍스트 (해석 결과)
    ├─→ design-doc         →  설계문서.md
    ├─→ design-prototype-docs → create-prototype  →  HTML 프로토타입
    └─→ (직접 참조)         →  제안서 작성 등
```

---

## 입력 형식

```
/rfp-ingest SFR-019
/rfp-ingest SFR-019,SFR-021
/rfp-ingest SFR-019 SFR-021 SFR-023
```

- RFP 파일은 `@` 태그로 사전 지정되어 있어야 한다.
- RFP 원본 저장 루트: `D:\Dev_Workspace\pre_working\`
- SFR 번호는 쉼표 또는 공백으로 구분 가능.

---

## 워크플로우

### Step 1 — RFP 파일 확인 및 SFR 추출

RFP 파일이 `@`로 지정되지 않은 경우 요청한다.

> "분석할 RFP 파일을 `@` 태그로 지정해 주세요.
> 예: `@D:\Dev_Workspace\pre_working\프로젝트명\제안요청서.pdf`"

RFP 파일에서 요청된 SFR 번호에 해당하는 기능요구사항을 찾아 추출한다.
추출 규칙은 `prompts/sfr-extraction.md` 참조.

**SFR이 2개 이상인 경우**: STEP 0에서 병렬을 선택한 경우에 한해, 아래 Task 목록을 제시하고 `prompts/parallel-setup.md`의 [모델 확정] 절차를 실행한다.

| # | Task | 담당 SFR |
|---|------|----------|
| 1 | subagent-A | SFR-{번호} 분석 |
| … | … | … |

STEP 0에서 순차를 선택한 경우 SFR을 하나씩 순서대로 처리한다.

---

### Step 2 — 기능 단위 해석

각 SFR에 대해 `prompts/sfr-extraction.md`의 **해석 프레임(5관점)**으로 분석한다.
분석 항목과 품질 기준은 해당 prompts 파일 참조.

---

### Step 3 — 화면 후보 매핑

`prompts/screen-mapping.md`의 규칙에 따라 화면 후보를 도출한다.

- SFR의 기능을 수행하기 위해 필요한 화면을 예측한다.
- 화면 후보별: 화면명, 목적, 주요 액션, 우선순위를 정리한다.
- SFR ↔ 화면 매핑 표를 작성한다.
- 화면 간 이동 흐름을 도식화한다.

---

### Step 4 — 불명확 항목 확인

`prompts/clarification.md`의 판단 기준에 따라 불명확 항목을 식별한다.

**인라인으로 사용자에게 확인한다** (별도 스킬 호출 없음).
최대 5개까지 한 번에 묻는다.

> "분석 중 아래 항목이 불명확합니다. 확인 부탁드립니다:
> 1. [항목]: [왜 불명확한지] — 선택지: A / B / 기타
> 2. ..."

확인된 내용은 즉시 Step 2~3 결과에 반영한다.

---

### Step 5 — 해석 결과 출력 및 확인

`templates/rfp-design-input.md` 양식으로 해석 결과 초안을 작성한다. (이 양식은 대화 출력 포맷이며, 파일로 저장하지 않는다.)

초안을 **대화창에 출력**하고 승인을 요청한다. **별도 파일로 저장하지 않는다.**

> "위 분석 결과를 검토해 주세요.
> 화면 후보나 기능 해석 중 수정할 부분이 있으면 말씀해 주세요."

수정 요청 시 해당 섹션만 재작성한다.

---

### Step 6 — 다음 단계 안내

승인 후 사용자에게 다음 단계를 안내한다. 해석 결과는 대화 컨텍스트에 남아 있으므로 후속 스킬이 그대로 참조한다.

> "분석 결과가 확정되었습니다. 다음 단계를 선택해 주세요:
> 1. `/design-doc` — 이 해석 결과를 기반으로 설계 문서 도출
> 2. `/design-prototype-docs` → `/create-prototype` — 추상 화면 목업 먼저 생성
> 3. 추가 SFR 분석 — `/rfp-ingest SFR-XXX`"

## 서브에이전트 전략

STEP 0에서 병렬을 선택하고 SFR이 2개 이상일 때 병렬 분석을 실행한다.
STEP 0에서 확정된 모델을 각 서브에이전트에 적용한다.

```
/rfp-ingest SFR-019, SFR-021, SFR-023
  │
  ├── subagent-A: SFR-019 분석 (입력/처리/출력/예외/연계)
  ├── subagent-B: SFR-021 분석
  └── subagent-C: SFR-023 분석
        │
        ▼ (오케스트레이터 병합)
  대화 컨텍스트 (통합 해석 결과)
```

각 서브에이전트에게 전달하는 컨텍스트:
- RFP 파일 전체 (공통)
- 담당 SFR 번호
- `prompts/sfr-extraction.md` 해석 프레임
- `prompts/screen-mapping.md` 화면 매핑 규칙

오케스트레이터가 병합할 때 처리:
- 중복 화면 후보 통합
- SFR 간 연계 관계 교차 검증
- 화면 간 이동 흐름 통합 도식

---

## 운영 규칙

1. **RFP 전체를 한 번에 처리하지 않는다.** 필요한 SFR만 선택해서 실행한다.
2. **불명확 항목은 이 단계에서 해결한다.** 다음 스킬로 넘기지 않는다.
3. **화면 후보는 예측이다.** design-doc 단계에서 확정된다. 이 단계에서는 "후보"로 표기한다.
4. **같은 SFR을 다시 실행하면 이전 해석 결과를 대체한다.** 대화 컨텍스트의 기존 분석을 새 분석으로 갱신한다.
