# SFR 추출 및 해석 규칙 (sfr-extraction)

---

## SFR 식별 규칙

RFP 문서에서 SFR(기능요구사항)을 찾는 패턴:

```
일련번호 패턴:
  SFR-XXX        (표준)
  SFR_XXX        (언더스코어 변형)
  기능-XXX       (한글 접두사)
  FR-XXX         (Functional Requirement)
  REQ-XXX        (Requirement)

위치:
  - "기능요구사항" 표 / 테이블
  - "시스템 요구사항" 섹션
  - "요구사항 명세" 부록
  - 별첨 문서
```

찾지 못한 경우:
> "RFP에서 '{SFR번호}'를 찾지 못했습니다.
> 유사한 항목이 있으면 알려주세요. RFP 구조가 다를 수 있습니다."

## 기능 해석 프레임 (5관점)

각 SFR에 대해 아래 5개 관점으로 분석한다.

### 1. 입력 (Input)
- 이 기능이 받는 데이터는 무엇인가?
- 사용자 입력인가, 시스템 자동 수집인가?
- 입력 데이터의 형식과 제약 조건은?
- 다른 SFR이나 외부 시스템에서 오는 데이터가 있는가?

### 2. 처리 (Process)
- 핵심 비즈니스 로직은 무엇인가?
- 계산, 변환, 집계, 판정 로직이 있는가?
- 분기 처리(조건별 다른 동작)가 있는가?
- 배치 처리인가, 실시간 처리인가?
- AI/ML 모델 연동이 필요한가?

### 3. 출력 (Output)
- 처리 결과를 어디에 표시하는가? (화면, 리포트, 알림)
- 데이터를 어디에 저장하는가? (DB, 파일, 외부 시스템)
- 출력 형식(표, 차트, 문서)이 명시되어 있는가?
- 다른 SFR에 전달하는 결과가 있는가?

### 4. 예외 (Exception)
- 입력이 잘못된 경우 처리는?
- 처리 실패 시 복구 전략은?
- 권한 부족 시 동작은?
- 동시성 충돌 시 처리는?
- 대용량 데이터 한계는?

### 5. 연계 (Integration)
- 선행 SFR이 있는가? (이 기능 실행 전 완료되어야 하는 것)
- 후행 SFR이 있는가? (이 기능 결과를 사용하는 것)
- 외부 시스템 연동이 있는가? (API, 파일 교환, DB 공유)
- 공통 코드/데이터가 있는가? (코드 테이블, 공통 모듈)

## 해석 품질 기준

```
✅ 좋은 해석
  "SFR-019: 전기설비 점검 결과를 AI가 분석하여 위험도를 3단계(위험/주의/양호)로
   판정하고, 판정 결과를 점검 이력 DB에 저장한다.
   입력: 점검 데이터(센서값, 측정일시, 설비코드)
   처리: AI 모델(분류기)로 위험도 판정, 판정 근거 텍스트 생성
   출력: 판정 결과 화면 표시 + DB 저장
   예외: 센서값 누락 시 '데이터 부족' 상태로 보류 처리
   연계: SFR-018(점검 데이터 수집) 완료 후 실행"

❌ 나쁜 해석
  "SFR-019: AI 분석 기능을 구현한다."
  → 입력/처리/출력/예외/연계가 모두 빠져 있음
```

## 불명확 판단 기준

아래 항목이 RFP 원문에서 명확하지 않으면 불명확으로 분류한다:

🔴 반드시 확인:
1. 처리 로직의 구체적 알고리즘이 명시되지 않은 경우
2. "등", "기타" 로 끝나는 열거형 요구사항
3. 비기능 요구사항(성능, 보안)이 기능에 영향을 미치는 경우

🟡 선택 확인:
4. 화면 레이아웃이 언급만 되고 상세가 없는 경우
5. 외부 시스템 연계 프로토콜이 미정인 경우
6. 데이터 보존 기간, 용량 기준이 미정인 경우
