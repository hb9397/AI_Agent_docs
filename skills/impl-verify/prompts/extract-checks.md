# 검증 항목 추출 규칙 (extract-checks)

작업지침서에서 PASS/FAIL 판정이 필요한 항목을 모두 뽑는다.

---

## 추출 대상

### 1. 태스크 검증 기준

각 태스크 블록에서 다음 줄을 추출:

```
**검증 기준** : {내용}
```

내부 구조:
- `task_id`: 태스크 ID (예: BE-03, UI-01, CORE-02)
- `source_path`: 작업지침서의 줄 번호
- `text`: 검증 기준 본문 그대로
- `task_files`: 같은 블록의 `(파일 경로)` (검증 시 참고)

### 2. Phase 통합 검증

`### Phase N 통합 검증` 또는 `### ✅ Phase N 통합 검증` 블록 전체:

```
**시나리오** : {한 줄 요약}

1. {단계 1}
2. {단계 2}
3. {단계 3}

**합격 조건** : {조건 A} ✅ / {조건 B} ✅ / {조건 C} ✅
```

내부 구조:
- `phase_no`: Phase 번호
- `scenario`: 시나리오 한 줄
- `steps`: 단계별 액션 배열
- `pass_conditions`: 합격 조건 배열 (개별 판정 단위)

### 3. 화면 중심 체크리스트 (impl-fe-be-doc)

`#### 기능 검증` / `#### 상태 검증` / `#### 에지 케이스` 섹션의 `- [ ]` 항목 전부:

```
#### 기능 검증
- [ ] {데이터 조회}: 화면 진입 시 목록/상세 데이터가 API로부터 로드된다
- [ ] {등록}: 사용자 액션 후 성공 메시지와 화면 반영을 확인한다

#### 상태 검증
- [ ] {로딩}: API 호출 중 로딩 인디케이터 표시
...
```

내부 구조:
- `category`: 기능 / 상태 / 에지
- `tag`: `{}` 안의 라벨 (있으면)
- `text`: 본문 그대로
- `sfr_req`: `[SFR-XXX]` / `[REQ-XXX]` 태깅 (있으면)

### 4. 전역 주의사항 중 검증 가능 항목

`## 전역 주의사항` 섹션에서 다음 패턴이 있으면 검증 항목으로 등록:
- "X 시 Y 해야 한다"
- "반드시 X를 포함"
- "X 금지"

판정 가능한 표현만 추출 (모호한 권고는 제외).

---

## 추출 결과 표준 구조

다음 yaml 형태로 통합:

```yaml
checks:
  - id: BE-03.criterion
    kind: task
    task_id: BE-03
    source: "작업지침서 L142"
    text: "curl happy/validation/권한 3케이스 통과"
    related_files: ["api/routers/sites.py"]

  - id: phase-2.integration.A
    kind: phase-condition
    phase: 2
    scenario: "폼 제출 → API 저장 → 성공/실패 표시"
    step_order: 1
    text: "필수 입력 후 제출 시 201 응답"

  - id: phase-2.screen.fn.1
    kind: screen-checklist
    phase: 2
    category: 기능
    tag: 등록
    text: "사용자 액션 후 성공 메시지와 화면 반영"
    sfr_req: "SFR-005"
```

---

## 누락 처리

- `**검증 기준**` 없는 태스크: WARN으로 표시. 리포트에서 "검증 기준 누락" 섹션에 모음.
- Phase 통합 검증 누락: ERROR. 진행하기 어렵다고 사용자에게 알린다.
- 화면 중심 체크리스트가 비어 있음: 일반 통합 검증만 사용.

---

## 우선순위

여러 항목이 있을 때 검증 순서:

1. **빌드/타입체크 류 (실패 시 다른 검증 무의미)**
2. **태스크 검증 기준** (단위 테스트, lint)
3. **Phase 통합 검증의 합격 조건** (실제 시나리오 실행)
4. **화면 중심 체크리스트**
5. **전역 주의사항 검증**

빌드/타입 실패 시 이후 검증은 SKIP으로 표시하고 사용자에게 알린다.
