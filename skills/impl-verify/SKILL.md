# 구현 검증 스킬 (impl-verify)

작업지침서에 박혀 있는 **태스크별 검증 기준**과 **Phase 통합 검증 시나리오**를
실제로 실행하거나 사용자 확인을 받아 PASS/FAIL 판정을 내린다.

검증 리포트만 산출한다. 코드·작업지침서·구현 결과를 수정하지 않는다.

---

## 핵심 원칙

1. **검증 결과만 보고**. 자동 수정·재구현·롤백을 수행하지 않는다.
2. **자동/수동 분리**. CLI·테스트·curl처럼 명령으로 결과를 얻을 수 있는 것은 직접 실행,
   시각·UX·a11y처럼 사람 판단이 필요한 것은 체크리스트로 제시.
3. **출처 명시**. 모든 검증 항목은 작업지침서의 어느 줄에서 왔는지 표기.
4. **게이트 권고만**. Phase 통합 검증 실패 시 다음 Phase 진입 차단을 권고하되,
   강제하지 않는다. 사용자가 결정한다.
5. **부수효과 최소화**. 검증 명령이 운영 DB/외부 시스템에 쓰기 작업을 하지 않도록 조심한다.
   `references/safe-execution.md`의 게이트를 거친다.

---

## 스킬 연계

```
impl-doc / impl-fe-be-doc → 작업지침서
                                  ↓
                        impl-reuse-scan (preflight, 선택)
                                  ↓
                              실제 구현
                                  ↓
                          impl-verify   ← 지금 여기
                                  ↓
                  태스크 검증 + Phase 통합 검증
                                  ↓
                   PASS → 다음 Phase / FAIL → 재작업 권고
```

---

## 검증 단위

이 스킬은 3가지 단위로 호출 가능하다:

| 단위 | 범위 | 호출 예시 |
|------|------|----------|
| **task** | 단일 태스크 1건 | "BE-03 검증 돌려줘" |
| **phase** | Phase 하나의 통합 검증만 | "Phase 2 통합 검증 돌려줘" |
| **full** | Phase 내 모든 태스크 + 통합 검증 | "Phase 2 전체 검증" |

기본은 full. 사용자가 명시하면 task/phase로 좁힌다.

---

## 워크플로우

### Step 0 — 플랫폼·실행 방식 확인 + 프로젝트 유형 확인

#### Step 0-A — 플랫폼·실행 방식 확인

사용자에게 아래를 확인한다:

> 1. 서브에이전트(병렬 처리)를 사용할 수 있는 환경인가요? (Claude Code / Codex / 기타)
> 2. 사용할 경우 병렬 실행을 원하시나요?

서브에이전트 미지원 또는 미사용 선택 시 순차 실행한다.

#### Step 0-B — 프로젝트 유형 확인 (C-1 게이트)

검증 범위를 확정하기 위해 **반드시** 아래를 수행한다.

1. 현재 수행 위치에서 프로젝트 구조를 탐색한다 (git repo 경계, 하위 앱 폴더 후보 스캔).
2. **단일 애플리케이션 프로젝트**인지 **복수 애플리케이션 프로젝트**인지 판정한다.
3. 판정 결과 + 적용 대상 애플리케이션(폴더)을 사용자에게 **반드시 재확인**한다.
4. 확인된 범위 밖은 건드리지 않는다.

> ✋ **확인 게이트**
>
> - 프로젝트 유형: **단일 / 복수** 애플리케이션
> - 검증 대상 애플리케이션(폴더): `{폴더명}`
> - 작업지침서 참조 경로:
>   - (단일앱) `.docs/impl-doc/{사용자}/{기능}.md`
>   - (복수앱) `.docs/{앱}/impl-doc/{사용자}/{기능}.md`
>
> 맞습니까? **(승인 / 수정 / 취소)**

---

### Step 1 — 입력 수집

다음 3가지가 필요하다. 빠진 게 있으면 요청한다.

> "검증할 대상을 알려주세요.
> 1. 작업지침서 파일 경로 (impl-doc / impl-fe-be-doc 산출물)
>    - (단일앱) `.docs/impl-doc/{사용자}/{기능}.md`
>    - (복수앱) `.docs/{앱}/impl-doc/{사용자}/{기능}.md`
> 2. 검증 단위: 태스크 ID 또는 Phase 번호 또는 'full'
> 3. 구현 산출물 위치 (코드베이스 루트, 실행 명령 기준 경로)"

추가 선택 입력:
- 실행 환경 (로컬 / Docker / 스테이지 서버 URL 등)
- 자격증명 필요 여부 (DB 접근, API 토큰)
- 자동 실행 권한 (Bash 허용 범위)

---

### Step 2 — 검증 항목 추출

`prompts/extract-checks.md` 기준으로 작업지침서에서 검증 항목을 뽑는다.

추출 대상:
- 각 태스크 블록의 `**검증 기준** :` 줄
- 각 Phase의 `### Phase N 통합 검증` 블록 (시나리오 단계 + 합격 조건)
- 화면 중심 Phase의 기능/상태/에지 체크리스트
- 전역 주의사항 중 검증 가능한 항목

내부 표현(사용자에게 보이지 않아도 됨):

```yaml
checks:
  - id: BE-03.criterion
    source: "task BE-03 검증 기준"
    text: "curl happy/validation/권한 3케이스 통과"
    type: auto
    commands:
      - "curl -X POST http://localhost:8000/api/sites -d '{유효 페이로드}' → 2xx"
      - "curl -X POST ... -d '{필수 누락}' → 4xx"
      - "curl -X POST ... (토큰 없이) → 401"

  - id: phase-2.integration
    source: "Phase 2 통합 검증"
    text: "폼 제출 → API 저장 → 성공/실패 표시"
    type: manual
    checklist:
      - "화면 진입 시 폼 렌더링"
      - "유효 입력 후 제출 → 성공 토스트"
      - "필수 누락 후 제출 → 필드별 에러 표시"
```

---

### Step 3 — 자동/수동 분류

`prompts/classify-checks.md` 기준으로 분류한다.

**auto (자동 실행 가능)**
- CLI 명령 / pytest / npm test / lint / typecheck
- curl / HTTP 요청
- DB SELECT 검증
- 파일 존재·내용 대조
- 빌드 스크립트 실행

**manual (사람 확인 필요)**
- 화면 렌더링 / 레이아웃 / 반응형
- 인터랙션 (클릭·드래그·키보드)
- UX 흐름 / 토스트·모달 노출
- 접근성 (스크린리더, 색 대비)
- 시각적 출력 품질

**hybrid (자동+수동 혼합)**
- 자동으로 환경 준비 후 수동 확인 (서버 기동 후 브라우저 확인)
- 자동 결과를 보고 사람이 판정 (스냅샷·차트 출력)

---

### Step 4 — 자동 검증 실행

`prompts/run-auto.md` 기준으로 분류된 auto 항목을 순차 실행한다.

실행 전 안전 게이트(`references/safe-execution.md`):
- 운영 DB/외부 결제 등 위험 명령은 사용자 승인 받는다.
- 멱등성이 보장되지 않는 쓰기 명령은 사용자 승인 받는다.
- 비밀번호·토큰은 환경변수로만 받는다. 명령행 로깅 금지.

각 명령:
1. 실행 전 명령을 사용자에게 보여준다(긴 배치는 묶어서).
2. 실행 결과(stdout/stderr/exit code)를 캡처.
3. 합격 조건과 대조 → PASS/FAIL/UNKNOWN 판정.
4. UNKNOWN은 manual로 재분류한다.

---

### Step 5 — 수동 검증 안내

`prompts/run-manual.md`의 체크리스트 형식으로 사용자에게 출력한다.

```
## 수동 검증 — Phase 2

다음 항목을 직접 확인하고 결과를 알려주세요.

[화면 진입]
☐ /sites 경로 진입 시 등록 폼 렌더링
☐ 로딩 인디케이터 노출

[성공 흐름]
☐ 유효 입력 → 제출 → 성공 토스트
☐ 목록 화면으로 자동 이동

[에러 흐름]
☐ 필수 누락 → 필드별 빨간 메시지
☐ 서버 500 → 재시도 안내 + 입력 보존

답변 형식 예시:
  "전부 PASS"
  "1,2 PASS / 3 FAIL (필드별 메시지 없이 토스트만 떠요) / 4 PASS / ..."
```

응답이 없으면 manual은 SKIP 처리하고 리포트에 표시한다.

---

### Step 6 — 매트릭스·진단·게이트 산출

`templates/verification-report.md` 양식으로 리포트를 산출한다.

리포트 구성:
1. **요약**: 전체 PASS/FAIL/SKIP 카운트, Phase 게이트 판정
2. **태스크별 PASS/FAIL 매트릭스**
3. **FAIL 항목 진단**: 추정 원인 + 권장 액션 (재구현 / 작업지침서 보강 / 환경 문제)
4. **게이트 권고**: 다음 Phase 진입 가능/불가 + 사유
5. **부록**: 실행한 명령 + 출력 캡처 (긴 것은 파일로)

작업지침서 파일은 수정하지 않는다.
재작업이 필요하면 사용자가 직접 impl-doc/impl-fe-be-doc로 돌아간다.

---

## 게이트 판정 규칙

`references/gate-rules.md` 참조.

기본 규칙:
- 태스크 검증 1건이라도 FAIL → 해당 태스크 미완료
- Phase 통합 검증의 합격 조건 1건이라도 FAIL → Phase 미통과
- Phase 미통과 시 **다음 Phase 진입 금지 권고**
- SKIP 항목이 있으면 게이트는 "조건부 통과" — 사용자가 SKIP 사유 확인 후 결정

## 입력 분기

| 작업지침서 유형 | 검증 출처 |
|---------------|----------|
| 작업지침서 (머리말 `생성 스킬: impl-doc`) | 태스크 `**검증 기준**` + `### Phase N 통합 검증` |
| 작업지침서 (머리말 `생성 스킬: impl-fe-be-doc`) | 위 + 화면 중심 체크리스트 (기능/상태/에지) |
| 그 외 (수기 작업지침서) | 패턴 매칭 — `검증 기준`/`통합 검증`/`합격 조건` 키워드 |

패턴 매칭 실패 시 사용자에게 검증 항목을 직접 입력받는다.

---

## 한계와 주의

1. **자동 판정의 오탐** — 명령 exit code만으로 판단하면 빈 응답·잘못된 응답 형식을 놓칠 수 있음.
   합격 조건을 응답 본문 패턴까지 명시해야 정확.
2. **외부 시스템 의존** — 결제·메일·이벤트는 dry-run 또는 mock 환경에서만 실행 권장.
3. **재현성** — 한 번 PASS 한 검증이 다음 회차에 FAIL 날 수 있음(데이터 의존). 리포트에 실행 일시·환경 기록.
4. **수동 검증의 누락** — 사용자가 답하지 않으면 SKIP. 게이트 판정에서 "조건부 통과"로 표시.
5. **권한·자격증명** — DB/API 자격증명은 환경변수로만 받는다. 로그 캡처 시 마스킹.
