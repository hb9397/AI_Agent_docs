# 공통 자산 발견·보고 스킬 (impl-reuse-scan)

작업지침서나 태스크 설명을 입력받아, 해당 작업이 만들 예정인 공통 자산이
이미 코드베이스에 존재하는지 스캔하고 **발견 리포트만** 산출한다.

자동 반영은 하지 않는다. 사용자가 항목별로 결정한 뒤에만 Agent 지시문 패치를 만든다.

---

## 핵심 원칙

이 스킬은 **발견(Discover)과 보고(Report)** 전용이다.

1. 발견 결과를 작업지침서 파일에 자동 반영하지 않는다.
2. Agent 지시문에 자동 주입하지 않는다.
3. 사용자가 항목별로 "재사용 / 확장 / 신규 / 보류"를 결정한 뒤,
   사용자의 명시적 요청이 있을 때만 Agent 지시문 패치를 산출한다.
4. 패치 산출 후에도 작업지침서 파일을 직접 수정하지 않는다.
   적용은 사용자가 직접 수행한다.
5. 리포트만 받고 패치 없이 종료해도 정상 흐름이다.

---

## 스킬 연계

```
design-doc → impl-doc / impl-fe-be-doc → 작업지침서
                                              ↓
                                       impl-reuse-scan  ← 지금 여기
                                       (Phase/태스크 시작 직전)
                                              ↓
                                      발견 리포트 + 결정 인터뷰
                                              ↓
                                  (사용자 승인 시) Agent 지시문 패치
                                              ↓
                                          실제 구현
                                              ↓
                                       (향후) 검증 스킬
```

호출 시점은 **Phase 또는 태스크 단위**다.
한 Phase 전체를 한 번에 스캔할 수도 있고, 태스크 1개만 좁게 스캔할 수도 있다.

---

## 워크플로우

### Step 1 — 입력 수집 및 의도 추출

입력이 충분치 않으면 요청한다.

> "스캔할 대상을 알려주세요.
> - 작업지침서 파일 경로 + 대상 Phase 번호 또는 태스크 ID, 또는
> - 임의 태스크 설명 (무엇을 만들 예정인지)
> 그리고 스캔할 코드베이스 루트 경로도 알려주세요."

`prompts/extract-intents.md` 기준으로 **생성 예정 자산 후보**를 추출한다:

- API 라우트, 핸들러
- DTO / 요청·응답 스키마 / 타입 정의
- Entity / DB 모델
- FE 컴포넌트, 페이지/라우트
- 훅, 상태 스토어
- 유틸 함수, 상수, enum, i18n 키
- 디자인 토큰 / 공통 스타일

---

### Step 2 — 카테고리별 코드베이스 스캔

`prompts/scan-patterns.md`의 카테고리별 grep/glob 패턴을 따른다.

스캔 범위 제어:
- 기본은 코드베이스 루트 전체.
- 작업지침서의 "파일 경로" 필드로 디렉토리가 좁혀지면 해당 디렉토리부터 시작하되
  공통 자산이 있을 만한 `common/`, `shared/`, `utils/`, `dto/`, `models/`, `components/`,
  `hooks/` 등 디렉토리를 추가로 스캔한다.

성능을 위해 **카테고리당 grep 1~3회**로 제한한다. 결과 30건 초과 시 상위 후보만 표시한다.

---

### Step 3 — 매칭 강도 분류

`prompts/match-rules.md`의 기준으로 분류한다:

- 🔴 **동일**: 이름·시그니처·책임이 거의 동일 → 재사용 강력 권장
- 🟡 **유사**: 도메인/책임이 겹치지만 시그니처 일부 다름 → 확장/리팩터 후보
- 🟢 **참고**: 이름은 비슷하지만 책임 다름 → 사용자 참고용

`references/common-pitfalls.md`로 false positive를 차단한다.
("이름만 같고 도메인이 완전히 다른 경우는 재사용 금지" 등)

---

### Step 4 — 발견 리포트 산출

`templates/discovery-report.md` 양식으로 출력한다.

각 자산 항목에 반드시 포함할 정보:
- 생성 예정 자산 이름
- 기존 자산의 **파일 경로 + 줄 번호**
- 시그니처 / 핵심 인터페이스
- 매칭 근거
- 현재 사용처 (import / reference 위치)
- (유사인 경우) 확장 방법 예시

리포트 끝에 **결정 요청 표**를 둔다:

```
| # | 자산 | 발견 위치 | 결정 |
|---|------|----------|------|
| 1 | ... | ... | ☐ 재사용 / ☐ 확장 / ☐ 신규 / ☐ 보류 |
```

리포트만 출력하고 끝낼 수도 있다는 사실을 사용자에게 알린다.

---

### Step 5 — 사용자 결정 인터뷰

`prompts/decision-interview.md` 기준으로 사용자 응답을 받는다.

응답 형식 예시:
- "1·2는 재사용, 3은 확장"
- "전부 신규로"
- "리포트만 받을게요"

답변하지 않은 항목은 **보류** 상태로 둔다. 강요하지 않는다.

---

### Step 6 — Agent 지시문 패치 산출 (사용자 승인 후에만)

사용자가 "지시문 패치 만들어줘" 라고 명시했을 때만 실행한다.

`templates/instruction-patch.md` 양식으로 출력한다:
- 재사용 의무 항목 (import 경로 + "신규 정의 금지")
- 확장 의무 항목 (대상 파일 + 추가할 인터페이스 + 기존 사용처 영향 점검)
- 신규 생성 항목 (의도적으로 신규 결정한 사유 — 사용자 입력 있으면 포함)

패치를 작업지침서 파일에 직접 쓰지 않는다.
대화창에 출력하고, 사용자가 복사·붙여넣기로 적용한다.

---

## 입력별 분기

| 입력 형태 | 처리 방식 |
|----------|----------|
| `impl-general-*.md` + Phase 번호 | 해당 Phase의 모든 태스크 일괄 스캔 |
| `impl-workflow-*.md` + 태스크 ID | 해당 태스크 1건만 스캔 |
| 자유 텍스트 ("사이트 등록 API 만들 건데") | 자유 텍스트에서 의도 추출 후 스캔 |

---

## 출력 모드

| 모드 | 산출물 | 사용 시점 |
|------|--------|----------|
| **report-only** (기본) | 발견 리포트 | 사용자 직접 검토하고 끝 |
| **patch** (승인 시) | 발견 리포트 + Agent 지시문 패치 | 결정을 Agent에게 그대로 전달하고 싶을 때 |

기본은 report-only다. 사용자가 명시적으로 패치를 요청했을 때만 patch 모드로 전환한다.

---

## 카테고리 범위 (MVP)

MVP에서는 다음 3개 묶음만 다룬다. 운영 후 확장.

1. **BE**: API 라우트, DTO/스키마, Entity/모델
2. **FE**: 컴포넌트, 훅
3. **공통**: util 함수, constant/enum

이후 확장 후보: 디자인 토큰, i18n 키, 마이그레이션 충돌, 환경변수.

---

## 한계와 주의

1. **false positive 가능** — 이름이 같지만 도메인이 다른 경우. 사용자 결정 게이트로 완화.
2. **거대 코드베이스 성능** — 전체 grep은 비쌈. 작업지침서 디렉토리 기준으로 좁힌다.
3. **자산 책임이 주석에 없으면 매칭 약함** — 시그니처와 사용처로 보완 추론.
4. **다국어/네이밍 규칙 차이** — `getUser`/`fetch_user`/`load_user` 같은 변형을 놓칠 수 있음.
   `scan-patterns.md`에 정의된 규칙으로만 커버한다.
