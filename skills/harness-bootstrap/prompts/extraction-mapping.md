# 코드 인벤토리 → OUTPUT_V2 섹션 매핑 (extraction-mapping)

Step 2 스캔 결과와 Step 3 인터뷰 답변을 `design-doc`의 `OUTPUT_V2.md` 섹션에
어떻게 채울지 정의한다.

> 원칙: **관찰 기반만 서술**. 설계 의도·의사결정 근거를 추측하지 않는다.
> 코드에서 추출 불가한 부분은 `미정 — [이유]` 로 남긴다.

---

## 섹션별 매핑

### 01 개요 🔵

| 추출 요소 | 소스 |
|----------|------|
| 프로젝트명 | 매니페스트의 name 필드 |
| 한 줄 요약 | 인터뷰 질문 1 답변 (도메인·목적) |
| 스케일 | 디렉토리 규모·의존성 수 기반 자동 판정 (프로젝트 / 기능 / 컴포넌트) |
| 핵심 기술 스택 태그 | stack-detection 결과 상위 3~5개 |

도메인 설명이 누락되면: `미정 — 사용자 미제공`

---

### 02 동작 흐름 🔵

엔트리포인트 → 라우터 → 서비스/도메인 로직 → 데이터 접근 순으로
**관찰된 호출 체인**을 역추적한다.

- 엔트리포인트 파일을 읽고 초기화 순서 파악 (DB 연결·라우터 등록·미들웨어)
- 주요 라우터 1~3개를 샘플로 뽑아 STEP 1~N으로 전개
- 각 STEP에 관찰된 함수/모듈명 표기
- 추적 깊이는 **최대 3 hop**으로 제한:
  - 엔트리포인트 → 라우터/핸들러
  - 라우터/핸들러 → 서비스/도메인
  - 서비스/도메인 → 저장소/DB/외부 호출
- 그 이후 하위 호출이 더 이어지면 `하위 호출 다수 — 별도 모듈 참조`로 요약한다.
- 동적 디스패치, 리플렉션, 큐 소비자 체인처럼 정적 추적이 불안정하면 추적을 멈추고 `미정 — 정적 역추적 한계`로 적는다.

추론이 필요한 분기(예: 성공/실패 케이스)는 **코드에 실제로 구현된 분기만** 기록.
상상한 분기는 쓰지 않는다.

---

### 03 집중 로직 🔵

엔트리포인트 주변에서 가장 큰 모듈(파일 수·LOC 기준)을 1~2개 선정해
"이 모듈이 무엇을 담당하는가"를 **파일명·함수 시그니처 기반**으로 요약.

- 핵심 결정 사항·설계 의도는 쓰지 않는다 (알 수 없음)
- "현재 이 모듈은 X, Y, Z 책임을 가진다"로만 서술

---

### 04 인터페이스 🔵

- **REST API**: Step 2에서 추출한 엔드포인트 목록을 표로 정리
  - 메서드 / 경로 / 파일:라인 / 요청·응답 스키마 (있으면)
- **WebSocket**: 채널 URL + 메시지 타입 후보 목록
- **외부 API 호출**: HTTP 클라이언트 사용 지점 grep 결과
  - 호출 대상 URL이 환경 변수면 변수명만 기록

스키마는 Pydantic/Zod/DTO 클래스 정의에서 직접 추출. 없으면 `미정 — 스키마 미정의`.

---

### 05 데이터 🟢

ORM 모델·마이그레이션·스키마 파일에서 추출:

- 테이블/엔티티명
- 핵심 컬럼 (타입, 제약)
- 관계 (FK, 카디널리티)
- 인덱스 (있으면)

JSONB/임베디드 문서 필드는 구조를 덤프하지 말고 **필드 존재만** 기록.

---

### 06 파일 구성 🔵

Step 2의 디렉토리 트리를 그대로 사용. 각 폴더에 **한 줄 역할 주석** 추가.
역할은 **폴더명·포함 파일명**에서만 추론. 확신이 없으면 `역할 미상`.

---

### 07 라이브러리 🟢

stack-detection의 카테고리 분류 결과를 그대로 사용.
버전은 매니페스트 기준. 범위 지정(`^1.2.0` 등)은 그대로 옮긴다.

---

### 08 화면 / UI 🟠

- 라우팅 설정 파일에서 페이지 목록 추출 (React Router, Next.js app/, Vue Router 등)
- 각 페이지의 컴포넌트명·경로만 기록
- 와이어프레임·UX 흐름은 **쓰지 않는다** (코드에서 알 수 없음)

---

### 09 상태 관리 🟠

- 글로벌 vs 로컬 상태 분리 패턴이 코드에 보이면 기록
