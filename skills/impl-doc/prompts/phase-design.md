# 범용 Phase 분할 기준 (phase-design)

---

## 프로젝트 유형별 Phase 패턴

### CLI 도구

```
Phase 0: 프로젝트 구조 + 의존성           ← 스케일이 클 때만
Phase 1: CLI 파싱 + 최소 기능 1개 동작
Phase N: 핵심 기능 (서브커맨드별 또는 기능별)
마지막:  에러 처리 + 헬프 메시지 + 패키징
```

### 자동화 스크립트

```
Phase 1: 데이터 소스 접근 + 최소 파이프라인
Phase N: 변환/처리 로직 확장
마지막:  스케줄링 + 로깅 + 알림
```

### 라이브러리 / SDK

```
Phase 0: 프로젝트 구조 + 빌드 설정
Phase 1: 코어 API (public interface) 최소 1개
Phase N: API 확장 + 유틸리티
마지막:  테스트 + 문서화 + NPM/PyPI 패키징
```

### 단독 백엔드

```
Phase 0: 프로젝트 구조 + DB 연결
Phase 1: 가장 단순한 엔드포인트 1개 + 헬스체크
Phase N: 핵심 비즈니스 로직 API
마지막:  인증 + 배포 + 모니터링
```

### BE 단일 기능 (엔드포인트 1~수개 / 단일 도메인 로직)

기존 BE 코드베이스에 기능 1건을 추가/수정하는 경우.

```
Phase 0: (생략 가능) 의존성·환경 확인
Phase 1: DB 스키마 / 모델 / 마이그레이션
  → DB-XX: 모델, 인덱스, 마이그레이션 스크립트
  → 검증: 마이그레이션 적용 + 롤백 가능
Phase 2: 도메인 로직 + 엔드포인트
  → CORE-XX: 비즈니스 규칙, 검증 로직
  → API-XX: 라우터, DTO, 권한 가드
  → 검증: curl/Postman으로 happy/error 응답 확인
Phase 3: 테스트 + 문서
  → TEST-XX: 단위 + 통합 테스트
  → 검증: 테스트 통과 + OpenAPI/Swagger 갱신
```

마이그레이션이 없거나 도메인 로직이 단순하면 Phase 1·2를 1개로 합친다.

### FE 단일 기능 (컴포넌트/훅/화면 1개 신규·수정)

기존 FE 코드베이스에 컴포넌트, 훅, 화면 1개를 추가/수정하는 경우.

```
Phase 0: (생략 가능) 디자인 토큰·라우팅 확인
Phase 1: UI 골격 + 상태 설계
  → UI-XX: 컴포넌트 트리, 마크업, 스타일
  → STATE-XX: 로컬 상태, 폼 스키마, 훅
  → 검증: 정적 렌더링 + 스토리북 또는 라우트 진입
Phase 2: 데이터 연동 + 인터랙션
  → STATE-XX: 서버 상태(react-query/swr/redux), 캐시 키
  → UI-XX: 이벤트 핸들러, 라우팅, 네비게이션
  → 검증: 로딩/에러/빈 상태/성공 4가지 시나리오
Phase 3: 테스트 + 접근성
  → TEST-XX: 컴포넌트 테스트, 인터랙션 테스트
  → 검증: a11y(레이블·키보드·콘트라스트) + 반응형
```

순수 UI 컴포넌트(데이터 연동 없음)면 Phase 2를 생략한다.
훅 단독 추가면 UI-XX 없이 STATE-XX + TEST-XX만 사용한다.

### 데이터 파이프라인

```
Phase 1: 데이터 소스 연결 + 최소 수집
Phase 2: 변환 로직 (핵심 ETL)
Phase 3: 적재 (타겟 시스템 연동)
마지막:  스케줄링 + 모니터링 + 실패 복구
```

### AI Agent / LLM 도구

```
Phase 1: LLM 클라이언트 연결 + 기본 프롬프트
Phase 2: 체인/에이전트 구성 (핵심 로직)
Phase N: 도구(Tool) 연동 + 메모리/컨텍스트 관리
마지막:  프롬프트 최적화 + 평가 + 배포
```

---

## 공통 분할 원칙

```
✅ 지켜야 할 것
  - 각 Phase는 이전 Phase 없이 실행 불가능해야 한다
  - 각 Phase 끝에서 사람이 확인할 수 있는 결과물이 있어야 한다
  - 한 Phase에 모듈 5개 이상이면 쪼갠다
  - 핵심 로직은 최대한 앞 Phase에서 검증한다

❌ 피해야 할 것
  - 테스트만 있는 Phase (구현과 합친다)
  - 설정만 있는 Phase (구현과 합친다)
  - 순환 의존 (Phase A → B → A)
```

## Phase 검증 기준 예시

| 프로젝트 유형 | Phase 종료 검증 |
|-------------|---------------|
| CLI | 명령 실행 → stdout에 기대 결과 출력 → 종료 코드 0 |
| 라이브러리 | `npm test` / `pytest` → 테스트 전체 통과 |
| 백엔드 | `curl` 또는 Postman으로 API 호출 → 기대 응답 |
| BE 단일 기능 | curl happy/error → 응답 스키마 일치 + DB 반영 확인 + 테스트 통과 |
| FE 단일 기능 | 화면 진입 → 로딩/에러/빈/성공 4상태 확인 + 인터랙션 + a11y |
| 파이프라인 | 입력 데이터 → 처리 → 출력 파일/DB 대조 |
| AI Agent | 테스트 프롬프트 → 기대 동작 확인 |
