# 범용 검증 시나리오 규칙 (verification)

---

## 통합 검증 형식

```markdown
### Phase N 통합 검증

**시나리오**: [한 줄 요약 — 무엇을 확인하는 Phase인지]

1. [실행할 명령 또는 액션]
2. [입력 데이터 / 파라미터]
3. [기대 출력 확인]
4. [부수 효과 확인 — 파일 생성, DB 변경, 로그 등]

**합격 조건** : [조건 A] ✅ / [조건 B] ✅ / [조건 C] ✅
```

## 프로젝트 유형별 검증 패턴

### CLI 도구

```
1. 명령 실행: `tool-name --option value input.txt`
2. stdout 확인: 기대 출력 패턴 매칭
3. 종료 코드 확인: `echo $?` → 0
4. 에러 케이스: 잘못된 입력 → stderr 에러 메시지 + 종료 코드 1
```

### 자동화 스크립트

```
1. 테스트 입력 데이터 준비
2. 스크립트 실행
3. 출력 파일/DB 레코드 대조
4. 로그 파일에 처리 완료 메시지 확인
```

### 라이브러리

```
1. 테스트 스위트 실행: `npm test` / `pytest`
2. 전체 테스트 통과 확인
3. 커버리지 확인: 핵심 모듈 80% 이상
4. 타입 체크 통과: `tsc --noEmit` / `mypy`
```

### 단독 백엔드

```
1. 서버 기동: `npm start` / `python main.py`
2. 헬스체크: `curl localhost:PORT/health` → 200
3. API 호출: `curl -X POST ...` → 기대 응답
4. DB 확인: 데이터 저장 확인
```

### BE 단일 기능 (엔드포인트 1~수개)

```
1. 마이그레이션 적용: `npm run migrate` → 새 컬럼/테이블 존재 확인
2. Happy path: `curl -X POST /api/{endpoint} -d '{유효한 페이로드}'`
   → 2xx 응답 + 응답 스키마 일치
3. Validation error: 필수 필드 누락 페이로드 → 4xx + 에러 메시지 필드
4. 권한 에러: 토큰 없이/잘못된 토큰 → 401/403
5. DB 반영: `SELECT * FROM {table} WHERE ...` → 레코드 존재
6. 단위 테스트: `pytest tests/{feature}` → 전체 통과
7. OpenAPI/Swagger 갱신 확인 (있는 경우)
```

합격 조건 예시:
- "POST /api/sites 200 + body.id 존재" ✅
- "필수 필드 누락 시 400 + error.field='name'" ✅
- "SELECT FROM sites WHERE id=1 → 1 row" ✅

### FE 단일 기능 (컴포넌트/훅/화면 1개)

```
1. 진입: 스토리북 스토리 열기 또는 라우트 `/path` 직접 접속
2. 정적 렌더링: 마크업·스타일·반응형(모바일/데스크톱) 확인
3. 4상태 검증:
   - 로딩: 스켈레톤/스피너 노출
   - 빈: 빈 상태 일러스트/메시지 + 다음 액션 노출
   - 에러: 에러 메시지 + 재시도 경로 노출
   - 성공: 데이터 정상 렌더링
4. 인터랙션: 클릭/입력/제출 → 기대 상태 변화 + 네트워크 호출 1회만
5. 키보드 접근성: Tab 순서, Enter/Space 동작, 포커스 표시
6. a11y 점검: 레이블, alt, aria 속성, 색 대비
7. 컴포넌트/훅 테스트: `npm test` → 통과
```

훅 단독이면 1~6 생략, 7만 사용 (입력별 반환값 + 사이드이펙트 검증).

합격 조건 예시:
- "API 로딩 중 Skeleton 렌더링 확인" ✅
- "에러 응답 시 ErrorBoundary 메시지 + 재시도 버튼 노출" ✅
- "빈 결과 시 EmptyState + 신규 추가 CTA 노출" ✅
- "submit 더블 클릭 시 fetch 1회만 호출" ✅

### AI Agent / LLM 도구

```
1. 테스트 프롬프트 입력
2. 응답 확인: 기대 형식 + 내용 품질
3. 도구 호출 확인: 올바른 도구가 선택되었는가
4. 토큰 사용량 확인: 예산 범위 내
```

## 좋은 검증 vs 나쁜 검증

```
✅ 좋은 검증
  "echo 'hello world' | tool-name --upper → 'HELLO WORLD' 출력"
  "pytest tests/ → 15 passed, 0 failed"
  "curl localhost:3000/api/users → { 'users': [...], 'total': 5 }"

❌ 나쁜 검증
  "프로그램이 정상 실행되는지 확인"
  "테스트가 통과하는지 확인"
  "에러가 없는지 확인"
```

## 합격 조건 개수 기준

### 일반 프로젝트
- 초기화 Phase: 1~2개 (빌드 성공, 실행 확인)
- 핵심 기능 Phase: 3~5개 (입력별 기대 출력 확인)
- 마무리 Phase: 2~3개 (에러 케이스 + 패키징 확인)

### BE 단일 기능 (Phase 2~3개 기준)
- DB Phase: 2개 (마이그레이션 적용 + 롤백 가능)
- 도메인+API Phase: 4~5개 (happy + validation 4xx + 권한 401/403 + DB 반영 + 응답 스키마)
- 테스트 Phase: 2개 (단위/통합 테스트 통과 + OpenAPI 갱신)

### FE 단일 기능 (Phase 2~3개 기준)
- UI 골격 Phase: 2~3개 (정적 렌더링 + 반응형 + 스토리북 진입)
- 데이터 연동 Phase: 4개 (로딩 + 에러 + 빈 + 성공 4상태)
- 테스트+접근성 Phase: 3개 (컴포넌트 테스트 + 키보드/스크린리더 + 색 대비)

순수 UI 컴포넌트(데이터 없음)는 데이터 연동 Phase를 생략하고
UI 골격 Phase 합격 조건에 인터랙션 1~2개를 추가한다.
