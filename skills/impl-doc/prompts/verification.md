# 범용 검증 시나리오 규칙 (verification)

---

## 통합 검증 형식

```markdown
### Phase N 통합 검증

**시나리오**: [한 줄 요약 — 무엇을 확인하는 Phase인지]

1. [실행할 명령 또는 액션]
2. [입력 데이터 / 파라미터]
3. [기대 출력 확인]
4. [부수 효과 확인 — 파일 생성, DB 변경, 로그 등]

**합격 조건** : [조건 A] ✅ / [조건 B] ✅ / [조건 C] ✅
```

## 프로젝트 유형별 검증 패턴

### CLI 도구

```
1. 명령 실행: `tool-name --option value input.txt`
2. stdout 확인: 기대 출력 패턴 매칭
3. 종료 코드 확인: `echo $?` → 0
4. 에러 케이스: 잘못된 입력 → stderr 에러 메시지 + 종료 코드 1
```

### 자동화 스크립트

```
1. 테스트 입력 데이터 준비
2. 스크립트 실행
3. 출력 파일/DB 레코드 대조
4. 로그 파일에 처리 완료 메시지 확인
```

### 라이브러리

```
1. 테스트 스위트 실행: `npm test` / `pytest`
2. 전체 테스트 통과 확인
3. 커버리지 확인: 핵심 모듈 80% 이상
4. 타입 체크 통과: `tsc --noEmit` / `mypy`
```

### 단독 백엔드

```
1. 서버 기동: `npm start` / `python main.py`
2. 헬스체크: `curl localhost:PORT/health` → 200
3. API 호출: `curl -X POST ...` → 기대 응답
4. DB 확인: 데이터 저장 확인
```

### AI Agent / LLM 도구

```
1. 테스트 프롬프트 입력
2. 응답 확인: 기대 형식 + 내용 품질
3. 도구 호출 확인: 올바른 도구가 선택되었는가
4. 토큰 사용량 확인: 예산 범위 내
```

## 좋은 검증 vs 나쁜 검증

```
✅ 좋은 검증
  "echo 'hello world' | tool-name --upper → 'HELLO WORLD' 출력"
  "pytest tests/ → 15 passed, 0 failed"
  "curl localhost:3000/api/users → { 'users': [...], 'total': 5 }"

❌ 나쁜 검증
  "프로그램이 정상 실행되는지 확인"
  "테스트가 통과하는지 확인"
  "에러가 없는지 확인"
```

## 합격 조건 개수 기준

- 초기화 Phase: 1~2개 (빌드 성공, 실행 확인)
- 핵심 기능 Phase: 3~5개 (입력별 기대 출력 확인)
- 마무리 Phase: 2~3개 (에러 케이스 + 패키징 확인)
