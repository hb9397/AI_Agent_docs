# 검증 시나리오 작성 규칙 (verification)

---

## 통합 검증 형식

```markdown
### ✅ Phase N 통합 검증

**시나리오**: [한 줄 요약 — 무엇을 확인하는 Phase인지]

1. [URL 접속 또는 명령 실행]
2. [사용자 액션]
3. [기대 동작 확인]
4. [DB 또는 로그에서 확인]

**합격 조건** : [조건 A] ✅ / [조건 B] ✅ / [조건 C] ✅
```

---

## 좋은 검증 vs 나쁜 검증

```
✅ 좋은 검증
  "DB에서 SELECT * FROM sites; 실행 — 레코드 존재 확인"
  "LogViewer에 SUCCESS 타입 로그 실시간 표시 확인"
  "VITE_NOVNC_URL 미설정 시 텍스트 안내, 설정 시 iframe 렌더링"

❌ 나쁜 검증
  "정상적으로 동작하는지 확인"
  "에러 없이 실행되는지 확인"
  "BE가 잘 동작하는지 확인"
```

---

## 합격 조건 개수 기준

- 단순 Phase: 2~3개
- 핵심 기능 Phase: 4~5개
- 배포 Phase: 서비스별 1개씩

---

## 화면 중심 Phase 검증

화면을 Phase 단위로 삼는 경우 통합 검증 안에 아래 체크리스트를 포함한다.

```markdown
#### 기능 검증
- [ ] {데이터 조회}: 화면 진입 시 목록/상세 데이터가 API로부터 로드된다
- [ ] {필터링}: 검색어 또는 필터 변경 시 API 재호출 후 결과가 갱신된다
- [ ] {등록/수정/삭제}: 사용자 액션 후 성공 메시지와 화면 반영을 확인한다

#### 상태 검증
- [ ] {로딩}: API 호출 중 로딩 인디케이터 또는 스켈레톤이 표시된다
- [ ] {에러}: API 에러 시 에러 메시지와 재시도 경로가 표시된다
- [ ] {빈 상태}: 데이터가 없을 때 안내 메시지와 다음 액션이 표시된다

#### 에지 케이스
- [ ] {대량 데이터}: 페이징/스크롤/정렬이 깨지지 않는다
- [ ] {권한 부족}: 권한 없는 기능 접근 시 적절한 안내 또는 리다이렉트가 동작한다
- [ ] {중복 제출}: 더블 클릭이나 빠른 필터 변경으로 중복 요청이 발생하지 않는다
```

RFP 기반 프로젝트에서는 검증 항목 앞에 `[SFR-XXX]` 또는 `[REQ-XXX]`를 붙인다.
