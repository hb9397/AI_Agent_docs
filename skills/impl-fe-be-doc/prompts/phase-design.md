## 분할 패턴

스케일이 프로젝트 전체가 아니면 Phase 0을 생략하고 해당 기능부터 시작한다.

```
Phase 0: 인프라 & 환경 세팅          ← 프로젝트 전체 스케일에서만
  → DB 컨테이너, 개발 환경, 폴더 구조
  → FE 없음 (BE/INF만)
  → 검증: 서버 기동 확인

Phase 1: 가장 단순한 CRUD            ← BE-FE 통신 파이프라인 전체 검증
  → BE: DB 모델 + 기본 API
  → FE: 라우터 + 목록/추가 UI
  → 검증: FE → API → DB 파이프라인

Phase N: 핵심 기능                   ← 설계 G섹션 "집중 로직" 기준
  → BE: 핵심 로직 모듈
  → FE: 해당 기능 UI + 실시간 연동
  → 검증: 기능 E2E 동작

마지막 Phase: 배포 환경              ← 있는 경우에만
  → BE/FE Dockerfile, compose.yaml
  → 검증: 컨테이너 기동 + 전체 플로우
```

## 페어 구성 규칙

```
✅ 올바른 페어
  BE: POST /sites API 구현
  FE: 사이트 추가 폼 + 목록 렌더링
  → 함께 완성되면 해당 기능이 E2E로 동작함

❌ 잘못된 페어 — 범위 과다
  BE: DB 모델 5개 + API 12개
  FE: 전체 페이지 4개
  → 어디서 막혔는지 알 수 없음

❌ 잘못된 페어 — 한쪽만 완성
  BE: 온보딩 API만 완성
  FE: 아직 없음
  → 사람이 결과를 확인할 수 없음
```

## Phase 내 BE 태스크 구현 순서

```
1. 의존성 없는 유틸/헬퍼 먼저 (AI chain, 변환 함수 등)
2. 데이터 접근 레이어 (DB 모델, 세션)
3. 핵심 비즈니스 로직
4. API 엔드포인트 / WebSocket (FE와 연결점)
```

FE는 BE API가 준비된 후 연동 구현.
단, UI 레이아웃(컴포넌트 껍데기)은 BE와 병렬 작업 가능.

## [화면 중심 Phase 패턴]

화면 중심 프로젝트에서는 화면 1개를 Phase 1개로 본다.
각 Phase 안에 그 화면을 완성하는 데 필요한 BE API, DTO, 상태 관리, FE 컴포넌트, 검증을 함께 넣는다.

```
Phase 0: 공통 레이아웃 / 인증 / 라우팅 기반
  → 공통 Shell, 권한 가드, API 클라이언트, 공통 상태

Phase 1: 핵심 데이터 입력 화면
  → BE: 등록/수정 API, 검증 DTO
  → FE: 폼 컴포넌트, 유효성, 제출/에러 상태
  → 검증: 폼 제출 → API 저장 → 성공/실패 표시

Phase 2: 핵심 목록/조회 화면
  → BE: 목록/상세 API, 검색/페이징
  → FE: 필터, 테이블, 상세 진입
  → 검증: 검색/필터/빈 상태/에러 상태

Phase N: 대시보드/차트/복합 화면
  → BE: 집계 API, 비동기/실시간 채널
  → FE: 차트, 서버 상태, 자동 갱신
```

화면 구현 순서 결정 기준:
1. 공통 레이아웃 / 인증 → 가장 먼저
2. 데이터 입력 화면 → 조회 화면보다 먼저 (데이터가 있어야 조회 가능)
3. 핵심 SFR/REQ 화면 → 보조 화면보다 먼저
4. 독립 화면 → 의존 화면보다 먼저
5. 실시간/차트/대시보드 → 기본 API와 데이터 구조 확정 후

## [BE 전용 Phase 패턴]

BE 전용 프로젝트에서는 FE 페어 요구를 적용하지 않는다.
Phase 종료 조건: API를 curl 또는 테스트 코드로 직접 검증 가능한 상태.

```
Phase 1: 가장 단순한 엔드포인트 1개 + DB 연결 검증
Phase N: 핵심 비즈니스 로직 API 구현
마지막: 배포 환경 세팅 (있는 경우)
```

## [FE 전용 Phase 패턴]

FE 전용 프로젝트에서는 BE 페어 요구를 적용하지 않는다.
Phase 종료 조건: 브라우저에서 화면이 정상 렌더링되는 상태.

```
Phase 1: 기본 레이아웃 + 라우터 설정
Phase N: 핵심 화면 컴포넌트 구현
마지막: 빌드 및 배포 설정 (있는 경우)
```
