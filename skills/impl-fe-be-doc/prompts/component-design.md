# 컴포넌트 설계 규칙 (component-design)

화면 중심 Phase에서 FE 태스크와 Phase 본문을 작성할 때 사용한다.

---

## 컴포넌트 트리 작성 규칙

각 화면의 주요 컴포넌트를 트리 구조로 표현한다.

````markdown
### 컴포넌트 구조

```text
{화면명}Page
├── CommonLayout (공통)
│   ├── Header
│   └── Sidebar
├── {기능}Filter
├── {기능}Table
│   ├── TableRow × N
│   └── Pagination
└── {기능}Modal (조건부)
```
````

---

## 컴포넌트 명세 형식

복합 화면은 주요 컴포넌트마다 아래를 기술한다. 단순 화면은 트리와 책임만으로 충분하다.

```markdown
#### {컴포넌트명}

- **책임**: {이 컴포넌트가 하는 일 — 1문장}
- **Props**: {부모로부터 받는 데이터}
- **로컬 상태**: {이 컴포넌트 내부에서 관리하는 상태}
- **이벤트**: {부모에게 올리는 이벤트 / 콜백}
- **API 호출**: {이 컴포넌트가 직접 호출하는 API — 없으면 "없음"}
```

---

## 공통 컴포넌트 식별 기준

```text
공통 컴포넌트로 분리하는 경우:
  - 2개 이상 화면에서 동일한 UI 패턴 사용
  - 동일한 API 호출 + 동일한 렌더링 로직
  - 프로젝트 전역 상태에 접근하는 UI (인증 상태, 알림 등)

분리하지 않는 경우:
  - 비슷해 보이지만 비즈니스 로직이 다른 경우
  - 1개 화면에서만 사용하는 경우
  - 분리 시 props가 10개 이상 필요한 경우
```

## 상태 관리 분류

```markdown
### 상태 관리

| 상태 | 유형 | 위치 | 설명 |
|------|------|------|------|
| 검색 필터 값 | 로컬 | Filter 컴포넌트 | URL 쿼리 파라미터와 동기화 |
| 목록 데이터 | 서버 | React Query / SWR / 서버 상태 캐시 | API 응답 캐싱 + 자동 재검증 |
| 선택된 행 | 로컬 | Table 컴포넌트 | 상세 모달 열기용 |
| 인증 토큰 | 전역 | Context / Store | 전체 앱 공유 |
```

로딩, 에러, 빈 상태는 각 화면마다 반드시 정의한다.
